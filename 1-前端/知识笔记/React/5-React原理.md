## 一、JSX 与 createElement

### 1.1 对应关系

**JSX** 是语法糖，经构建工具（如 Babel）编译后，会转换为对 **React.createElement** 的调用（React 17+ 可使用 `jsx` 运行时），最终得到描述 UI 的 **JavaScript 对象**。

| 层级       | 说明 |
| ---------- | ---- |
| **源码**   | 开发者写 JSX 标签 |
| **编译后** | `createElement(type, props, ...children)` 调用 |
| **运行时** | 得到元素描述对象，供后续协调与渲染使用 |

**核心思想：** 用声明式写法描述 UI，由运行时根据描述对象生成或更新真实 DOM。

### 1.2 createElement 返回值（了解即可）

返回值可简化为 `{ type, props }`：`type` 为标签名或组件，`props` 包含属性及 `children`。源码中会处理 **config**（属性）、**children**（子节点）、**defaultProps** 等，最终得到标准的 **ReactElement** 描述对象。

```
JSX → createElement(type, props, children) → { type, props }
```

**类组件与 Component（了解即可）：** 类组件需继承 **React.Component**。Component 源码中会挂载 `props`、`context`、`refs` 以及 `updater`（由渲染器注入，用于 setState 等），理解“组件实例上有哪些字段”即可。

## 二、虚拟 DOM 与 Diff

### 2.1 虚拟 DOM

**虚拟 DOM（Virtual DOM）** 是用 JavaScript 对象描述的 DOM 结构，是真实 DOM 的一层抽象。

| 项目     | 说明 |
| -------- | ---- |
| **本质** | 描述节点树结构的 JS 对象（如 React 的 ReactElement） |
| **作用** | 在内存中先对“新树”与“旧树”做对比，再按差异更新真实 DOM，减少直接操作 DOM 的次数 |

**直接操作 DOM vs 虚拟 DOM：**

| 对比项     | 直接操作 DOM     | 虚拟 DOM                     |
| ---------- | ---------------- | ---------------------------- |
| **更新方式** | 手动查节点、改属性、改子节点 | 先 Diff 再批量更新真实 DOM   |
| **性能**   | 多次改 DOM 易触发多次重排重绘 | 合并变更，减少重排重绘       |
| **跨平台** | 依赖浏览器 DOM   | 同一套描述可渲染到不同平台   |
| **适用**   | 简单、局部更新   | 复杂树结构、频繁更新         |

### 2.2 Diff 策略（了解即可）

React 对两棵虚拟 DOM 树进行对比（Diff）时，采用约定策略以控制复杂度：

| 策略         | 说明 |
| ------------ | ---- |
| **同层比较** | 只对同一层级的节点比较，不跨层递归，复杂度 O(n) |
| **类型一致** | 节点类型（标签名或组件）相同则复用并递归子节点；类型不同则拆掉整棵子树重建 |
| **key 优化** | 列表项用 **key** 稳定标识，便于复用与最小化增删；避免用 index 作 key 导致顺序错乱 |

> 💡 列表渲染时为每项提供**稳定、唯一**的 key，否则可能引发多余重绘或状态错位。

**Diff 执行流程（了解即可）：** 初次渲染根据 state 生成虚拟 DOM 树 → 据此生成真实 DOM 并渲染；数据变化后根据新 state 生成新虚拟 DOM → 与上一棵虚拟 DOM 做 Diff → 得到需变更的 Patch → 仅将变更部分更新到真实 DOM。

## 三、渲染与任务调度（了解即可）

### 3.1 整体流程

从组件树到屏幕像素可简化为：

```
组件树（state/props）→ 生成虚拟 DOM → Diff 算出变更 → 提交到渲染阶段 → 更新真实 DOM
```

### 3.2 任务调度

React 将“哪些组件需要更新”拆成**可中断的小任务**，在浏览器空闲或指定时机执行，避免长时间阻塞主线程。React 18 的 **Concurrent 模式**与 **Suspense** 都依赖这套调度机制。

| 概念       | 说明 |
| ---------- | ---- |
| **可中断** | 更新任务可被拆成多段，高优先级（如输入）可打断低优先级 |
| **双缓冲** | 内存中维护两棵 Fiber 树（**current** 与 **workInProgress**），当前屏与待更新树交替，减少闪烁 |

### 3.3 Fiber 与 Reconciler（了解即可）

React 16 之前采用 **Stack Reconciler**，自顶向下递归整棵虚拟 DOM，任务一旦开始无法中断，易造成卡顿。**Fiber** 将调和过程改为可中断的链表结构（Fiber 节点），配合调度器在适当时机让出主线程，高优先级任务（如用户输入）可先执行，浏览器空闲再恢复渲染，该机制又称 **Fiber Reconciler**。

| 对比项     | Stack Reconciler   | Fiber Reconciler     |
| ---------- | ------------------ | -------------------- |
| **结构**   | 递归、整树一次跑完 | 链表、可分段执行     |
| **可中断** | 否                 | 是                   |
| **优先级** | 无                 | 可区分高/低优先级   |

## 四、要点小结

| 主题           | 要点 |
| -------------- | ---- |
| **JSX 本质**   | 语法糖，编译为 createElement，得到描述 UI 的 JS 对象 |
| **虚拟 DOM**   | 用 JS 对象描述 DOM 树，在内存中先 Diff 再按需更新真实 DOM |
| **Diff**       | 同层比较、类型一致则复用、列表用 key 优化 |
| **为何用虚拟 DOM** | 减少直接操作 DOM、跨平台抽象、配合调度实现可中断更新 |
| **任务调度**   | 可中断、双缓冲、Fiber 链表结构，支持优先级与 Concurrent 模式 |
