## 一、服务端渲染简介

### 1.1 什么是 SSR

**服务端渲染（SSR，Server-Side Rendering）** 指在服务器上把组件渲染成 HTML 字符串并返回给浏览器；浏览器先显示首屏 HTML，再加载 JS 进行**注水（hydration）**，为已有 DOM 绑定事件与状态，使页面具备交互能力。

| 对比项       | 纯客户端渲染（CSR）       | 服务端渲染（SSR）             |
| ------------ | ------------------------- | ----------------------------- |
| **首屏 HTML** | 多为空壳，依赖 JS 渲染    | 服务器直接输出首屏完整 HTML   |
| **SEO**      | 爬虫可能拿不到正文       | 首屏即完整 HTML，利于收录    |
| **首屏速度** | 依赖 JS 下载与执行        | 首屏更快，再加载 JS 注水      |
| **服务器压力** | 小，多为静态资源         | 大，每次请求都执行渲染        |

### 1.2 为何考虑 SSR

| 场景           | 说明                                           |
| -------------- | ---------------------------------------------- |
| **SEO**        | 需要搜索引擎抓取首屏正文与关键信息时           |
| **首屏性能**   | 弱网或低端设备希望尽快看到内容，减少白屏时间   |
| **社交分享**   | 爬虫拉取的 HTML 需包含正文与缩略图，以便正确预览 |

## 二、React 中实现 SSR 的思路

### 2.1 核心 API（了解即可）

在 Node 环境中用 **react-dom/server** 将 React 组件转为 HTML 字符串：

| API                         | 说明                                                   |
| --------------------------- | ------------------------------------------------------ |
| **renderToString**          | 将组件树同步渲染为 HTML 字符串，用于首屏输出           |
| **renderToPipeableStream**  | React 18 推荐，流式输出，支持 Suspense，首屏可逐步返回 |

### 2.2 简要流程

```
请求进入
    ↓
服务端根据当前 URL 匹配路由、取数
    ↓
renderToString(组件) 得到 HTML 字符串
    ↓
将 HTML 注入 HTML 模板，一并返回给浏览器
    ↓
浏览器展示首屏 → 加载同构的 React 代码 → hydration 绑定事件与状态
```

**同构（Isomorphism）**：同一套 React 组件在服务端执行一遍（输出首屏 HTML），再在客户端执行一遍（注水、绑定事件）；服务端与客户端共用组件与路由逻辑，仅运行环境不同。

### 2.3 注意点

| 要点     | 说明                                                                 |
| -------- | -------------------------------------------------------------------- |
| **路由** | 服务端需根据当前 URL 渲染对应组件，与前端路由约定一致                |
| **数据** | 首屏数据在服务端请求并注入，避免注水后再请求导致闪烁；可与 Redux/Context 同构 |
| **样式** | 服务端需输出当前路由用到的 CSS（如 CSS-in-JS 服务端提取，或按路由打包 CSS） |

## 三、Next.js 简介

### 3.1 定位

**Next.js** 是基于 React 的**全栈框架**，内置 SSR/SSG、文件系统路由、API 路由与构建优化。官方定位是 “React framework for building full-stack web applications”，**不仅用于 SSR**，也支持静态站点、增量静态再生成、偏前端的 SPA 以及全栈应用，按需选择渲染与部署方式。

| 能力         | 说明                                                                 |
| ------------ | -------------------------------------------------------------------- |
| **SSR / SSG** | 服务端渲染（每次请求时渲染）或静态生成（构建时生成 HTML），按需选择 |
| **文件路由** | 根据 `pages/` 或 `app/` 下文件结构自动生成路由                       |
| **API 路由** | 在项目内写接口（如 `pages/api/xxx.js`），无需单独后端服务            |
| **构建优化** | 自动代码分割、图片优化、环境变量等                                   |

**渲染与部署方式概览：**

| 方式 / 能力       | 说明                                                                 |
| ----------------- | -------------------------------------------------------------------- |
| **SSR**           | 每次请求在服务端生成 HTML，适合强实时、个性化内容                     |
| **SSG**           | 构建时生成静态 HTML，适合内容相对固定的页面                           |
| **ISR**           | 增量静态再生成：静态页 + 按需/定时再生成，兼顾性能与更新              |
| **CSR**           | 通过 `"use client"`、客户端组件与客户端取数，实现以浏览器为主的渲染   |
| **静态导出**      | 导出为纯静态文件，可部署到任意静态托管，无需 Node 服务器              |
| **全栈**          | Route Handlers、Server Actions、服务端取数等，前后端一体              |

**SSR 与 SSG 对比：**

| 对比项     | SSR（服务端渲染）     | SSG（静态生成）           |
| ---------- | --------------------- | ------------------------- |
| **生成时机** | 每次请求时在服务器渲染 | 构建时生成 HTML           |
| **适用场景** | 数据实时性要求高、因人而异 | 内容相对固定、可预先生成 |
| **服务器压力** | 高                    | 低，多为静态文件          |

### 3.2 创建与运行（了解即可）

```bash
# 使用最新版本创建项目
npx create-next-app@latest <项目名>
cd <项目名>
# 开发环境运行
npm run dev
```

> 💡 根据提示可选择是否使用 TypeScript、ESLint、Tailwind、`src/` 目录、App Router 等。

### 3.3 数据获取方式（了解即可）

| 方式 / API        | 说明                                           |
| ----------------- | ---------------------------------------------- |
| **getServerSideProps**（Pages Router） | 在服务端每请求执行一次，用于 SSR，可访问请求上下文 |
| **getStaticProps**（Pages Router）     | 构建时执行，用于 SSG；可配 `revalidate` 做增量静态再生成 |
| **服务端组件 + fetch**（App Router）   | 在服务端组件中直接 `async/await fetch`，默认缓存，适合 App Router 项目 |

### 3.4 何时选用

| 场景                     | 建议                                           |
| ------------------------ | ---------------------------------------------- |
| 需要 SEO 或首屏性能      | 使用 Next.js 的 SSR/SSG 或 ISR                 |
| 纯静态站、无服务端       | Next.js 静态导出，部署到任意静态托管           |
| 偏前端的 SPA             | Next.js 也可胜任（客户端组件 + 静态导出等）    |
| 全栈小项目、前后端一体   | Next.js API 路由、Server Actions 可简化部署   |

**选择建议：** 偏 SEO 的站点、非强交互页面（如新闻、落地页）更适合 SSR；后台管理、强交互应用首屏压力相对小，用前端渲染即可，且更易维护、服务器负载更小。是否用 Next.js 取决于是否需要其路由、构建与全栈能力，而不仅是“要不要 SSR”。
