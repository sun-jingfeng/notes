## 一、服务端渲染简介

### 1.1 什么是 SSR

**服务端渲染（SSR，Server-Side Rendering）** 指在服务器上把组件渲染成 HTML 字符串，再返回给浏览器；浏览器先显示首屏 HTML，再加载 JS 进行“注水”（hydration），使页面具备交互能力。

| 对比项     | 纯客户端渲染（CSR）     | 服务端渲染（SSR）           |
| ---------- | ----------------------- | --------------------------- |
| **首屏 HTML** | 多为空壳，依赖 JS 渲染 | 服务器直接输出首屏 HTML    |
| **SEO**    | 爬虫可能拿不到内容     | 首屏即完整 HTML，利于收录 |
| **首屏速度** | 依赖 JS 下载与执行      | 首屏更快，再加载 JS 注水   |

### 1.2 为何考虑 SSR

| 场景         | 说明 |
| ------------ | ---- |
| **SEO**      | 需要搜索引擎抓取首屏内容时 |
| **首屏性能** | 弱网或低端设备希望尽快看到内容 |
| **社交分享** | 爬虫拉取到的 HTML 需包含正文与缩略图 |

## 二、React 中实现 SSR 的思路

### 2.1 核心 API（了解即可）

在 Node 环境中用 **react-dom/server** 将 React 组件转为 HTML 字符串：

| API                | 说明 |
| ------------------ | ---- |
| **renderToString** | 将组件树渲染为 HTML 字符串，用于首屏输出 |
| **renderToPipeableStream** | React 18 推荐，流式输出，支持 Suspense |

### 2.2 简要流程

```
请求进入 → 在服务端根据路由取数 → renderToString(组件) → 将 HTML 注入模板 → 返回给浏览器
→ 浏览器展示首屏 → 加载同构的 React 代码 → hydration 绑定事件
```

**同构（Isomorphism）**：同一套 React 组件在服务端执行一遍（输出首屏 HTML），再在客户端执行一遍（注水、绑定事件与交互）；服务端与客户端共用组件与路由逻辑，仅运行环境不同。

### 2.3 注意点

| 要点         | 说明 |
| ------------ | ---- |
| **路由**     | 服务端需根据当前 URL 渲染对应组件，与前端路由约定一致 |
| **数据**     | 首屏数据在服务端请求，避免注水后再请求导致闪烁；可与 Redux/Context 同构 |
| **样式**     | 服务端需输出当前路由用到的 CSS（如 CSS-in-JS 服务端提取、或按路由打包 CSS） |

## 三、Next.js 简介

### 3.1 定位

**Next.js** 是基于 React 的全栈框架，内置 **SSR/SSG**、文件系统路由、API 路由、构建与优化等，适合需要服务端渲染或静态生成的 React 项目。

| 能力         | 说明 |
| ------------ | ---- |
| **SSR / SSG** | 服务端渲染或静态生成页面，按需选择 |
| **文件路由** | 根据 `pages/` 或 `app/` 下文件结构自动生成路由 |
| **API 路由** | 在项目内写接口（如 `pages/api/xxx.js`），无需单独后端服务 |
| **构建优化** | 自动代码分割、图片优化、环境变量等 |

### 3.2 创建与运行（了解即可）

```bash
npx create-next-app@latest <项目名>
cd <项目名>
npm run dev
```

### 3.3 何时选用

| 场景             | 建议 |
| ---------------- | ---- |
| 需要 SEO 或首屏性能 | 考虑 Next.js 的 SSR/SSG |
| 纯 SPA、无 SEO 需求 | Vite + React Router 即可 |
| 全栈小项目、希望前后端一体 | Next.js API 路由可简化部署 |

**选择建议**：偏 SEO 的站点、非强交互页面（如新闻、落地页）更适合 SSR；后台管理、强交互应用首屏压力相对小，用前端渲染即可，且更易维护、服务器负载更小。
