## 一、微服务概述

### 1.1 什么是微服务

**微服务（Microservices）** 是一种软件架构风格，将应用程序构建为一组小型、独立部署的服务集合。

**核心特点：**

| 特点         | 说明                              |
| ------------ | --------------------------------- |
| **独立进程** | 每个服务运行在自己的进程中        |
| **业务聚焦** | 围绕特定的业务功能构建            |
| **轻量通信** | 通过 HTTP/REST 或消息队列进行通信 |
| **独立部署** | 可以独立部署、扩展和维护          |

### 1.2 单体架构 vs 微服务架构

| 特性         | 单体架构         | 微服务架构         |
| ------------ | ---------------- | ------------------ |
| **部署**     | 整体部署         | 独立部署           |
| **技术栈**   | 统一技术栈       | 可以使用不同技术栈 |
| **扩展性**   | 整体扩展         | 按需扩展单个服务   |
| **开发效率** | 初期快，后期慢   | 初期慢，后期快     |
| **容错性**   | 一处故障影响全局 | 故障隔离           |
| **团队协作** | 耦合度高         | 团队独立开发       |

### 1.3 微服务的优缺点

| 类型     | 说明                                                                       |
| -------- | -------------------------------------------------------------------------- |
| **优点** | 独立部署快速迭代、技术栈灵活、按需扩展、故障隔离、团队自治                 |
| **缺点** | 分布式复杂性增加、运维成本提高、服务间通信开销、数据一致性难保证、测试复杂 |

**与 SOA 的区别：**

| 对比项       | SOA          | 微服务   |
| ------------ | ------------ | -------- |
| **管理方式** | ESB 集中管理 | 去中心化 |
| **服务粒度** | 较大         | 较小     |
| **部署**     | 通常整体部署 | 独立部署 |
| **技术选型** | 统一         | 灵活     |

## 二、Spring Cloud 生态系统

**Spring Cloud** 是 Java 微服务的主流解决方案，基于 Spring Boot 构建。

### 2.1 核心组件概览

```
┌─────────────────────────────────────────────────────────────┐
│                     Spring Cloud 生态                        │
├─────────────────────────────────────────────────────────────┤
│  服务注册发现: Nacos / Eureka / Consul                       │
│  服务调用: OpenFeign / RestTemplate                          │
│  负载均衡: LoadBalancer / Ribbon                             │
│  服务网关: Spring Cloud Gateway / Zuul                       │
│  配置中心: Nacos Config / Spring Cloud Config                │
│  服务熔断: Sentinel / Resilience4j / Hystrix                 │
│  分布式事务: Seata                                           │
│  链路追踪: Sleuth + Zipkin / SkyWalking                      │
│  消息总线: Spring Cloud Bus                                  │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **Ribbon**、**Zuul**、**Hystrix** 已进入维护或停用，当前推荐：负载均衡用 **LoadBalancer**、网关用 **Spring Cloud Gateway**、熔断用 **Sentinel** 或 **Resilience4j**。

## 三、服务注册与发现

### 3.1 服务治理中的角色与机制

**服务治理中的三个角色：**

| 角色           | 说明                                                             |
| -------------- | ---------------------------------------------------------------- |
| **服务提供者** | 暴露服务接口，供其他服务调用                                     |
| **服务消费者** | 调用其他服务提供的接口                                           |
| **注册中心**   | 记录并监控各微服务实例状态，推送服务变更信息（如 Nacos、Eureka） |

**常见问题梳理：**

| 问题                                       | 解答                                                               |
| ------------------------------------------ | ------------------------------------------------------------------ |
| 消费者如何知道提供者的地址？               | 提供者启动时注册到注册中心；消费者从注册中心订阅并拉取服务信息     |
| 消费者如何得知服务状态变更？               | 提供者通过**心跳**上报健康状态；异常时注册中心剔除实例并通知消费者 |
| 当提供者有多个实例时，消费者该选择哪一个？ | 通过**负载均衡**算法，从多个实例中选出一个进行调用                 |

### 3.2 为什么需要服务注册发现

在微服务架构中，服务实例动态变化（扩缩容、故障恢复），需要一个中心化的注册中心来管理服务实例信息。

```
┌──────────┐      注册        ┌──────────────┐
│ Service A│ ──────────────▶  │              │
└──────────┘                  │   注册中心    │
                              │  (Nacos等)   │
┌──────────┐      发现        │              │
│ Service B│ ◀──────────────  └──────────────┘
└──────────┘
```

> 💡 下面介绍两种常见的注册中心实现，**项目中选其一即可**：Nacos（Spring Cloud Alibaba 推荐）、Eureka（Netflix 方案，多用于老项目）。

### 3.3 Nacos 服务注册

**前置条件：** 先部署并启动 Nacos 服务端。项目内引入的是 Nacos **客户端**，用于连接已运行的服务端。

#### 部署 Nacos 服务端

**方式一：Docker（推荐）**

```bash
docker run -d \
  --name nacos \
  -p 8848:8848 \
  -p 9848:9848 \
  -e MODE=standalone \
  -e JVM_XMS=256m \
  -e JVM_XMX=256m \
  -e NACOS_AUTH_ENABLE=true \
  -e NACOS_AUTH_IDENTITY_KEY=serverIdentity \
  -e NACOS_AUTH_IDENTITY_VALUE=security \
  nacos/nacos-server:v2.5.2
```

> 💡 开启认证时，Nacos 2.x 必须配置 `NACOS_AUTH_IDENTITY_KEY` 和 `NACOS_AUTH_IDENTITY_VALUE`，否则启动报错 Empty identity。
> 💡 Mac Apple Silicon（M1/M2/M3）需在 `docker run -d` 后增加：`--platform linux/amd64`

- 控制台：http://localhost:8848/nacos ，默认账号/密码 `nacos`
- 日常使用：`docker start nacos` 启动，`docker stop nacos` 停止

**方式二：官网安装包**

从 [Nacos releases](https://github.com/alibaba/nacos/releases) 下载 `nacos-server-xxx.tar.gz`，解压后执行：

```bash
# Linux/Mac
sh bin/startup.sh -m standalone

# Windows
bin/startup.cmd -m standalone
```

#### 项目接入步骤

**1. 添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

**2. 配置 application.yml**

```yaml
spring:
  application:
    name: order-service           # 服务名称
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848  # Nacos 服务端地址
        namespace: dev               # 命名空间，用于环境隔离
        group: DEFAULT_GROUP         # 分组
```

**3. 启用服务发现**

| 注解                       | 作用                                                                                             |
| -------------------------- | ------------------------------------------------------------------------------------------------ |
| **@EnableDiscoveryClient** | 启用服务发现客户端，使当前应用向注册中心注册并拉取服务列表；加在启动类上，Nacos、Eureka 等通用。 |

```java
@SpringBootApplication
@EnableDiscoveryClient  // 启用服务发现
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

## 四、服务调用

服务间通过 HTTP 进行远程调用，可使用 RestTemplate 或声明式的 OpenFeign。

### 4.1 服务发现与负载均衡原理

用**服务名**调用（如 `http://user-service/...`）时，底层实际做了三件事：

```
从注册中心拉取实例列表 → 负载均衡选一个实例 → 用该实例的 IP:端口 发请求
```

其中「负载均衡选实例」由 **Spring Cloud LoadBalancer** 完成：当存在多个实例时，由它按策略（默认轮询）选出一个再发请求。

以下用 `DiscoveryClient` 手写一遍，便于理解框架在背后的流程：

```java
@Service
public class CartService {
    
    @Autowired
    private DiscoveryClient discoveryClient;

    private void handleCartItems(List<CartVO> vos) {
        // 1. 根据服务名称，拉取该服务的实例列表
        List<ServiceInstance> instances = discoveryClient.getInstances("item-service");
        if (instances == null || instances.isEmpty()) {
            throw new RuntimeException("item-service 无可用实例");
        }
        
        // 2. 负载均衡：从多个实例中选一个（此处为随机）
        ServiceInstance instance = instances.get(
            new Random().nextInt(instances.size())
        );
        
        // 3. 获取该实例的地址
        URI uri = instance.getUri();
        
        // 4. 用 uri 发起 HTTP 调用
        // restTemplate.getForObject(uri + "/items/" + id, Item.class);
    }
}
```

**服务发现的两种模式：**

| 模式           | 说明                                                       |
| -------------- | ---------------------------------------------------------- |
| **客户端发现** | 客户端从注册中心拉取服务列表，自己做负载均衡（本笔记采用） |
| **服务端发现** | 通过负载均衡器代理，客户端只访问代理，无感知实例列表       |

> 💡 实际开发中一般用 **RestTemplate + @LoadBalanced** 或 **OpenFeign**，由框架完成上述流程。

### 4.2 RestTemplate

Spring 提供的 HTTP 客户端工具。使用**服务名**调用时需配合负载均衡，RestTemplate 的 Bean 上要加 **@LoadBalanced**。

**远程调用的四要素：** 服务名（用于从注册中心拉取实例）、请求方式与路径、返回值类型、请求参数。RestTemplate 的 `exchange(url, method, entity, responseType, uriVariables)` 即对应这四类信息。

#### 1. 注入 RestTemplate 到 Spring 容器

```java
@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 为该 RestTemplate 启用 LoadBalancer，才能用服务名调用并做负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

#### 2. 发起远程调用

```java
@Service
public class OrderService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public User getUser(Long userId) {
        // 使用服务名代替具体 IP
        String url = "http://user-service/users/" + userId;
        return restTemplate.getForObject(url, User.class);
    }
}
```

#### 3. exchange 方法（灵活度最高）

需要自定义请求头、获取响应状态码时使用：

```java
// 构建请求头
HttpHeaders headers = new HttpHeaders();
headers.set("Authorization", "Bearer token");
HttpEntity<Void> entity = new HttpEntity<>(headers);

// 发起请求
ResponseEntity<User> response = restTemplate.exchange(
    "http://user-service/users/" + userId,
    HttpMethod.GET,
    entity,
    User.class
);

// 获取结果
User user = response.getBody();
HttpStatus status = response.getStatusCode();
```

### 4.3 OpenFeign（推荐）

**OpenFeign** 是声明式 HTTP 客户端，更优雅的服务调用方式。使用前需了解两个核心注解。

#### 关键注解

| 注解                    | 作用                                                                                                                                 |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **@EnableFeignClients** | 加在启动类上，开启 Feign，扫描并注册 `@FeignClient` 接口；若 Feign 接口在其他包（如 API 模块），需指定 `basePackages` 或 `clients`。 |
| **@FeignClient**        | 声明式 HTTP 客户端接口。`name`/`value` 为服务名（走注册中心+负载均衡）；可配 `fallback`/`fallbackFactory` 做熔断降级。               |

#### 1. 添加依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
```

#### 2. 启用 Feign

```java
@SpringBootApplication
@EnableFeignClients  // 启用 Feign 客户端
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

#### 3. 定义 Feign 接口

```java
@FeignClient(name = "user-service", fallback = UserClientFallback.class)
public interface UserClient {
    
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
    
    @PostMapping("/users")
    User createUser(@RequestBody User user);
    
    @GetMapping("/users")
    List<User> getUsers(@RequestParam("ids") List<Long> ids);
}
```

#### 4. 使用 Feign 客户端

```java
@Service
public class OrderService {
    
    @Autowired
    private UserClient userClient;
    
    public OrderVO getOrderDetail(Long orderId) {
        Order order = orderRepository.findById(orderId);
        // 像调用本地方法一样调用远程服务
        User user = userClient.getUserById(order.getUserId());
        return new OrderVO(order, user);
    }
}
```

#### 5. Feign 配置

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 5000   # 连接超时
        readTimeout: 5000      # 读取超时
        loggerLevel: BASIC     # 日志级别：NONE/BASIC/HEADERS/FULL
  compression:
    request:
      enabled: true   # 请求压缩
    response:
      enabled: true   # 响应压缩
```

#### 6. 日志

Feign 可打印请求与响应，便于排查调用问题。无论用哪种方式设日志级别，都需在 **logging.level** 中将 Feign 接口所在包设为 **DEBUG**（Feign 以 DEBUG 级别输出，否则不打印）。

**日志级别**

| 级别        | 说明                                        |
| ----------- | ------------------------------------------- |
| **NONE**    | 不记录（默认）                              |
| **BASIC**   | 仅记录请求方法、URL、响应状态码、耗时       |
| **HEADERS** | 在 BASIC 基础上增加请求/响应头              |
| **FULL**    | 记录请求/响应头、正文，便于排查参数与返回值 |

**方式一：YAML 配置**

在 `feign.client.config` 中指定 `loggerLevel`，`default` 表示全局，也可按服务名单独配置：

```yaml
feign:
  client:
    config:
      default:
        loggerLevel: FULL
      user-service:
        loggerLevel: BASIC

logging:
  level:
    com.example.api.client: DEBUG   # Feign 接口所在包，必须设为 DEBUG
```

**方式二：Java 配置（Logger.Level Bean）**

定义配置类并声明 `Logger.Level` Bean，再通过注解指定生效范围：

```java
// 配置类
public class DefaultFeignConfig {
    @Bean
    public Logger.Level feignLogLevel() {
        return Logger.Level.FULL;
    }
}

// 仅对单个 Feign 客户端生效
@FeignClient(value = "item-service", configuration = DefaultFeignConfig.class)
public interface ItemClient { ... }

// 对所有 Feign 客户端生效（加在启动类上）
@EnableFeignClients(defaultConfiguration = DefaultFeignConfig.class)
@SpringBootApplication
public class OrderServiceApplication { ... }
```

- **单客户端**：在 `@FeignClient` 上写 `configuration = DefaultFeignConfig.class`。
- **全局**：在 `@EnableFeignClients` 上写 `defaultConfiguration = DefaultFeignConfig.class`。

两种方式对比：无绝对优劣，按习惯或场景选即可。YAML 不改代码即可调级别，适合按环境切换（如 dev 用 FULL、prod 用 NONE）；Java 配置适合在同一个配置类里统一管理 Logger、拦截器、编解码器等多项 Feign 配置。

#### 7. 连接池（OpenFeign 整合 OKHttp）

默认 Feign 使用 URLConnection，连接不复用。改用 OKHttp 可开启连接池，提升高并发下的性能。

**1. 引入依赖**

```xml
<!-- ok-http -->
<dependency>
    <groupId>io.github.openfeign</groupId>
    <artifactId>feign-okhttp</artifactId>
</dependency>
```

**2. 开启连接池**

```yaml
feign:
  okhttp:
    enabled: true   # 开启 OKHttp 连接池支持
```

#### 8. 最佳实践

| 实践项 | 说明 |
| ------ | ---- |
| **接口独立成模块** | 将 Feign 接口、DTO 放在独立 API 模块，供多个消费者依赖；消费者通过 `@EnableFeignClients(basePackages = "xxx.api.client")` 或 `@EnableFeignClients(clients = {UserClient.class})` 指定扫描。 |
| **超时与重试** | 在 `feign.client.config` 中合理设置 `connectTimeout`、`readTimeout`；重试需谨慎，写操作避免重复提交，可配合熔断使用。 |
| **熔断降级** | 生产环境建议为 `@FeignClient` 配置 `fallback` 或 `fallbackFactory`，与 Sentinel/Resilience4j 集成，避免下游故障拖垮调用方。 |
| **连接池** | 生产环境建议开启 OKHttp 连接池，减少连接建立开销，提高并发能力。 |
| **日志级别** | 生产默认 NONE 或 BASIC，排查问题时再开 FULL；用 YAML 按环境区分，避免日志量过大。 |
| **统一请求头** | 通过 `RequestInterceptor` 在配置类中统一添加 Token、TraceId、租户 ID 等，避免每个接口手写。 |
| **接口设计** | 避免单次请求体/响应体过大，列表类接口考虑分页；接口方法语义清晰，与提供方 REST 约定一致。 |

**请求拦截器示例（统一传 Token）**

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String token = UserContext.getToken();  // 从上下文或 ThreadLocal 获取
            if (token != null) {
                template.header("Authorization", "Bearer " + token);
            }
        };
    }
}
```

在 `@EnableFeignClients(defaultConfiguration = FeignConfig.class)` 或对应 `@FeignClient(configuration = FeignConfig.class)` 中引用即可生效。

## 五、负载均衡

用**服务名**调用（RestTemplate 或 OpenFeign）时，「从多个实例里选一个」由 **Spring Cloud LoadBalancer** 完成。本节说明其角色、如何启用以及常用策略。

### 5.1 客户端负载均衡

微服务中通常使用**客户端负载均衡**：调用方从注册中心拿到实例列表后，在本地按策略选出一个实例再发请求（而非经过独立的负载均衡器代理）。

```
                    ┌─────────────────┐
                ┌──▶│ user-service-1  │
                │   └─────────────────┘
┌──────────┐    │   ┌─────────────────┐
│  调用方   │────┼──▶│ user-service-2  │
└──────────┘    │   └─────────────────┘
  (含负载       │   ┌─────────────────┐
   均衡器)      └──▶│ user-service-3  │
                    └─────────────────┘
```

### 5.2 Spring Cloud LoadBalancer

Spring Cloud 2020+ 使用 **LoadBalancer** 替代已停用的 Ribbon，负责「按服务名解析 + 多实例时选一个」。

#### 关键注解

| 注解              | 作用                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **@LoadBalanced** | 标记 RestTemplate、WebClient 等 Bean，使其在请求「服务名」时通过 LoadBalancer 解析并做负载均衡；不加则只能用具体 IP:端口，无法走注册中心。 |

#### 如何生效

| 使用方式         | 启用方式                                                                                                                                               |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **RestTemplate** | 在 RestTemplate 的 `@Bean` 上标注 **@LoadBalanced**，该 Bean 发出的请求才会走 LoadBalancer。                                                           |
| **OpenFeign**    | 无需额外注解；classpath 中存在 `spring-cloud-starter-loadbalancer` 时，Feign 会自动用 LoadBalancer 解析 `@FeignClient(name = "服务名")` 并做负载均衡。 |

#### 依赖

使用 RestTemplate 做服务调用时，若未由其他 starter 传递引入，需显式添加：

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-loadbalancer</artifactId>
</dependency>
```

使用 `spring-cloud-starter-openfeign` 时，通常已传递引入，无需重复添加。

#### 自定义负载均衡策略示例

```java
@Configuration
public class LoadBalancerConfig {
    
    // 配置随机负载均衡策略
    @Bean
    ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment environment,
            LoadBalancerClientFactory clientFactory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            clientFactory.getLazyProvider(name, ServiceInstanceListSupplier.class), 
            name
        );
    }
}
```

### 5.3 常见负载均衡策略

不自定义时，默认使用 **RoundRobin**（轮询）。常用策略如下：

| 策略                     | 说明               |
| ------------------------ | ------------------ |
| **RoundRobin**           | 轮询（默认）       |
| **Random**               | 随机               |
| **WeightedResponseTime** | 根据响应时间加权   |
| **BestAvailable**        | 选择并发最小的实例 |
| **Retry**                | 带重试的轮询       |

## 六、服务网关

### 6.1 网关的作用

**服务网关** 是微服务架构的统一入口，负责路由转发、认证与限流等。

| 功能         | 说明                         |
| ------------ | ---------------------------- |
| **统一入口** | 所有请求通过网关进入         |
| **路由转发** | 根据规则将请求转发到对应服务 |
| **认证授权** | 统一的安全验证               |
| **限流熔断** | 保护后端服务                 |
| **日志监控** | 统一的请求日志               |

```
                        ┌─────────────┐
                    ┌──▶│ user-service │
┌────────┐    ┌─────────┐   └─────────────┘
│ Client │───▶│ Gateway │──▶┌─────────────┐
└────────┘    └─────────┘   │order-service│
                    │       └─────────────┘
                    └──▶┌─────────────────┐
                        │ product-service │
                        └─────────────────┘
```

### 6.2 Spring Cloud Gateway

#### 1. 添加依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
```

#### 2. 路由属性

网关路由在 Java 中对应类型为 **`RouteDefinition`**，配置时常用的属性如下：

| 属性           | 说明                                   |
| -------------- | -------------------------------------- |
| **id**         | 路由唯一标识                           |
| **uri**        | 路由目标地址（如 `lb://user-service`） |
| **predicates** | 路由断言，判断请求是否匹配当前路由     |
| **filters**    | 路由过滤器，对请求或响应做特殊处理     |

#### 3. 路由配置

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 用户服务路由
        - id: user-service
          uri: lb://user-service       # lb 表示负载均衡
          predicates:
            - Path=/api/users/**       # 路径匹配
          filters:
            - StripPrefix=1            # 去掉前缀 /api
            
        # 订单服务路由
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Request-Source, gateway
```

#### 4. 网关请求处理流程

请求进入网关后的处理链路：

```
客户端请求 → HandlerMapping（路由映射）→ WebHandler（过滤器链）→ 微服务
```

| 环节           | 说明                                                                 |
| -------------- | -------------------------------------------------------------------- |
| **HandlerMapping** | 默认实现为 `RoutePredicateHandlerMapping`，根据请求匹配路由并放入上下文 |
| **WebHandler**     | 默认实现为 `FilteringWebHandler`，加载并排序过滤器，形成过滤器链     |
| **过滤器执行**     | pre 阶段：各 Filter 的 pre 逻辑**顺序**执行，全部通过后才转发到微服务 |
| **转发与 post**    | 由 **NettyRoutingFilter** 将请求转发到微服务；返回后 **倒序**执行各 Filter 的 post 逻辑 |

#### 5. 断言工厂（Predicate）

Spring 提供多种 `RoutePredicateFactory`，常用如下：

| 断言          | 说明             | 示例                                  |
| ------------- | ---------------- | ------------------------------------- |
| `Path`        | 路径匹配         | `Path=/api/**`                        |
| `Method`      | HTTP 方法        | `Method=GET,POST`                     |
| `Header`      | 请求头匹配       | `Header=X-Token, \d+`                 |
| `Query`       | 请求参数         | `Query=name, jack` 或 `Query=name`    |
| `Cookie`      | 必须包含某 Cookie | `Cookie=chocolate, ch.p`              |
| `Host`        | 访问的域名       | `Host=**.somehost.org`                 |
| `RemoteAddr`  | 请求者 IP 范围   | `RemoteAddr=192.168.1.1/24`           |
| `After`       | 某时间点之后     | `After=2024-01-01T00:00:00+08:00[Asia/Shanghai]` |
| `Before`      | 某时间点之前     | `Before=2024-12-31T23:59:59+08:00[Asia/Shanghai]` |
| `Between`     | 两时间点之间     | `Between=2024-01-01T00:00:00+08:00[Asia/Shanghai], 2024-12-31T23:59:59+08:00[Asia/Shanghai]` |
| `Weight`      | 权重路由         | `Weight=group1, 8`                     |

#### 6. 过滤器工厂（Filter）

网关提供多种路由过滤器，配置到具体路由后生效。常用如下：

| 过滤器               | 说明                 |
| -------------------- | -------------------- |
| `AddRequestHeader`   | 添加请求头           |
| `RemoveRequestHeader`| 移除请求头           |
| `AddResponseHeader`  | 添加响应头           |
| `RemoveResponseHeader` | 移除响应头         |
| `StripPrefix`        | 去除路径前 N 段       |
| `RewritePath`        | 重写路径             |
| `Retry`              | 重试                 |
| `CircuitBreaker`     | 熔断                 |

#### 7. GatewayFilter 与 GlobalFilter

| 类型             | 作用范围     | 生效方式                         |
| ---------------- | ------------ | -------------------------------- |
| **GatewayFilter** | 指定路由     | 需在路由的 `filters` 中配置才生效 |
| **GlobalFilter**  | 所有路由     | 声明为 Spring Bean 后自动生效    |

两者过滤方法签名一致：`Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain)`。pre 逻辑在 `chain.filter(exchange)` 之前执行；post 逻辑在之后通过 `.then(Mono.fromRunnable(...))` 执行。

#### 8. 全局过滤器（自定义）

自定义全局过滤器时常用注解：

| 注解           | 作用                                                                     |
| -------------- | ------------------------------------------------------------------------ |
| **@Component** | 将过滤器注册为 Spring Bean，由 Gateway 自动发现。                        |
| **@Order**     | 指定过滤器执行顺序，数值越小越先执行；如 -1 表示较早执行（常用于鉴权）。 |

```java
@Component
@Order(-1)  // 优先级，数值越小越先执行
public class AuthGlobalFilter implements GlobalFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        
        // 1. 获取 token
        String token = request.getHeaders().getFirst("Authorization");
        
        // 2. 验证 token
        if (token == null || !validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        // 3. 放行；post 阶段在微服务返回后倒序执行
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            // post：可在此记录日志、统计耗时等
        }));
    }
    
    private boolean validateToken(String token) {
        // 验证逻辑
        return true;
    }
}
```

## 八、服务熔断与降级

### 8.1 雪崩效应

**雪崩效应** 是指在微服务中，如果一个服务不可用，可能导致调用链上的所有服务都不可用。

```
服务A ──▶ 服务B ──▶ 服务C（故障）
   │         │
   ▼         ▼
 超时阻塞   超时阻塞
   │         │
   ▼         ▼
 资源耗尽   资源耗尽
   │
   ▼
 雪崩
```

**高可用常见手段：**

| 手段           | 说明                   |
| -------------- | ---------------------- |
| **多实例部署** | 避免单点故障           |
| **负载均衡**   | 分散压力               |
| **熔断降级**   | 快速失败，防止级联故障 |
| **限流保护**   | 控制流量，保护服务     |
| **健康检查**   | 及时发现异常实例       |

### 8.2 Sentinel（推荐）

**Sentinel** 是阿里巴巴开源的流量控制组件。

#### 关键注解

| 注解                                           | 作用                                                                                                                        |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **@SentinelResource**                          | 定义受 Sentinel 保护的资源。`value` 为资源名；`blockHandler` 处理限流/熔断（BlockException）；`fallback` 处理业务异常降级。 |
| **@FeignClient 的 fallback / fallbackFactory** | Feign 与 Sentinel 集成时，指定限流或调用失败时的降级实现类；`fallbackFactory` 可拿到异常原因。                              |

#### 1. 添加依赖

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

#### 2. 配置

```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080  # Sentinel 控制台
        port: 8719
```

#### 3. 资源定义与降级

```java
@Service
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        blockHandler = "getOrderBlockHandler",   // 限流/熔断处理
        fallback = "getOrderFallback"            // 业务异常处理
    )
    public Order getOrder(Long orderId) {
        return orderRepository.findById(orderId);
    }
    
    // 限流/熔断时的处理（BlockException 参数必须有）
    public Order getOrderBlockHandler(Long orderId, BlockException ex) {
        return new Order().setMessage("服务繁忙，请稍后重试");
    }
    
    // 业务异常时的降级处理
    public Order getOrderFallback(Long orderId, Throwable ex) {
        return new Order().setMessage("服务暂时不可用");
    }
}
```

#### 4. Feign 集成 Sentinel

```yaml
feign:
  sentinel:
    enabled: true  # 开启 Feign 的 Sentinel 支持
```

```java
// Feign 接口
@FeignClient(name = "user-service", fallbackFactory = UserClientFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
}

// 降级工厂
@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUserById(Long id) {
                log.error("调用用户服务失败: {}", cause.getMessage());
                return new User().setName("未知用户");
            }
        };
    }
}
```

### 8.3 Sentinel 核心功能

| 功能             | 说明                       |
| ---------------- | -------------------------- |
| **流量控制**     | QPS / 线程数限流           |
| **熔断降级**     | 慢调用 / 异常比例 / 异常数 |
| **热点参数限流** | 针对特定参数限流           |
| **系统保护**     | CPU / 负载 / 入口 QPS      |
| **授权规则**     | 黑白名单控制               |

### 8.4 Resilience4j（了解）

Spring Cloud 官方推荐的熔断/重试/限流组件（可替代 Hystrix），与 Sentinel 二选一。常用注解：

| 注解                | 作用                                                    |
| ------------------- | ------------------------------------------------------- |
| **@CircuitBreaker** | 熔断：失败率等达到阈值后短路，直接走 `fallbackMethod`。 |
| **@Retry**          | 失败重试，可配次数与间隔。                              |
| **@RateLimiter**    | 限流，控制调用速率。                                    |

```java
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
@Retry(name = "userService")
@RateLimiter(name = "userService")
public User getUser(Long userId) {
    return userClient.getUserById(userId);
}

public User getUserFallback(Long userId, Exception ex) {
    return new User().setName("默认用户");
}
```

## 九、分布式事务

### 9.1 分布式事务问题

微服务中每个服务有独立数据库，跨服务的事务无法用本地事务保证。

```
订单服务（创建订单） ──▶ 库存服务（扣减库存） ──▶ 账户服务（扣款）
    │                        │                       │
   DB1                      DB2                     DB3
```

**CAP 理论：** 分布式系统最多同时满足其中两个：

| 特性                                | 说明               |
| ----------------------------------- | ------------------ |
| **C**onsistency（一致性）           | 所有节点数据一致   |
| **A**vailability（可用性）          | 服务始终可用       |
| **P**artition tolerance（分区容错） | 网络分区时仍能运行 |

> 💡 通常选择 AP（最终一致性）或 CP（强一致性）。

### 9.2 Seata 分布式事务

**Seata** 是阿里巴巴开源的分布式事务解决方案。

#### 1. 事务模式

| 模式     | 说明               | 适用场景           |
| -------- | ------------------ | ------------------ |
| **AT**   | 自动补偿，无侵入   | 大多数场景（推荐） |
| **TCC**  | Try-Confirm-Cancel | 高性能场景         |
| **SAGA** | 长事务             | 业务流程长         |
| **XA**   | 强一致性           | 传统数据库         |

#### 关键注解（AT 模式）

| 注解                     | 作用                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------- |
| **@GlobalTransactional** | 声明分布式事务边界；加在业务方法上，方法内跨服务的调用由 Seata 协调，任一步失败则全局回滚。 |

#### 2. AT 模式使用

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

```java
@Service
public class OrderService {
    
    @Autowired
    private StockClient stockClient;
    
    @Autowired
    private AccountClient accountClient;
    
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单
        orderMapper.insert(orderDTO);
        
        // 2. 扣减库存（远程调用）
        stockClient.deduct(orderDTO.getProductId(), orderDTO.getCount());
        
        // 3. 扣减余额（远程调用）
        accountClient.debit(orderDTO.getUserId(), orderDTO.getMoney());
        
        // 任意一步失败，全部回滚
    }
}
```

#### 关键注解（TCC 模式）

| 注解                                | 作用                                                                           |
| ----------------------------------- | ------------------------------------------------------------------------------ |
| **@LocalTCC**                       | 标记接口为 TCC 模式，需提供 try/confirm/cancel 两阶段方法。                    |
| **@TwoPhaseBusinessAction**         | 标注 try 方法，并指定 `commitMethod`、`rollbackMethod` 对应 confirm/rollback。 |
| **@BusinessActionContextParameter** | 将参数放入上下文，供 confirm/rollback 方法使用。                               |

#### 3. TCC 模式

```java
@LocalTCC
public interface StockTccService {
    
    @TwoPhaseBusinessAction(
        name = "deductStock",
        commitMethod = "commit",
        rollbackMethod = "rollback"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    // 确认方法
    boolean commit(BusinessActionContext context);
    
    // 回滚方法
    boolean rollback(BusinessActionContext context);
}
```

        orderMapper.insert(order);
        // 发送订单创建事件

    }
}

@Component
public class StockEventHandler {
    
    @Bean
    public Consumer<Order> orderCreated() {
        return order -> {
            // 扣减库存
            stockService.deduct(order.getProductId(), order.getCount());
        };
    }

## 十三、微服务最佳实践

### 13.1 服务拆分原则

**什么时候拆分：**

| 项目类型     | 建议                                           |
| ------------ | ---------------------------------------------- |
| **创业型项目** | 先采用单体架构快速开发、试错，规模扩大后再拆分 |
| **确定的大型项目** | 可直接选微服务架构，避免后期拆分的成本       |

**拆分目标：** 高内聚、低耦合。**高内聚**指每个微服务职责尽量单一、业务关联度高；**低耦合**指功能相对独立，减少对其它服务的依赖。

**拆分方式：**

| 方式         | 说明                         |
| ------------ | ---------------------------- |
| **纵向拆分** | 按业务模块拆（如用户、订单） |
| **横向拆分** | 抽取公共服务，提高复用       |

| 原则           | 说明                   |
| -------------- | ---------------------- |
| **单一职责**   | 每个服务只做一件事     |
| **业务边界**   | 按业务领域划分（DDD）  |
| **数据独立**   | 每个服务拥有独立数据库 |
| **接口稳定**   | 服务间通过稳定接口通信 |
| **渐进式拆分** | 从单体逐步拆分         |

### 13.2 工程结构

微服务可按两种方式组织代码：

| 方式           | 说明                                       |
| -------------- | ------------------------------------------ |
| **独立 Project** | 每个微服务一个独立工程，互不依赖，便于独立部署 |
| **Maven 聚合**   | 父工程下多个 module，统一依赖版本、一键构建   |

按业务拆出的模块（如 `item-service`、`cart-service`）可作为父工程下的子 module，便于统一管理。

### 13.3 服务间通信原则

| 场景         | 推荐方式                     |
| ------------ | ---------------------------- |
| **同步调用** | OpenFeign + LoadBalancer     |
| **异步解耦** | 消息队列（RocketMQ / Kafka） |
| **事件驱动** | Spring Cloud Stream          |

### 13.4 微服务登录与用户传递

网关统一做登录校验，校验通过后将用户信息传给下游，微服务内和跨服务调用时都能拿到当前用户。

**① 网关：校验后写入请求头**

在网关的 `GlobalFilter` 中校验 JWT，通过后把用户信息放入转发请求的请求头，再放行：

```java
// 修改转发到微服务的请求，添加请求头
ServerWebExchange newExchange = exchange.mutate()
    .request(builder -> builder.header("user-info", userInfoJson))
    .build();
return chain.filter(newExchange);
```

**② 微服务：拦截器取用户并放入 ThreadLocal**

在公共模块（如 `hm-common`）中定义 Spring MVC 拦截器，从请求头读取 `user-info`，解析后放入 ThreadLocal，业务层从 ThreadLocal 获取当前用户。

**③ OpenFeign 调用时传递用户**

微服务间通过 OpenFeign 调用时，需把当前用户带到下游。使用 **RequestInterceptor** 统一在请求头中携带用户信息：

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String userInfo = UserContext.getUserInfo();  // 从 ThreadLocal 获取
            if (userInfo != null) {
                template.header("user-info", userInfo);
            }
        };
    }
}
```

| 环节         | 做法                                       |
| ------------ | ------------------------------------------ |
| **网关**     | GlobalFilter 校验 JWT，写 `user-info` 到请求头 |
| **微服务内** | HandlerInterceptor 读请求头，写入 ThreadLocal   |
| **Feign 调用** | RequestInterceptor 从 ThreadLocal 取用户，写入请求头 |
