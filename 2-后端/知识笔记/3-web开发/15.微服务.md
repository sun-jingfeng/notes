# Java 微服务

## 一、微服务概述

### 1.1 什么是微服务

微服务（Microservices）是一种软件架构风格，它将应用程序构建为一组小型、独立部署的服务集合。每个服务：

- 运行在自己的进程中
- 围绕特定的业务功能构建
- 通过轻量级机制（通常是 HTTP/REST 或消息队列）进行通信
- 可以独立部署、扩展和维护

### 1.2 单体架构 vs 微服务架构

| 特性     | 单体架构         | 微服务架构         |
| -------- | ---------------- | ------------------ |
| 部署     | 整体部署         | 独立部署           |
| 技术栈   | 统一技术栈       | 可以使用不同技术栈 |
| 扩展性   | 整体扩展         | 按需扩展单个服务   |
| 开发效率 | 初期快，后期慢   | 初期慢，后期快     |
| 容错性   | 一处故障影响全局 | 故障隔离           |
| 团队协作 | 耦合度高         | 团队独立开发       |

### 1.3 微服务的优缺点

**优点：**
- 服务独立部署，快速迭代
- 技术栈灵活，可根据业务选择合适技术
- 按需扩展，资源利用率高
- 故障隔离，提高系统可用性
- 团队自治，提高开发效率

**缺点：**
- 分布式系统复杂性增加
- 运维成本提高
- 服务间通信开销
- 数据一致性难以保证
- 测试复杂度增加

## 三、服务注册与发现

### 3.1 为什么需要服务注册发现

在微服务架构中，服务实例动态变化（扩缩容、故障恢复），需要一个中心化的注册中心来管理服务实例信息。

```
┌──────────┐      注册        ┌──────────────┐
│ Service A│ ──────────────▶  │              │
└──────────┘                  │   注册中心    │
                              │  (Nacos等)   │
┌──────────┐      发现        │              │
│ Service B│ ◀──────────────  └──────────────┘
└──────────┘
```

### 3.2 Nacos 服务注册

**1. 添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

**2. 配置 application.yml**

```yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        namespace: dev  # 命名空间，用于环境隔离
        group: DEFAULT_GROUP
```

**3. 启用服务发现**

```java
@SpringBootApplication
@EnableDiscoveryClient
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

## 四、服务调用

### 4.1 RestTemplate

最基础的 HTTP 调用方式：

```java
@Configuration
public class RestTemplateConfig {
    @Bean
    @LoadBalanced  // 启用负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

@Service
public class OrderService {
    @Autowired
    private RestTemplate restTemplate;
    
    public User getUser(Long userId) {
        // 使用服务名代替具体IP
        String url = "http://user-service/users/" + userId;
        return restTemplate.getForObject(url, User.class);
    }
}
```

### 4.2 OpenFeign（推荐）

声明式 HTTP 客户端，更优雅的服务调用方式。

**1. 添加依赖**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
```

**2. 启用 Feign**

```java
@SpringBootApplication
@EnableFeignClients
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

**3. 定义 Feign 接口**

```java
@FeignClient(name = "user-service", fallback = UserClientFallback.class)
public interface UserClient {
    
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
    
    @PostMapping("/users")
    User createUser(@RequestBody User user);
    
    @GetMapping("/users")
    List<User> getUsers(@RequestParam("ids") List<Long> ids);
}
```

**4. 使用 Feign 客户端**

```java
@Service
public class OrderService {
    @Autowired

    public OrderVO getOrderDetail(Long orderId) {
        Order order = orderRepository.findById(orderId);
        // 调用用户服务
        User user = userClient.getUserById(order.getUserId());
        return new OrderVO(order, user);
    }
}
```

**5. Feign 配置**

        connectTimeout: 5000
        readTimeout: 5000
        loggerLevel: BASIC

      enabled: true

      enabled: true

## 五、负载均衡

### 5.1 客户端负载均衡

微服务中通常使用客户端负载均衡，由调用方决定调用哪个实例。

```
                    ┌─────────────────┐
                ┌──▶│ user-service-1  │
                │   └─────────────────┘
┌──────────┐    │   ┌─────────────────┐
│ 调用方   │────┼──▶│ user-service-2  │
└──────────┘    │   └─────────────────┘
   (含负载     │   ┌─────────────────┐
    均衡器)    └──▶│ user-service-3  │
                    └─────────────────┘
```

### 5.2 Spring Cloud LoadBalancer

Spring Cloud 2020+ 版本使用 LoadBalancer 替代 Ribbon。

```java
@Configuration
public class LoadBalancerConfig {
    
    // 配置随机负载均衡策略
    @Bean
    ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment environment,
            LoadBalancerClientFactory clientFactory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            clientFactory.getLazyProvider(name, ServiceInstanceListSupplier.class), 
            name
        );
    }
}
```

### 5.3 常见负载均衡策略

| 策略                 | 说明               |
| -------------------- | ------------------ |
| RoundRobin           | 轮询（默认）       |
| Random               | 随机               |
| WeightedResponseTime | 根据响应时间加权   |
| BestAvailable        | 选择并发最小的实例 |
| Retry                | 带重试的轮询       |

### 6.1 网关的作用

- **统一入口**：所有请求通过网关进入
- **路由转发**：根据规则将请求转发到对应服务
- **认证授权**：统一的安全验证
- **限流熔断**：保护后端服务
- **日志监控**：统一的请求日志

```
                        ┌─────────────┐
                    ┌──▶│ user-service │
┌────────┐    ┌─────────┐   └─────────────┘
│ Client │───▶│ Gateway │──▶┌─────────────┐
└────────┘    └─────────┘   │order-service│
                    │       └─────────────┘
                    └──▶┌─────────────────┐
                        │ product-service │
                        └─────────────────┘
```

### 6.2 Spring Cloud Gateway

**1. 添加依赖**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
```

**2. 路由配置**

```yaml
spring:
  cloud:
    gateway:
      routes:
        # 用户服务路由
        - id: user-service
          uri: lb://user-service  # lb表示负载均衡
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1  # 去掉前缀 /api
            
        # 订单服务路由
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Request-Source, gateway
```

**3. 断言工厂（Predicate）**

```yaml
predicates:
  - Path=/api/**           # 路径匹配
  - Method=GET,POST        # HTTP方法
  - Header=X-Token, \d+    # 请求头匹配
  - Query=name, jack       # 请求参数
  - After=2024-01-01T00:00:00.000+08:00  # 时间之后
  - Weight=group1, 8       # 权重路由
```

**4. 过滤器工厂（Filter）**

```yaml
filters:
  - AddRequestHeader=X-Request-Id, {random.uuid}
  - AddResponseHeader=X-Response-Time, {now}
  - RewritePath=/api/(?<segment>.*), /$\{segment}
  - Retry=3
  - CircuitBreaker=myCircuitBreaker
```

**5. 全局过滤器（自定义）**

## 八、服务熔断与降级

### 8.1 雪崩效应

在微服务中，如果一个服务不可用，可能导致调用链上的所有服务都不可用，形成雪崩效应。

```
服务A ──▶ 服务B ──▶ 服务C（故障）
   │         │
   ▼         ▼
 超时阻塞   超时阻塞
   │         │
   ▼         ▼
 资源耗尽   资源耗尽
   │
   ▼
 雪崩
```

### 8.2 Sentinel（推荐）

阿里巴巴开源的流量控制组件。

**1. 添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

**2. 配置**

```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080  # Sentinel控制台
        port: 8719
```

**3. 资源定义与降级**

```java
@Service
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        blockHandler = "getOrderBlockHandler",
        fallback = "getOrderFallback"
    )
    public Order getOrder(Long orderId) {
        return orderRepository.findById(orderId);
    }
    
    // 限流/熔断时的处理
    public Order getOrderBlockHandler(Long orderId, BlockException ex) {
        return new Order().setMessage("服务繁忙，请稍后重试");
    }
    
    // 业务异常时的降级处理
    public Order getOrderFallback(Long orderId, Throwable ex) {
        return new Order().setMessage("服务暂时不可用");
    }
}
```

**4. Feign 集成 Sentinel**

```yaml
feign:
  sentinel:
    enabled: true
```

```java
@FeignClient(name = "user-service", fallbackFactory = UserClientFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
}

@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUserById(Long id) {
                log.error("调用用户服务失败: {}", cause.getMessage());
                return new User().setName("未知用户");
            }
        };
    }
}
```

### 8.3 Sentinel 核心功能

| 功能         | 说明                   |
| ------------ | ---------------------- |
| 流量控制     | QPS/线程数限流         |
| 熔断降级     | 慢调用/异常比例/异常数 |
| 热点参数限流 | 针对特定参数限流       |
| 系统保护     | CPU/负载/入口QPS       |
| 授权规则     | 黑白名单控制           |

### 8.4 Resilience4j

Spring Cloud 官方推荐的熔断器（替代 Hystrix）。

```java
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
@Retry(name = "userService")
@RateLimiter(name = "userService")
public User getUser(Long userId) {
    return userClient.getUserById(userId);
}

public User getUserFallback(Long userId, Exception ex) {
    return new User().setName("默认用户");
}
```

## 九、分布式事务

### 9.1 分布式事务问题

微服务中每个服务有独立数据库，跨服务的事务无法用本地事务保证。

```
订单服务（创建订单） ──▶ 库存服务（扣减库存） ──▶ 账户服务（扣款）
    │                        │                       │
   DB1                      DB2                     DB3
```

### 9.2 Seata 分布式事务

**1. 事务模式**

| 模式 | 说明               | 适用场景           |
| ---- | ------------------ | ------------------ |
| AT   | 自动补偿，无侵入   | 大多数场景（推荐） |
| TCC  | Try-Confirm-Cancel | 高性能场景         |
| SAGA | 长事务             | 业务流程长         |
| XA   | 强一致性           | 传统数据库         |

**2. AT 模式使用**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

```java
@Service
public class OrderService {
    
    @Autowired
    private StockClient stockClient;
    
    @Autowired
    private AccountClient accountClient;
    
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单
        orderMapper.insert(orderDTO);
        
        // 2. 扣减库存（远程调用）
        stockClient.deduct(orderDTO.getProductId(), orderDTO.getCount());
        
        // 3. 扣减余额（远程调用）
        accountClient.debit(orderDTO.getUserId(), orderDTO.getMoney());
    }
}
```

**3. TCC 模式**

```java
@LocalTCC
public interface StockTccService {
    
    @TwoPhaseBusinessAction(
        name = "deductStock",
        commitMethod = "commit",
        rollbackMethod = "rollback"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    boolean commit(BusinessActionContext context);
    
    boolean rollback(BusinessActionContext context);
}
```

## 十三、微服务最佳实践

### 13.1 服务拆分原则

1. **单一职责**：每个服务只做一件事
2. **业务边界**：按业务领域划分（DDD）
3. **数据独立**：每个服务拥有独立数据库
4. **接口稳定**：服务间通过稳定接口通信
5. **渐进式拆分**：从单体逐步拆分

### 13.2 服务间通信原则

| 场景     | 推荐方式                   |
| -------- | -------------------------- |
| 同步调用 | OpenFeign + LoadBalancer   |
| 异步解耦 | 消息队列（RocketMQ/Kafka） |
| 事件驱动 | Spring Cloud Stream        |

### 13.3 安全实践

```java
// JWT 认证
@Configuration
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt());
        return http.build();
    }
}
```

### 13.4 可观测性

- **日志**：ELK（Elasticsearch + Logstash + Kibana）
- **指标**：Prometheus + Grafana
- **追踪**：Zipkin / SkyWalking

## 十四、常见面试题

### Q1: 微服务和 SOA 的区别？

- SOA：企业级，ESB 集中管理，服务粒度大
- 微服务：轻量级，去中心化，服务粒度小

### Q2: 如何保证微服务的高可用？

- 服务多实例部署
- 负载均衡
- 熔断降级
- 限流保护
- 健康检查

### Q3: 微服务如何处理分布式事务？

- Seata（AT/TCC/SAGA 模式）
- 最终一致性（消息队列）
- 补偿事务

### Q4: CAP 理论是什么？

- **C**onsistency（一致性）
- **A**vailability（可用性）
- **P**artition tolerance（分区容错）

分布式系统只能同时满足其中两个，通常选择 AP 或 CP。

### Q5: 服务发现的两种模式？

- **客户端发现**：客户端从注册中心获取服务列表，自己做负载均衡
- **服务端发现**：通过负载均衡器代理，客户端无感知
