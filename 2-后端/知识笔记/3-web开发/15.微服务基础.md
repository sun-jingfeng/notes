## 一、微服务概述

### 1.1 什么是微服务

**微服务（Microservices）** 是一种软件架构风格，将应用程序构建为一组小型、独立部署的服务集合。

**核心特点：**

| 特点         | 说明                              |
| ------------ | --------------------------------- |
| **独立进程** | 每个服务运行在自己的进程中        |
| **业务聚焦** | 围绕特定的业务功能构建            |
| **轻量通信** | 通过 HTTP/REST 或消息队列进行通信 |
| **独立部署** | 可以独立部署、扩展和维护          |

### 1.2 架构对比

#### 单体架构 vs 微服务架构

| 维度         | 单体架构         | 微服务架构                       | 微服务代价                     |
| ------------ | ---------------- | -------------------------------- | ------------------------------ |
| **部署**     | 整体部署         | 独立部署，快速迭代               | 运维复杂，需容器化与自动化支持 |
| **技术栈**   | 统一技术栈       | 各服务可独立选型                 | 技术异构增加学习与维护成本     |
| **扩展性**   | 整体扩展         | 按需扩展单个服务                 | 分布式带来网络通信与延迟开销   |
| **容错性**   | 一处故障影响全局 | 故障隔离                         | 跨服务数据一致性难保证         |
| **开发效率** | 初期快，后期慢   | 初期慢，后期快；团队自治独立开发 | 集成测试与端到端测试更复杂     |

#### 微服务 vs SOA

| 对比项       | SOA          | 微服务   |
| ------------ | ------------ | -------- |
| **管理方式** | ESB 集中管理 | 去中心化 |
| **服务粒度** | 较大         | 较小     |
| **部署**     | 通常整体部署 | 独立部署 |
| **技术选型** | 统一         | 灵活     |

## 二、Spring Cloud 生态系统

**Spring Cloud** 是 Java 微服务的主流解决方案，基于 Spring Boot 构建。

### 2.1 核心组件概览

```
┌─────────────────────────────────────────────────────────────┐
│                     Spring Cloud 生态                        │
├─────────────────────────────────────────────────────────────┤
│  服务注册发现: Nacos / Eureka / Consul                       │
│  服务调用: OpenFeign / RestTemplate                          │
│  负载均衡: LoadBalancer / Ribbon                             │
│  服务网关: Spring Cloud Gateway / Zuul                       │
│  配置中心: Nacos Config / Spring Cloud Config                │
│  服务熔断: Sentinel / Resilience4j / Hystrix                 │
│  分布式事务: Seata                                           │
│  链路追踪: Sleuth + Zipkin / SkyWalking                      │
│  消息总线: Spring Cloud Bus                                  │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **Ribbon**、**Zuul**、**Hystrix** 已进入维护或停用，当前推荐：负载均衡用 **LoadBalancer**、网关用 **Spring Cloud Gateway**、熔断用 **Sentinel** 或 **Resilience4j**。

### 2.2 Spring Cloud Alibaba

目前常用的微服务解决方案，与 Nacos、Sentinel、Seata 等配套使用。主要组件：

| 组件         | 功能                    |
| ------------ | ----------------------- |
| **Nacos**    | 服务注册发现 + 配置中心 |
| **Sentinel** | 流量控制、熔断降级      |
| **Seata**    | 分布式事务              |
| **RocketMQ** | 消息队列                |
| **Dubbo**    | RPC 框架                |

### 2.3 Spring Cloud 与 Spring Boot 版本对应

## 三、服务注册与发现

### 3.1 服务注册发现原理

微服务架构中，服务实例动态变化（扩缩容、故障恢复），需要注册中心统一管理实例信息。

```
┌──────────┐      注册        ┌──────────────┐
│ Service A│ ──────────────▶  │              │
└──────────┘                  │   注册中心    │
                              │  (Nacos等)   │
┌──────────┐      发现        │              │
│ Service B│ ◀──────────────  └──────────────┘
└──────────┘
```

**服务治理中的三个角色：**

| 角色           | 说明                                                             |
| -------------- | ---------------------------------------------------------------- |
| **服务提供者** | 暴露服务接口，供其他服务调用                                     |
| **服务消费者** | 调用其他服务提供的接口                                           |
| **注册中心**   | 记录并监控各微服务实例状态，推送服务变更信息（如 Nacos、Eureka） |

| 问题                                       | 机制                                                               |
| ------------------------------------------ | ------------------------------------------------------------------ |
| 消费者如何知道提供者的地址？               | 提供者启动时注册到注册中心；消费者从注册中心订阅并拉取服务信息     |
| 消费者如何得知服务状态变更？               | 提供者通过**心跳**上报健康状态；异常时注册中心剔除实例并通知消费者 |
| 当提供者有多个实例时，消费者该选择哪一个？ | 通过**负载均衡**算法，从多个实例中选出一个进行调用                 |

> 💡 **项目中选其一即可**：Nacos（Spring Cloud Alibaba 推荐）、Eureka（Netflix 方案，多用于老项目）。

### 3.2 Nacos 服务注册

先部署并启动 Nacos **服务端**，项目内引入的是 Nacos **客户端**，用于连接该服务端。

#### 部署 Nacos 服务端

**方式一：Docker（推荐）**

```bash
docker run -d \
  --name nacos \
  -p 8848:8848 \
  -p 9848:9848 \
  -e MODE=standalone \
  -e JVM_XMS=256m \
  -e JVM_XMX=256m \
  -e NACOS_AUTH_ENABLE=true \
  -e NACOS_AUTH_IDENTITY_KEY=serverIdentity \
  -e NACOS_AUTH_IDENTITY_VALUE=security \
  nacos/nacos-server:v2.5.2
```

> 💡 开启认证时，Nacos 2.x 必须配置 `NACOS_AUTH_IDENTITY_KEY` 和 `NACOS_AUTH_IDENTITY_VALUE`，否则启动报错 Empty identity。Mac Apple Silicon（M1/M2/M3）需在 `docker run -d` 后增加：`--platform linux/amd64`。

- 控制台：http://localhost:8848/nacos ，默认账号/密码 `nacos`
- 日常使用：`docker start nacos` 启动，`docker stop nacos` 停止

**方式二：官网安装包**

从 [Nacos releases](https://github.com/alibaba/nacos/releases) 下载 `nacos-server-xxx.tar.gz`，解压后执行：

```bash
# Linux/Mac
sh bin/startup.sh -m standalone

# Windows
bin/startup.cmd -m standalone
```

#### 项目接入步骤

**1. 添加依赖**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

**2. 配置 application.yml**

```yaml
spring:
  application:
    name: order-service           # 服务名称
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848  # Nacos 服务端地址
        namespace: dev               # 命名空间，用于环境隔离
        group: DEFAULT_GROUP         # 分组
```

| 配置项        | 作用         | 典型用法                                                             |
| ------------- | ------------ | -------------------------------------------------------------------- |
| **namespace** | 环境隔离     | dev、test、prod 各用一个命名空间，不同命名空间的服务互不可见         |
| **group**     | 同环境内分组 | 默认 `DEFAULT_GROUP`；同一命名空间下可按团队或项目分组，缩小发现范围 |

**3. 启用服务发现**

| 注解                       | 作用                                                                                             |
| -------------------------- | ------------------------------------------------------------------------------------------------ |
| **@EnableDiscoveryClient** | 启用服务发现客户端，使当前应用向注册中心注册并拉取服务列表；加在启动类上，Nacos、Eureka 等通用。 |

```java
@SpringBootApplication
@EnableDiscoveryClient  // 启用服务发现
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

> 💡 Spring Cloud 2021+ 起，若 classpath 中存在 `spring-cloud-starter-alibaba-nacos-discovery`，服务发现通过自动配置启用，`@EnableDiscoveryClient` 可省略。

### 3.3 Eureka 服务注册（Netflix 方案，可选阅读）

与 Nacos 类似：需先部署 **Eureka Server**，再在业务服务中配置 Client。Server 需关闭“注册自己”和“拉取注册表”，只作为注册中心；Client 配置 `defaultZone` 指向 Server 地址，可选 `prefer-ip-address: true` 用 IP 注册。

#### Eureka Server 配置

```yaml
server:
  port: 8761

eureka:
  instance:
    hostname: localhost
  client:
    register-with-eureka: false   # Server 不注册自己
    fetch-registry: false         # Server 不拉取服务
    service-url:
      defaultZone: http://${eureka.instance.hostname}:${server.port}/eureka/
```

#### Eureka Client 配置

```yaml
eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
  instance:
    prefer-ip-address: true  # 使用 IP 地址注册
```

## 四、服务调用

服务间通过 HTTP 进行远程调用，可使用 RestTemplate 或声明式的 OpenFeign。

### 4.1 服务发现与负载均衡原理

**用服务名调用时的底层流程：**

用**服务名**（如 `http://user-service/...`）发起请求时，底层依次完成三件事：从注册中心拉取该服务的实例列表 → 按负载均衡策略选出一个实例 → 用该实例的 IP:端口 发起真实 HTTP 请求。其中「选实例」由 **Spring Cloud LoadBalancer** 完成（多实例时默认轮询）。

> 💡 实际开发中一般用 **RestTemplate + @LoadBalanced** 或 **OpenFeign**，由框架完成上述流程。若需手写“拉取实例 + 选实例”逻辑，可用 `DiscoveryClient`：

```java
@Service
public class CartService {
    
    @Autowired
    private DiscoveryClient discoveryClient;

    private void handleCartItems(List<CartVO> vos) {
        // 1. 根据服务名称，拉取该服务的实例列表
        List<ServiceInstance> instances = discoveryClient.getInstances("item-service");
        if (instances == null || instances.isEmpty()) {
            throw new RuntimeException("item-service 无可用实例");
        }
        
        // 2. 负载均衡：从多个实例中选一个（此处为随机）
        ServiceInstance instance = instances.get(
            new Random().nextInt(instances.size())
        );
        
        // 3. 获取该实例的地址
        URI uri = instance.getUri();
        
        // 4. 用 uri 发起 HTTP 调用
        // restTemplate.getForObject(uri + "/items/" + id, Item.class);
    }
}
```

### 4.2 RestTemplate

**RestTemplate** 是 Spring 提供的 HTTP 客户端。用**服务名**调用时，必须为该 RestTemplate 启用负载均衡：在声明 Bean 时加上 **@LoadBalanced**，框架才会用 LoadBalancer 解析服务名并选实例。

**远程调用的四要素：** 服务名（从注册中心拉取实例）、请求方式与路径、返回值类型、请求参数。RestTemplate 的 `getForObject(url, responseType)`、`exchange(url, method, entity, responseType, uriVariables)` 等即对应这些信息。① 在配置类中声明带 `@LoadBalanced` 的 RestTemplate Bean；② 在业务中注入并用服务名拼 URL 发起调用；③ 需要自定义请求头或获取状态码时用 `exchange`。

#### 1. 注入 RestTemplate 到 Spring 容器

```java
@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 启用 LoadBalancer，才能用服务名调用并做负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

#### 2. 发起远程调用（服务名 + 路径 + 返回值类型）

```java
@Service
public class OrderService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public User getUser(Long userId) {
        // 使用服务名代替具体 IP，由 LoadBalancer 解析并选实例
        String url = "http://user-service/users/" + userId;
        return restTemplate.getForObject(url, User.class);
    }
}
```

#### 3. exchange 方法（自定义请求头、获取状态码）

需要自定义请求头、获取响应状态码时使用 `exchange`，同样体现四要素：URL、方法、请求体/头、返回值类型。

```java
// 构建请求头
HttpHeaders headers = new HttpHeaders();
headers.set("Authorization", "Bearer token");
HttpEntity<Void> entity = new HttpEntity<>(headers);

// 发起请求
ResponseEntity<User> response = restTemplate.exchange(
    "http://user-service/users/" + userId,
    HttpMethod.GET,
    entity,
    User.class
);

// 获取结果
User user = response.getBody();
HttpStatus status = response.getStatusCode();
```

### 4.3 OpenFeign（推荐）

**OpenFeign** 是声明式 HTTP 客户端：用接口 + 注解描述“调用哪个服务的哪个接口”，由框架在运行时生成实现并完成服务发现、负载均衡和 HTTP 调用。核心依赖两个注解。

| 注解                    | 作用                                                                                                                                 |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| **@EnableFeignClients** | 加在启动类上，开启 Feign，扫描并注册 `@FeignClient` 接口；若 Feign 接口在其他包（如 API 模块），需指定 `basePackages` 或 `clients`。 |
| **@FeignClient**        | 声明式 HTTP 客户端接口。`name`/`value` 为服务名（走注册中心+负载均衡）；可配 `fallback`/`fallbackFactory` 做熔断降级。               |

添加依赖 → 启动类加 `@EnableFeignClients` → 定义 `@FeignClient` 接口（方法签名与提供方 REST 一致）→ 在业务中注入接口并像调用本地方法一样调用。

#### 1. 添加依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
```

#### 2. 启用 Feign

```java
@SpringBootApplication
@EnableFeignClients  // 启用 Feign 客户端
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

#### 3. 定义 Feign 接口

```java
@FeignClient(name = "user-service", fallback = UserClientFallback.class)
public interface UserClient {
    
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
    
    @PostMapping("/users")
    User createUser(@RequestBody User user);
    
    @GetMapping("/users")
    List<User> getUsers(@RequestParam("ids") List<Long> ids);
}
```

#### 4. 使用 Feign 客户端

```java
@Service
public class OrderService {
    
    @Autowired
    private UserClient userClient;
    
    public OrderVO getOrderDetail(Long orderId) {
        Order order = orderRepository.findById(orderId);
        // 像调用本地方法一样调用远程服务
        User user = userClient.getUserById(order.getUserId());
        return new OrderVO(order, user);
    }
}
```

#### 5. Feign 配置

```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 5000   # 连接超时
        readTimeout: 5000      # 读取超时
        loggerLevel: BASIC     # 日志级别：NONE/BASIC/HEADERS/FULL
  compression:
    request:
      enabled: true   # 请求压缩
    response:
      enabled: true   # 响应压缩
```

#### 6. 日志

Feign 可打印请求与响应，便于排查调用问题。无论用哪种方式设日志级别，都需在 **logging.level** 中将 Feign 接口所在包设为 **DEBUG**（Feign 以 DEBUG 级别输出，否则不打印）。

**日志级别**

| 级别        | 说明                                        |
| ----------- | ------------------------------------------- |
| **NONE**    | 不记录（默认）                              |
| **BASIC**   | 仅记录请求方法、URL、响应状态码、耗时       |
| **HEADERS** | 在 BASIC 基础上增加请求/响应头              |
| **FULL**    | 记录请求/响应头、正文，便于排查参数与返回值 |

**方式一：YAML 配置**

在 `feign.client.config` 中指定 `loggerLevel`，`default` 表示全局，也可按服务名单独配置：

```yaml
feign:
  client:
    config:
      default:
        loggerLevel: FULL
      user-service:
        loggerLevel: BASIC

logging:
  level:
    com.example.api.client: DEBUG   # Feign 接口所在包，必须设为 DEBUG
```

**方式二：Java 配置（Logger.Level Bean）**

定义配置类并声明 `Logger.Level` Bean，再通过注解指定生效范围：

```java
// 配置类
public class DefaultFeignConfig {
    @Bean
    public Logger.Level feignLogLevel() {
        return Logger.Level.FULL;
    }
}

// 仅对单个 Feign 客户端生效
@FeignClient(value = "item-service", configuration = DefaultFeignConfig.class)
public interface ItemClient { ... }

// 对所有 Feign 客户端生效（加在启动类上）
@EnableFeignClients(defaultConfiguration = DefaultFeignConfig.class)
@SpringBootApplication
public class OrderServiceApplication { ... }
```

- **单客户端**：在 `@FeignClient` 上写 `configuration = DefaultFeignConfig.class`。
- **全局**：在 `@EnableFeignClients` 上写 `defaultConfiguration = DefaultFeignConfig.class`。

YAML 不改代码即可按环境调级别；Java 配置便于在同一类中统一管理 Logger、拦截器、编解码器等。

#### 7. 连接池（OpenFeign 整合 OKHttp）

默认 Feign 使用 URLConnection，连接不复用。改用 OKHttp 可开启连接池，提升高并发下的性能。

**1. 引入依赖**

```xml
<!-- ok-http -->
<dependency>
    <groupId>io.github.openfeign</groupId>
    <artifactId>feign-okhttp</artifactId>
</dependency>
```

**2. 开启连接池**

```yaml
feign:
  okhttp:
    enabled: true   # 开启 OKHttp 连接池支持
```

#### 8. 请求拦截器（RequestInterceptor）

**RequestInterceptor** 是 Feign 的请求拦截器接口：在每次 Feign 请求发出前会调用，可对 `RequestTemplate` 做修改（如统一添加请求头），常用于传递 Token、TraceId、租户 ID 或当前用户信息。在配置类中声明 `RequestInterceptor` 类型的 Bean，通过 `@EnableFeignClients(defaultConfiguration = FeignConfig.class)` 或 `@FeignClient(configuration = FeignConfig.class)` 注入后，对该 Feign 客户端的每次请求都会生效。

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String token = UserContext.getToken();  // 从上下文或 ThreadLocal 获取
            if (token != null) {
                template.header("Authorization", "Bearer " + token);
            }
        };
    }
}
```

#### 9. 最佳实践

| 实践项             | 说明                                                                                                                                                                                        |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **接口独立成模块** | 将 Feign 接口、DTO 放在独立 API 模块，供多个消费者依赖；消费者通过 `@EnableFeignClients(basePackages = "xxx.api.client")` 或 `@EnableFeignClients(clients = {UserClient.class})` 指定扫描。 |
| **超时与重试**     | 在 `feign.client.config` 中合理设置 `connectTimeout`、`readTimeout`；重试需谨慎，写操作避免重复提交，可配合熔断使用。                                                                       |
| **熔断降级**       | 生产环境建议为 `@FeignClient` 配置 `fallback` 或 `fallbackFactory`，与 Sentinel/Resilience4j 集成，避免下游故障拖垮调用方。                                                                 |
| **连接池**         | 生产环境建议开启 OKHttp 连接池，减少连接建立开销，提高并发能力。                                                                                                                            |
| **日志级别**       | 生产默认 NONE 或 BASIC，排查问题时再开 FULL；用 YAML 按环境区分，避免日志量过大。                                                                                                           |
| **统一请求头**     | 通过 `RequestInterceptor` 在配置类中统一添加 Token、TraceId、租户 ID 等，避免每个接口手写。                                                                                                 |
| **接口设计**       | 避免单次请求体/响应体过大，列表类接口考虑分页；接口方法语义清晰，与提供方 REST 约定一致。                                                                                                   |

## 五、负载均衡

用**服务名**调用（RestTemplate 或 OpenFeign）时，「从多个实例里选一个」由 **Spring Cloud LoadBalancer** 完成。

### 5.1 客户端负载均衡

微服务中通常使用**客户端负载均衡**：调用方从注册中心拿到实例列表后，在本地按策略选出一个实例再发请求（而非经过独立的负载均衡器代理）。

```
                    ┌─────────────────┐
                ┌──▶│ user-service-1  │
                │   └─────────────────┘
┌──────────┐    │   ┌─────────────────┐
│  调用方   │────┼──▶│ user-service-2  │
└──────────┘    │   └─────────────────┘
  (含负载       │   ┌─────────────────┐
   均衡器)      └──▶│ user-service-3  │
                    └─────────────────┘
```

### 5.2 Spring Cloud LoadBalancer

Spring Cloud 2020+ 使用 **LoadBalancer** 替代已停用的 Ribbon，负责「按服务名解析 + 多实例时选一个」。

| 注解              | 作用                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **@LoadBalanced** | 标记 RestTemplate、WebClient 等 Bean，使其在请求「服务名」时通过 LoadBalancer 解析并做负载均衡；不加则只能用具体 IP:端口，无法走注册中心。 |

| 使用方式         | 启用方式                                                                                                                                               |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **RestTemplate** | 在 RestTemplate 的 `@Bean` 上标注 **@LoadBalanced**，该 Bean 发出的请求才会走 LoadBalancer。                                                           |
| **OpenFeign**    | 无需额外注解；classpath 中存在 `spring-cloud-starter-loadbalancer` 时，Feign 会自动用 LoadBalancer 解析 `@FeignClient(name = "服务名")` 并做负载均衡。 |

使用 RestTemplate 做服务调用时，若未由其他 starter 传递引入，需显式添加 `spring-cloud-starter-loadbalancer`；使用 OpenFeign 时通常已传递引入，无需重复添加。

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-loadbalancer</artifactId>
</dependency>
```

### 5.3 常见负载均衡策略与自定义

不自定义时，默认使用 **RoundRobin**（轮询）。Spring Cloud LoadBalancer 内置策略：

| 策略                       | 说明         |
| -------------------------- | ------------ |
| **RoundRobinLoadBalancer** | 轮询（默认） |
| **RandomLoadBalancer**     | 随机         |

> 💡 早期 Ribbon 提供 WeightedResponseTime、BestAvailable 等策略，已随 Ribbon 停用。LoadBalancer 内置仅上述两种，如需更复杂策略可自定义。

需要自定义策略时，声明 `ReactorLoadBalancer<ServiceInstance>` 类型的 Bean，通过 `LoadBalancerClientFactory` 获取对应服务名的 `ServiceInstanceListSupplier`，再封装成具体策略。下述 Bean 会对所有使用 LoadBalancer 的服务生效；若仅对某服务生效，可在调用方所在配置类上使用 `@LoadBalancerClient(name = "服务名", configuration = LoadBalancerConfig.class)`。将某服务的负载均衡改为**随机**策略示例：

```java
@Configuration
public class LoadBalancerConfig {
    
    @Bean
    ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment environment,
            LoadBalancerClientFactory clientFactory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            clientFactory.getLazyProvider(name, ServiceInstanceListSupplier.class),
            name
        );
    }
}
```

## 六、服务网关

### 6.1 网关的作用

**服务网关（API Gateway）** 是微服务架构的**统一入口**，所有外部请求先经过网关，由网关负责路由转发、认证鉴权、限流熔断与日志监控，后端微服务无需各自暴露给客户端。

| 功能         | 说明                                             |
| ------------ | ------------------------------------------------ |
| **统一入口** | 客户端只访问网关，由网关将请求转发到对应微服务   |
| **路由转发** | 根据路径、请求头等规则将请求转发到目标服务       |
| **认证授权** | 在网关层统一校验 Token、权限，未通过则直接拒绝   |
| **限流熔断** | 对流量与异常进行保护，避免打垮后端               |
| **日志监控** | 统一记录请求日志、耗时，便于排查与监控           |

**常见网关方案：**

| 方案                     | 特点                                                                   |
| ------------------------ | ---------------------------------------------------------------------- |
| **Spring Cloud Gateway** | 基于 WebFlux 响应式，非阻塞，性能好；Spring 官方维护（推荐）          |
| **Zuul 1.x**             | 基于 Servlet 阻塞模型，性能较低；Netflix 已停止维护                    |
| **Kong / APISIX**        | 基于 Nginx/OpenResty，独立于 Java 生态；适合多语言微服务架构           |

```
    客户端请求
        ↓
    ┌─────────┐
    │ Gateway │
    └────┬────┘
         │ 按路由规则转发
    ┌────┼────┬────────────────┐
    ↓    ↓    ↓                ↓
 user  order  product  …（其他微服务）
```

### 6.2 Spring Cloud Gateway

**Spring Cloud Gateway** 是 Spring 官方基于 WebFlux 的响应式网关，通过**路由（Route）**决定“哪些请求 → 转发到哪个服务”。

| 概念            | 说明                                                                 |
| --------------- | -------------------------------------------------------------------- |
| **路由（Route）** | 一条转发规则，由 id、uri、断言、过滤器组成                           |
| **id**          | 路由唯一标识                                                         |
| **uri**         | 目标地址，如 `lb://user-service` 表示按服务名从注册中心负载均衡转发  |
| **predicates**  | 断言，判断当前请求是否匹配该路由（如 Path、Method、Header）          |
| **filters**     | 过滤器，对匹配到的请求或响应做修改（如加请求头、去路径前缀）          |

**请求处理流程：**

```
    请求到达 Gateway
        ↓
    RoutePredicateHandlerMapping（逐条匹配路由断言，命中第一条即停止）
        ↓
    FilteringWebHandler（收集该路由的 GatewayFilter + 全局 GlobalFilter，按 Order 排序组成链）
        ↓ pre 阶段（过滤器链正序执行）
    NettyRoutingFilter 转发到目标服务
        ↓ post 阶段（过滤器链逆序执行）
    响应返回客户端
```

> 💡 多条路由的断言都能匹配同一请求时，**配置顺序靠前的路由优先**。路径越具体的路由应放在前面，通配路由放在最后。

#### 1. 添加依赖

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
```

> 💡 Gateway 基于 **WebFlux**（响应式），与 **Spring MVC** 不兼容。若现有项目是 MVC 单体，应单独建一个网关模块只引入 Gateway，不要在同一模块中同时引入 `spring-boot-starter-web` 与 `spring-cloud-starter-gateway`。

#### 2. 路由配置示例

`Path` 匹配请求路径；`StripPrefix=1` 去掉路径第一段（如 `/api/users/1` 转发为 `/users/1`）；`AddRequestHeader` 为转发请求添加请求头。`lb://服务名` 表示从注册中心拉取该服务实例并做负载均衡。

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service                    # lb 表示从注册中心拉取实例
          predicates:
            - Path=/api/users/**                   # 匹配 /api/users 开头的请求
          filters:
            - StripPrefix=1                        # 去掉第一段路径 /api

        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Request-Source, gateway   # 下游可识别请求来自网关
```

#### 3. 断言工厂（Predicate）

断言决定请求是否命中当前路由，配置在路由的 `predicates` 下。常用断言：

| 断言                       | 说明              | 配置示例                                             |
| -------------------------- | ----------------- | ---------------------------------------------------- |
| **Path**                   | 路径匹配          | `Path=/api/**`                                       |
| **Method**                 | HTTP 方法         | `Method=GET,POST`                                    |
| **Header**                 | 请求头匹配        | `Header=X-Token, \d+`                                |
| **Query**                  | 查询参数          | `Query=name, jack` 或 `Query=name`                   |
| **Cookie**                 | 必须包含某 Cookie | `Cookie=chocolate, ch.p`                             |
| **Host**                   | 请求域名          | `Host=**.example.org`                                |
| **RemoteAddr**             | 请求者 IP 范围    | `RemoteAddr=192.168.1.1/24`                          |
| **After** / **Before** / **Between** | 时间范围  | `After=2024-01-01T00:00:00+08:00[Asia/Shanghai]`     |
| **Weight**                 | 权重路由（灰度）  | `Weight=group1, 8`                                   |

> 💡 同一条路由可配多个断言，**所有断言全部匹配**请求才命中该路由（AND 关系）。

#### 4. 过滤器工厂（Filter）

过滤器对命中该路由的请求或响应做修改，配置在路由的 `filters` 下。常用过滤器：

| 过滤器                    | 说明                                                             |
| ------------------------- | ---------------------------------------------------------------- |
| **AddRequestHeader**      | 添加请求头                                                       |
| **RemoveRequestHeader**   | 移除请求头                                                       |
| **AddResponseHeader**     | 添加响应头                                                       |
| **StripPrefix**           | 去掉路径前 N 段（如 `/api/users/1` → `/users/1`）               |
| **PrefixPath**            | 为路径添加前缀（如 `/users/1` → `/api/users/1`）                |
| **RewritePath**           | 按正则重写路径                                                   |
| **Retry**                 | 转发失败时重试，可配重试次数与状态码                             |
| **CircuitBreaker**        | 熔断，需引入 Resilience4j，触发后走 fallback URI                 |
| **RequestRateLimiter**    | 基于 Redis 的令牌桶限流                                          |
| **DedupeResponseHeader**  | 去除重复响应头（如 CORS 头被网关和微服务各加一次时去重）         |

**StripPrefix 路径转换：**

| StripPrefix 值 | 原始路径           | 转发路径    |
| -------------- | ------------------ | ----------- |
| 1              | `/api/users/1`     | `/users/1`  |
| 2              | `/api/v1/users/1`  | `/users/1`  |

**default-filters：** 路由级 `filters` 只对该路由生效。若多条路由需要相同过滤器（如统一添加请求头），可提取到 `default-filters` 中，对所有路由全局生效：

```yaml
spring:
  cloud:
    gateway:
      default-filters:
        - AddRequestHeader=X-Request-Source, gateway
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - StripPrefix=1    # 此过滤器只对该路由生效
```

> 💡 `default-filters` 是 YAML 配置层面的"全局过滤器"，与 `GlobalFilter` 接口不同：前者将内置过滤器工厂应用到所有路由，后者用 Java 代码实现自定义过滤逻辑。

#### 5. GatewayFilter 与 GlobalFilter

| 类型              | 作用范围 | 生效方式                                      |
| ----------------- | -------- | --------------------------------------------- |
| **GatewayFilter** | 单条路由 | 在路由的 `filters` 中配置后，仅该路由生效     |
| **GlobalFilter**  | 全局     | 实现 `GlobalFilter` 并注册为 Bean 后自动生效  |

方法签名均为：`Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain)`。pre 逻辑写在 `chain.filter(exchange)` 之前；post 逻辑写在之后，用 `.then(Mono.fromRunnable(...))`。

#### 6. 自定义全局过滤器

实现 `GlobalFilter` 并注册为 Spring Bean 即可生效；用 **@Order** 控制在过滤器链中的顺序（数值越小越先执行，鉴权建议设较小值如 -1，保证先于业务转发执行）。pre 逻辑在 `chain.filter(exchange)` 前执行（如校验 Token）；post 逻辑在 `.then(Mono.fromRunnable(...))` 中执行（如记录日志、耗时）。

```java
@Component
@Order(-1)   // 数值越小越先执行，鉴权一般放在前面
public class AuthGlobalFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();

        // pre：校验 Token，不通过直接返回 401
        String token = request.getHeaders().getFirst("Authorization");
        if (token == null || !validateToken(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // 放行，继续执行后续过滤器并转发；post 在 then 中执行
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            // post：可记录访问日志、统计耗时等
        }));
    }

    private boolean validateToken(String token) {
        // 实际项目中校验 JWT 等
        return true;
    }
}
```

| 要点           | 说明                                               |
| -------------- | -------------------------------------------------- |
| **@Order(-1)** | 保证该过滤器尽早执行，常用于鉴权、日志等全局逻辑   |
| **pre**        | 在 `chain.filter(exchange)` 之前写，可拦截并直接返回 |
| **post**       | 在 `.then(Mono.fromRunnable(...))` 中写，转发完成后执行 |

#### 7. 动态路由（结合 Nacos）

YAML 静态路由修改后需重启网关才能生效。**动态路由**把路由配置存到 Nacos，网关启动时拉取并注册监听器，Nacos 配置变更时自动刷新路由表，无需重启。

**实现要点：** Nacos 中维护一份 JSON 路由配置（Data ID 如 `gateway-routes.json`）→ 网关通过 `NacosConfigManager.getConfigAndSignListener` 拉取并监听 → 回调中解析 JSON 为 `RouteDefinition`，写入 `RouteDefinitionWriter` 并发布 `RefreshRoutesEvent`。

**① Nacos 中的路由配置（JSON 格式）**

在 Nacos 控制台创建配置，Data ID 为 `gateway-routes.json`，格式选 JSON：

```json
[
  {
    "id": "user-service",
    "uri": "lb://user-service",
    "predicates": [
      { "name": "Path", "args": { "pattern": "/api/users/**" } }
    ],
    "filters": [
      { "name": "StripPrefix", "args": { "_genkey_0": "1" } }
    ]
  },
  {
    "id": "order-service",
    "uri": "lb://order-service",
    "predicates": [
      { "name": "Path", "args": { "pattern": "/api/orders/**" } }
    ],
    "filters": [
      { "name": "StripPrefix", "args": { "_genkey_0": "1" } }
    ]
  }
]
```

**Path 断言多路径：** `Path` 的 `args.pattern` 只支持**单个**路径模式。若写成 `"pattern": "/items/**,/search/**"`，逗号会被当作字面量，请求 `/search/xxx` 不会匹配。同一服务要匹配多个路径有两种写法：

| 写法 | 说明 |
|------|------|
| **一条路由、多个 path** | 在 `args` 中用 `_genkey_0`、`_genkey_1` 等分别写每个 pattern，一条路由即可匹配多个路径。 |
| **多条路由** | 每个 path 单独一条路由，`uri` 指向同一服务，路由条数增多但结构简单。 |

多 path 示例（一条路由）：

```json
{
  "id": "item",
  "uri": "lb://item-service",
  "predicates": [
    { "name": "Path", "args": { "_genkey_0": "/items/**", "_genkey_1": "/search/**" } }
  ],
  "filters": []
}
```

**② 网关侧监听并刷新路由**

使用 `NacosConfigManager.getConfigAndSignListener` 拉取配置并注册监听器，Nacos 变更时回调 `receiveConfigInfo`；在回调中解析 JSON 为 `RouteDefinition` 列表，先删除旧路由、再写入新路由、最后发布 `RefreshRoutesEvent`，网关即生效。

```java
@Component
@RequiredArgsConstructor
public class DynamicRouteLoader {

    private final RouteDefinitionWriter routeDefinitionWriter;
    private final ApplicationEventPublisher publisher;
    private final NacosConfigManager nacosConfigManager;

    private final ObjectMapper objectMapper = new ObjectMapper();
    private static final String DATA_ID = "gateway-routes.json";
    private static final String GROUP = "DEFAULT_GROUP";
    private final Set<String> currentRouteIds = ConcurrentHashMap.newKeySet();

    @PostConstruct
    public void init() throws NacosException {
        String configInfo = nacosConfigManager.getConfigService()
                .getConfigAndSignListener(DATA_ID, GROUP, 5000, new Listener() {
                    @Override
                    public Executor getExecutor() { return null; }

                    @Override
                    public void receiveConfigInfo(String configInfo) {
                        updateRoutes(configInfo);
                    }
                });
        updateRoutes(configInfo);
    }

    private synchronized void updateRoutes(String json) {
        if (json == null || json.isBlank()) return;
        try {
            List<RouteDefinition> routes = objectMapper.readValue(
                    json, new TypeReference<List<RouteDefinition>>() {});
            // 1. 顺序删除旧路由并等待完成
            Mono.when(currentRouteIds.stream()
                            .map(id -> routeDefinitionWriter.delete(Mono.just(id)))
                            .toArray(Mono[]::new))
                    .block();
            currentRouteIds.clear();
            // 2. 顺序写入新路由并等待完成
            for (RouteDefinition route : routes) {
                routeDefinitionWriter.save(Mono.just(route)).block();
                currentRouteIds.add(route.getId());
            }
            // 3. 再发布刷新事件
            publisher.publishEvent(new RefreshRoutesEvent(this));
        } catch (Exception e) {
            throw new RuntimeException("动态路由更新失败", e);
        }
    }
}
```

| 实现要点 | 说明 |
| -------- | ---- |
| **顺序** | 先 `delete` 并 `block()` 等待完成，再 `save` 新路由并 `block()`，最后发布 `RefreshRoutesEvent`，避免并发错乱 |
| **并发** | `currentRouteIds` 用 `ConcurrentHashMap.newKeySet()`，`updateRoutes` 用 `synchronized`，防止 Nacos 回调与首次加载并发 |
| **Mono.when** | 传参用 `.toArray(Mono[]::new)`，避免与 varargs 兼容性问题 |

> 💡 `getConfigAndSignListener` 返回当前配置并注册监听器，Nacos 推送变更时自动回调。反序列化用 Spring Boot 自带的 Jackson（`ObjectMapper`），`RouteDefinition` 与 JSON 的 `id`、`uri`、`predicates`、`filters` 对应。

> **注意**：项目中同时存在 Hutool 和 Jackson 时，以下类有同名冲突，IDEA 自动导入容易选错：
>
> | 类名             | ✅ 正确（Jackson）                              | ❌ 易错（Hutool）                  |
> | ---------------- | ----------------------------------------------- | ---------------------------------- |
> | `ObjectMapper`   | `com.fasterxml.jackson.databind.ObjectMapper`   | `cn.hutool.json.ObjectMapper`      |
> | `TypeReference`  | `com.fasterxml.jackson.core.type.TypeReference` | `cn.hutool.core.lang.TypeReference`|
>
> 导入时认准包名前缀 `com.fasterxml.jackson`。

| 对比项       | YAML 静态路由              | Nacos 动态路由                       |
| ------------ | -------------------------- | ------------------------------------ |
| **配置位置** | 写在网关的 application.yml | 存储在 Nacos 配置中心                |
| **修改生效** | 需重启网关                 | Nacos 修改后自动推送，网关热加载     |
| **适用场景** | 路由少且稳定               | 路由频繁变更、多环境管理、灰度发布等 |

#### 8. 跨域配置（CORS）

前后端分离时，前端直连网关会触发浏览器同源策略，需在网关统一返回 CORS 响应头，下游微服务不必再配。

```yaml
spring:
  cloud:
    gateway:
      globalcors:
        cors-configurations:
          '[/**]':
            allowedOriginPatterns: "*"   # 生产环境建议写具体前端域名，如 https://app.example.com
            allowedMethods: "*"
            allowedHeaders: "*"
            allowCredentials: true
            maxAge: 3600
```

| 配置项                 | 说明 |
| ---------------------- | ---- |
| **allowedOriginPatterns** | 允许的来源，`*` 表示任意（与 `allowCredentials: true` 同用时需网关 2.4+ 的 pattern 支持） |
| **allowCredentials**   | 是否允许携带 Cookie |
| **maxAge**             | 预检请求结果缓存时间（秒） |

> **注意**：网关已配置 CORS 后，微服务内不要再加 `@CrossOrigin` 或 `CorsFilter`，否则会返回重复的 CORS 头，浏览器报错。若无法避免（如第三方服务已加 CORS 头），可在 `default-filters` 中添加 `DedupeResponseHeader` 去重：

```yaml
spring:
  cloud:
    gateway:
      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Origin Access-Control-Allow-Credentials, RETAIN_UNIQUE
```

## 七、配置中心

### 7.1 为什么需要配置中心

微服务各自维护 application.yml 时，公共配置（如数据库地址、日志级别）分散在多个服务中，修改一处需逐个重新部署。**配置中心** 将配置集中存储到外部（如 Nacos），统一管理并支持运行时刷新。

| 能力         | 说明                                                               |
| ------------ | ------------------------------------------------------------------ |
| **集中管理** | 所有服务的配置存放在同一位置，修改一份即可影响多个服务             |
| **动态刷新** | 修改配置后微服务无需重启即可生效（热更新）                         |
| **环境隔离** | 通过命名空间或 profile 隔离 dev/test/prod 配置，避免环境间相互影响 |
| **版本管理** | Nacos 自动保留配置历史版本，可对比差异、一键回滚                   |

### 7.2 Nacos 配置中心接入

Nacos 同时提供「服务注册发现」和「配置中心」两项能力。接入配置中心时，客户端需在启动早期连接 Nacos 拉取配置，再初始化应用上下文。

#### 1. 添加依赖

```xml
<!-- Nacos 配置中心 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
</dependency>
<!-- 读取 bootstrap 文件（Spring Boot 2.4+ 默认不加载 bootstrap，需显式引入） -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

> 💡 **Spring Boot 3.x 替代方式**：可不引入 `spring-cloud-starter-bootstrap`，改在 application.yml 中用 `spring.config.import=nacos:<data-id>?group=<group>&refreshEnabled=true` 导入 Nacos 配置，无需 bootstrap.yml。两种方式二选一，下文以 bootstrap 方式为主。

#### 2. bootstrap.yml 配置

Nacos 服务端地址、命名空间、配置文件名格式（file-extension）及可选共享配置写在 bootstrap.yml：

```yaml
spring:
  application:
    name: order-service
  profiles:
    active: dev
  cloud:
    nacos:
      config:
        server-addr: localhost:8848
        file-extension: yaml        # 配置文件格式
        namespace: dev              # 命名空间
        group: DEFAULT_GROUP
        shared-configs:             # 可选：共享配置
          - data-id: common.yaml
            group: DEFAULT_GROUP
            refresh: true
```

#### 3. 共享配置

共享配置供多个微服务共用（如公共数据库地址、日志级别、中间件连接信息等），在 Nacos 中存一份，各服务在 bootstrap 的 `shared-configs` 中引用。

**Nacos 控制台创建：** 配置管理 → 配置列表 → 选择命名空间 → 新建配置，填写 Data ID（如 `common.yaml`）、Group（如 `DEFAULT_GROUP`）及配置内容。

**应用侧引用：** 在 `spring.cloud.nacos.config.shared-configs` 中按 `data-id`、`group` 引用（已在上方 bootstrap.yml 示例中体现），需动态刷新时设 `refresh: true`。

> 💡 `shared-configs` 列表中可写多个共享配置，同一 key 出现在多份共享配置中时，**靠后的覆盖靠前的**。除 `shared-configs` 外还有 `extension-configs`（扩展配置），用法相同，加载优先级高于 `shared-configs`。

#### 4. 本服务专属配置

除共享配置外，Nacos 客户端会**自动拉取仅当前服务使用的配置**，Data ID 按以下公式生成，无需在 bootstrap 中手写：

```
${spring.application.name}-${spring.profiles.active}.${file-extension}
示例：<服务名>-<profile>.<扩展名>
```

在 Nacos 控制台按此命名创建配置即可存放本服务专属项（如开关、超时时间等）。若 Nacos 中无此配置，不影响启动，仅加载共享配置。

Nacos 客户端实际会依次尝试拉取以下 Data ID（存在则加载，不存在则跳过）：

| Data ID                    | 说明                               |
| -------------------------- | ---------------------------------- |
| `<服务名>-<profile>.yaml` | 本服务 + 当前环境的专属配置        |
| `<服务名>.yaml`            | 本服务的通用配置（不区分 profile） |
| 共享配置 / 扩展配置        | 多服务共用的公共配置               |

#### 5. 配置加载与优先级

**加载流程：**

```
启动 → 加载 bootstrap.yml → 连接 Nacos 拉取远程配置 → 初始化 ApplicationContext（加载 application.yml）
```

Nacos 地址、命名空间等必须写在 **bootstrap.yml** 中，否则启动早期无法连接 Nacos。

**配置优先级（高 → 低）：** 同一 key 在多个配置源中出现时，高优先级覆盖低优先级。

| 优先级 | 配置源                                            |
| ------ | ------------------------------------------------- |
| 最高   | Nacos `<服务名>-<profile>.yaml`                   |
| ↓      | Nacos `<服务名>.yaml`                             |
| ↓      | Nacos 扩展配置（extension-configs）               |
| ↓      | Nacos 共享配置（shared-configs）                  |
| ↓      | 本地 application-<profile>.yml / application.yml  |

> 💡 Nacos 远程配置优先于本地 application.yml。若需要本地配置不被覆盖，可将其放在 Nacos 中统一管理，或设置 `spring.cloud.config.override-none=true` 使远程配置不覆盖本地已有值。

#### 6. 配置热更新

修改 Nacos 中的配置后，微服务**无需重启**即可使新配置生效。

**前提**：① Nacos 中存在该服务的配置文件（Data ID 按公式命名）；② 代码中以可刷新的方式读取配置值。

| 方式                                    | 是否支持热更新 | 原理                                                                      |
| --------------------------------------- | -------------- | ------------------------------------------------------------------------- |
| **@ConfigurationProperties**            | ✅ 支持         | Nacos 发布 `RefreshEvent` 后，Spring Cloud 自动重新绑定属性，无需额外注解 |
| **@RefreshScope** + **@Value**          | ✅ 支持         | 刷新时 Bean 被销毁重建，`@Value` 重新注入                                 |
| 单独 **@Value**（不加 `@RefreshScope`） | ❌ 不支持       | 值在 Bean 创建时注入一次，之后不变                                        |

```java
// ✅ @ConfigurationProperties：Nacos 改配置后自动热更新
@Data
@Component
@ConfigurationProperties(prefix = "app.cart")
public class CartConfig {
    private int maxItems;
}

// ✅ @RefreshScope + @Value：Nacos 改配置后也可热更新
@RestController
@RefreshScope
public class CartController {
    @Value("${app.cart.maxItems}")
    private int maxItems;
}

// ❌ 单独 @Value 不加 @RefreshScope：不会热更新
@RestController
public class CartController {
    @Value("${app.cart.maxItems}")
    private int maxItems;  // 值注入后不再变化
}
```

**两种方式的选择：**

| 注解                           | 适用场景                                                       |
| ------------------------------ | -------------------------------------------------------------- |
| **@ConfigurationProperties**   | 配置项多且有统一前缀时推荐，类型安全、支持嵌套、IDE 自动提示   |
| **@RefreshScope** + **@Value** | 只需注入少量配置项时使用，适合在 Controller/Service 中零散引用 |

### 7.3 Spring Cloud Config（了解）

Spring 官方提供的配置中心，基于 Git 或文件存储；与 Nacos Config 二选一。

| 对比项       | Nacos Config                       | Spring Cloud Config                                |
| ------------ | ---------------------------------- | -------------------------------------------------- |
| **存储方式** | 内置数据库（MySQL / 嵌入式）       | Git 仓库 / 文件系统                                |
| **动态刷新** | 原生支持，秒级推送生效             | 需配合 Spring Cloud Bus 或手动触发 `/actuator/refresh` |
| **可视化**   | Nacos 控制台直接编辑               | 无内置 UI，需直接编辑 Git                          |
| **适用场景** | Spring Cloud Alibaba 体系（主流）  | 纯 Spring Cloud 体系                               |

```yaml
# Config Server 端配置示例
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/xxx/config-repo
          search-paths: config
          default-label: main
```
