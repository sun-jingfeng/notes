## 一、服务熔断与降级

### 1.1 雪崩效应

**雪崩效应** 是指在微服务中，如果一个服务不可用，可能导致调用链上的所有服务都不可用。

```
服务A ──▶ 服务B ──▶ 服务C（故障）
   │         │
   ▼         ▼
 超时阻塞   超时阻塞
   │         │
   ▼         ▼
 资源耗尽   资源耗尽
   │
   ▼
 雪崩
```

**高可用常见手段：**

| 手段           | 说明                   |
| -------------- | ---------------------- |
| **多实例部署** | 避免单点故障           |
| **负载均衡**   | 分散压力               |
| **熔断降级**   | 快速失败，防止级联故障 |
| **限流保护**   | 控制流量，保护服务     |
| **健康检查**   | 及时发现异常实例       |

### 1.2 Sentinel（推荐）

**Sentinel** 是阿里巴巴开源的流量控制组件，通过「资源」来标识要保护的接口或方法，并可为资源配置限流、熔断规则。触发限流或熔断时抛出 **BlockException**，需要单独的处理方法；业务自身抛出的异常则可用 **fallback** 做降级返回。

| 概念 / 注解                                    | 说明                                                                                          |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------- |
| **资源**                                       | 用 `@SentinelResource(value = "资源名")` 标记要保护的方法，资源名在控制台配置规则时使用。     |
| **blockHandler**                               | 处理限流、熔断触发的 **BlockException**，方法签名需包含 `BlockException` 参数，返回降级结果。 |
| **fallback**                                   | 处理业务异常（如远程调用抛出的异常），方法签名为原方法 + `Throwable`，返回降级结果。          |
| **@FeignClient 的 fallback / fallbackFactory** | Feign 与 Sentinel 集成时，指定限流或调用失败时的降级实现；`fallbackFactory` 可拿到异常原因。  |

添加依赖 → 配置控制台地址 → 在业务方法上使用 `@SentinelResource` 并指定 blockHandler/fallback →（可选）为 Feign 接口配置 fallback。

#### 1. 添加依赖

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

#### 2. 配置

**基础：与控制台通信**

```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8858  # 应用连接 Dashboard 的地址（应用 → Dashboard）
        port: 8719                 # 应用暴露给 Dashboard 的端口（Dashboard → 应用，用于规则推送）
```

**常用扩展配置**

| 配置项 | 说明 | 典型值 |
|--------|------|--------|
| **eager** | 启动即连 Dashboard，控制台立刻见机器、规则即生效；默认 false 为首次访问才连 | `true` |
| **filter.enabled** | 是否启用 CommonFilter，对 Web 请求自动生成资源 | `true` |
| **filter.url-patterns** | 参与统计与限流的 URL；默认 `/*` | `/*`、`/api/*` |
| **web-context-unify** | 统一 Web 上下文（入口与子资源同链）；默认 true | `true` |
| **http-method-specify** | 资源名带 HTTP 方法前缀（如 `GET:/api/order`），便于同一 URL 按 GET/POST 分别限流；默认 false | `true` / `false` |
| **block-page** | 限流/熔断时跳转的 URL，需自行实现该路径返回友好页 | `/custom-block` |
| **metric.file-single-size** | 单块监控日志大小（字节），控制磁盘占用 | `52428800` |
| **metric.file-total-count** | 监控日志文件保留个数 | `6` |
| **log.dir** | 日志与 metric 目录 | `logs/csp` |

#### 3. 资源定义与降级示例

`value = "getOrder"` 定义资源名；触发限流或熔断时走 `getOrderBlockHandler`（必须带 `BlockException` 参数）；业务抛异常时走 `getOrderFallback`（带 `Throwable` 参数）。

```java
@Service
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        blockHandler = "getOrderBlockHandler",
        fallback = "getOrderFallback"
    )
    public Order getOrder(Long orderId) {
        return orderRepository.findById(orderId);
    }
    
    public Order getOrderBlockHandler(Long orderId, BlockException ex) {
        return new Order().setMessage("服务繁忙，请稍后重试");
    }
    
    public Order getOrderFallback(Long orderId, Throwable ex) {
        return new Order().setMessage("服务暂时不可用");
    }
}
```

> **注意**：`blockHandler` 和 `fallback` 方法默认须与原方法在**同一类**中，返回值类型一致，参数为原方法参数 + 末尾 `BlockException`（blockHandler）或 `Throwable`（fallback）。若要放在其他类中，需通过 `blockHandlerClass` / `fallbackClass` 指定，且方法须为 **static**。

#### 4. Feign 集成 Sentinel

Feign 调用远程服务时，若下游超时或异常，希望返回默认结果而非把异常抛给上游，可开启 Feign 的 Sentinel 支持并为 `@FeignClient` 指定 **fallback** 或 **fallbackFactory**。限流或调用失败时会走降级实现。使用 `FallbackFactory` 可在降级方法中拿到异常原因。

```yaml
feign:
  sentinel:
    enabled: true  # 开启 Feign 的 Sentinel 支持
```

```java
// Feign 接口
@FeignClient(name = "user-service", fallbackFactory = UserClientFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
}

// 降级工厂
@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUserById(Long id) {
                log.error("调用用户服务失败: {}", cause.getMessage());
                return new User().setName("未知用户");
            }
        };
    }
}
```

### 1.3 Sentinel 核心功能

| 功能             | 说明                       |
| ---------------- | -------------------------- |
| **流量控制**     | QPS / 线程数限流           |
| **熔断降级**     | 慢调用 / 异常比例 / 异常数 |
| **热点参数限流** | 针对特定参数限流           |
| **系统保护**     | CPU / 负载 / 入口 QPS      |
| **授权规则**     | 黑白名单控制               |

#### 流量控制规则

流控规则的**阈值类型**：按 **QPS**（每秒请求数）或**线程数**（并发线程数）限流。**流控模式**决定统计维度：

| 模式     | 说明                                                                         |
| -------- | ---------------------------------------------------------------------------- |
| **直接** | 统计当前资源自身的 QPS/线程数，达到阈值即限流                                |
| **关联** | 统计关联资源的指标，关联资源达到阈值时限流当前资源（如写接口触发限流读接口） |
| **链路** | 统计从指定入口进入当前资源的请求，仅对该入口限流                             |

**流控效果**决定触发限流后的处理方式：

| 效果         | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| **快速失败** | 直接抛出 `FlowException`（默认）                         |
| **Warm Up**  | 预热模式，阈值从 `threshold / coldFactor` 逐步升到设定值 |
| **排队等待** | 匀速排队，控制请求间隔，适合突发流量削峰填谷             |

#### 熔断策略

当资源的请求满足熔断条件时，Sentinel 自动熔断该资源，在熔断时长内所有请求直接走降级逻辑：

| 策略           | 触发条件                            |
| -------------- | ----------------------------------- |
| **慢调用比例** | 慢调用（RT 超过设定值）比例超过阈值 |
| **异常比例**   | 异常请求占比超过阈值                |
| **异常数**     | 异常请求数超过阈值                  |

> 💡 以上规则均可在 **Sentinel 控制台**（Dashboard）中通过界面配置，也可通过代码或 Nacos 推送。

### 1.4 Resilience4j（了解）

Spring Cloud 官方推荐的熔断/重试/限流组件（可替代 Hystrix），与 Sentinel 二选一。常用注解：

| 注解                | 作用                                                    |
| ------------------- | ------------------------------------------------------- |
| **@CircuitBreaker** | 熔断：失败率等达到阈值后短路，直接走 `fallbackMethod`。 |
| **@Retry**          | 失败重试，可配次数与间隔。                              |
| **@RateLimiter**    | 限流，控制调用速率。                                    |

```java
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
@Retry(name = "userService")
@RateLimiter(name = "userService")
public User getUser(Long userId) {
    return userClient.getUserById(userId);
}

public User getUserFallback(Long userId, Exception ex) {
    return new User().setName("默认用户");
}
```

## 二、分布式事务

### 2.1 分布式事务问题

微服务中每个服务有独立数据库，跨服务的事务无法用本地事务保证。

```
订单服务（创建订单） ──▶ 库存服务（扣减库存） ──▶ 账户服务（扣款）
    │                        │                       │
   DB1                      DB2                     DB3
```

**CAP 理论：** 分布式系统最多同时满足其中两个：

| 特性                                | 说明               |
| ----------------------------------- | ------------------ |
| **C**onsistency（一致性）           | 所有节点数据一致   |
| **A**vailability（可用性）          | 服务始终可用       |
| **P**artition tolerance（分区容错） | 网络分区时仍能运行 |

> 💡 通常选择 AP（最终一致性）或 CP（强一致性）。

### 2.2 Seata 分布式事务

**Seata** 是阿里巴巴开源的分布式事务解决方案。

#### 1. 事务模式

| 模式     | 说明               | 适用场景           |
| -------- | ------------------ | ------------------ |
| **AT**   | 自动补偿，无侵入   | 大多数场景（推荐） |
| **TCC**  | Try-Confirm-Cancel | 高性能场景         |
| **SAGA** | 长事务             | 业务流程长         |
| **XA**   | 强一致性           | 传统数据库         |

**AT 模式** 下，在业务方法上加 **@GlobalTransactional** 即可声明分布式事务边界：方法内对本库的写操作与对其它服务的远程调用由 Seata 协调，任一步失败则全局回滚，无需手写补偿逻辑。

| 注解                     | 作用                                                                                        |
| ------------------------ | ------------------------------------------------------------------------------------------- |
| **@GlobalTransactional** | 声明分布式事务边界；加在业务方法上，方法内跨服务的调用由 Seata 协调，任一步失败则全局回滚。 |

#### 2. AT 模式使用示例

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

```yaml
seata:
  tx-service-group: default_tx_group     # 事务组名称
  service:
    vgroup-mapping:
      default_tx_group: default          # 事务组映射到 TC 集群名
  registry:
    type: nacos                          # TC 注册中心类型
    nacos:
      server-addr: localhost:8848
      namespace: ""
      group: DEFAULT_GROUP
      application: seata-server          # TC 在 Nacos 中的服务名
```

> 💡 AT 模式下，每个参与分布式事务的数据库须创建 **undo_log** 表（Seata 通过该表记录数据快照以实现自动回滚），建表 SQL 见 [Seata 官方文档](https://seata.apache.org/docs/user/quickstart)。同时需部署 **Seata Server（TC）** 并注册到 Nacos，各微服务作为 TM/RM 连接 TC 协调事务。

```java
@Service
public class OrderService {
    
    @Autowired
    private StockClient stockClient;
    
    @Autowired
    private AccountClient accountClient;
    
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单
        orderMapper.insert(orderDTO);
        
        // 2. 扣减库存（远程调用）
        stockClient.deduct(orderDTO.getProductId(), orderDTO.getCount());
        
        // 3. 扣减余额（远程调用）
        accountClient.debit(orderDTO.getUserId(), orderDTO.getMoney());
        
        // 任一步失败，Seata 协调全局回滚
    }
}
```

#### 3. TCC 模式

**TCC 模式** 需要业务自己实现两阶段：**Try**（预留资源）、**Confirm**（确认）、**Cancel**（回滚）。

| 注解                                | 作用                                                                           |
| ----------------------------------- | ------------------------------------------------------------------------------ |
| **@LocalTCC**                       | 标记接口为 TCC 模式，需提供 try/confirm/cancel 两阶段方法。                    |
| **@TwoPhaseBusinessAction**         | 标注 try 方法，并指定 `commitMethod`、`rollbackMethod` 对应 confirm/rollback。 |
| **@BusinessActionContextParameter** | 将参数放入上下文，供 confirm/rollback 方法使用。                               |

接口上标 `@LocalTCC`，Try 方法上标 `@TwoPhaseBusinessAction` 并指定 commitMethod/rollbackMethod，需在 confirm/rollback 中使用的参数用 `@BusinessActionContextParameter` 放入上下文。

```java
@LocalTCC
public interface StockTccService {
    
    @TwoPhaseBusinessAction(
        name = "deductStock",
        commitMethod = "commit",
        rollbackMethod = "rollback"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    // 确认方法
    boolean commit(BusinessActionContext context);
    
    // 回滚方法
    boolean rollback(BusinessActionContext context);
}
```

## 三、链路追踪

### 3.1 为什么需要链路追踪

微服务调用链复杂，需要追踪请求在各服务间的流转过程。

| 用途         | 说明               |
| ------------ | ------------------ |
| **问题定位** | 快速定位故障服务   |
| **性能分析** | 分析各服务耗时     |
| **依赖分析** | 了解服务间调用关系 |

### 3.2 Micrometer Tracing + Zipkin（Spring Boot 3.x 推荐）

Spring Cloud 2022+ / Spring Boot 3.x 起，链路追踪迁移到 **Micrometer Tracing**，替代了已停用的 Sleuth。Micrometer Tracing 提供统一的追踪 API，通过 bridge 适配 Brave 或 OpenTelemetry，再由 reporter 上报到 Zipkin 等后端。

#### 1. 添加依赖

```xml
<!-- Micrometer Tracing + Brave bridge -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<!-- Zipkin reporter -->
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>
<!-- Spring Boot Actuator（提供 /actuator 端点） -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

#### 2. 配置

```yaml
management:
  tracing:
    sampling:
      probability: 1.0        # 采样率，生产环境建议 0.1
  zipkin:
    tracing:
      endpoint: http://localhost:9411/api/v2/spans

logging:
  pattern:
    level: "%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]"
```

#### 3. 核心概念

| 字段        | 说明                     |
| ----------- | ------------------------ |
| **traceId** | 整个调用链的唯一标识     |
| **spanId**  | 当前服务在调用链中的标识 |

日志中会自动注入 traceId 和 spanId（通过 MDC），格式如：

```
INFO [order-service,5c7f3b4a2d1e6f8a,3a2b1c4d5e6f7890]
```

> 💡 OpenFeign、RestTemplate、WebClient 等调用会自动传播 traceId，无需额外配置。

## 四、消息驱动

### 4.1 Spring Cloud Stream

**Spring Cloud Stream** 提供统一的消息中间件编程模型：应用只和 **Binder** 抽象打交道，通过 **binding**（绑定）连接消息目的地，底层可切换 RabbitMQ、Kafka、RocketMQ 等而不改业务代码。旧版通过 `@EnableBinding`、`@StreamListener` 定义输入/输出；新版推荐函数式模型：用 **StreamBridge** 发送、用 `Consumer<T>` Bean 消费。

> 💡 旧版通过 `@EnableBinding(Sink.class)` + `@StreamListener(Sink.INPUT)` 消费消息，已在 Spring Cloud 2022+ 中标记为废弃，新项目统一使用函数式写法。

### 4.2 事件驱动架构（函数式）

**事件驱动** 下，服务完成本地操作后向消息中间件**发送事件**，其他服务**订阅并消费**，实现异步解耦。Spring Cloud Stream 函数式写法：**发送端** 使用 `StreamBridge.send(bindingName, payload)` 发往指定 binding；**消费端** 声明 `Consumer<T>` 类型的 Bean，方法名或 `spring.cloud.function.definition` 与配置的 binding 对应即可接收消息。

**发送端：** 订单创建后通过 `streamBridge.send("order-created", order)` 发送到名为 `order-created` 的 binding（需在配置中绑定到具体 Topic/Queue）。

```java
@Service
public class OrderService {
    
    @Autowired
    private StreamBridge streamBridge;
    
    public void createOrder(Order order) {
        orderMapper.insert(order);
        streamBridge.send("order-created", order);
    }
}
```

**消费端：** 定义 `Consumer<Order>` 类型的 Bean，Spring Cloud Stream 会将其与配置的 input binding 关联，收到消息时调用。

```java
@Component
public class StockEventHandler {
    
    @Bean
    public Consumer<Order> orderCreated() {
        return order -> {
            stockService.deduct(order.getProductId(), order.getCount());
        };
    }
}
```

#### Binding 配置

代码中的 binding 名称需在 YAML 中与实际 Topic/Queue 绑定，否则消息无法路由。以 RabbitMQ 为例：

```yaml
spring:
  cloud:
    stream:
      bindings:
        # 发送端 binding：StreamBridge.send("order-created", payload) 使用此名称
        order-created:
          destination: order-topic     # 实际 Topic/Exchange 名称
          content-type: application/json
        # 消费端 binding：函数名 + "-in-0" 为默认命名规则
        orderCreated-in-0:
          destination: order-topic     # 与发送端的 destination 一致
          group: stock-group           # 消费者组，避免重复消费
      function:
        definition: orderCreated       # 声明要激活的函数 Bean 名称
```

> 💡 **消费端 binding 命名规则**：`<函数名>-in-<index>`，如函数 Bean 名为 `orderCreated`，则 input binding 名为 `orderCreated-in-0`。

## 五、容器化部署

### 5.1 Docker 部署

将应用与运行环境打成镜像，便于在任意宿主机上一致部署。**Dockerfile** 指定基础镜像、工作目录、拷贝 jar、暴露端口和启动命令；构建出的镜像可在本地或仓库运行。

#### Dockerfile 示例

```dockerfile
FROM openjdk:17-jdk-slim

EXPOSE 8080

ENV JAVA_OPTS="-Xms256m -Xmx512m"

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

#### docker-compose.yml

```yaml
services:

      - "8848:8848"

      MODE: standalone

      - nacos
```

### 5.2 Kubernetes 部署

**Kubernetes** 用 **Deployment** 管理 Pod 副本与更新策略，用 **Service** 对 Pod 做服务发现与负载均衡。**readinessProbe**（就绪探针）通过后 Pod 才接收流量；**livenessProbe**（存活探针）失败则重启容器。Spring Boot Actuator 提供 `/actuator/health/readiness` 和 `/actuator/health/liveness` 端点可直接用于探针。

```yaml
# Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: order-service
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
        - name: order-service
          image: order-service:latest
          ports:
            - containerPort: 8080
          resources:
            limits:
              memory: "512Mi"
              cpu: "500m"
          readinessProbe:          # 就绪探针：通过后才接收流量
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 20
            periodSeconds: 5
          livenessProbe:           # 存活探针：失败则重启容器
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
---
# Service
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order-service
  ports:
    - port: 80
      targetPort: 8080
  type: ClusterIP
```

## 六、微服务最佳实践

### 6.1 服务拆分原则

**什么时候拆分：**

| 项目类型           | 建议                                           |
| ------------------ | ---------------------------------------------- |
| **创业型项目**     | 先采用单体架构快速开发、试错，规模扩大后再拆分 |
| **确定的大型项目** | 可直接选微服务架构，避免后期拆分的成本         |

**拆分目标：** 高内聚、低耦合。**高内聚**指每个微服务职责尽量单一、业务关联度高；**低耦合**指功能相对独立，减少对其它服务的依赖。

**拆分方式：**

| 方式         | 说明                         |
| ------------ | ---------------------------- |
| **纵向拆分** | 按业务模块拆（如用户、订单） |
| **横向拆分** | 抽取公共服务，提高复用       |

| 原则           | 说明                   |
| -------------- | ---------------------- |
| **单一职责**   | 每个服务只做一件事     |
| **业务边界**   | 按业务领域划分（DDD）  |
| **数据独立**   | 每个服务拥有独立数据库 |
| **接口稳定**   | 服务间通过稳定接口通信 |
| **渐进式拆分** | 从单体逐步拆分         |

### 6.2 工程结构

微服务可按两种方式组织代码：

| 方式             | 说明                                           |
| ---------------- | ---------------------------------------------- |
| **独立 Project** | 每个微服务一个独立工程，互不依赖，便于独立部署 |
| **Maven 聚合**   | 父工程下多个 module，统一依赖版本、一键构建    |

按业务拆出的模块可作为父工程下的子 module，便于统一管理依赖版本与构建。

### 6.3 服务间通信原则

| 场景         | 推荐方式                     |
| ------------ | ---------------------------- |
| **同步调用** | OpenFeign + LoadBalancer     |
| **异步解耦** | 消息队列（RocketMQ / Kafka） |
| **事件驱动** | Spring Cloud Stream          |

### 6.4 微服务登录与用户传递

**整体思路：** 网关统一做登录校验（如 JWT），校验通过后将用户信息写入转发请求的请求头；下游微服务通过拦截器从请求头解析用户并放入 ThreadLocal，业务层和 Feign 调用时都能拿到当前用户；Feign 通过 **RequestInterceptor** 从 ThreadLocal 取用户并写入请求头，保证跨服务调用时用户信息不丢失。

**① 网关：校验后写入请求头**

在网关的 `GlobalFilter` 中校验 JWT，通过后把用户信息放入转发请求的请求头，再放行：

```java
// 修改转发到微服务的请求，添加请求头
ServerWebExchange newExchange = exchange.mutate()
    .request(builder -> builder.header("user-info", userInfoJson))
    .build();
return chain.filter(newExchange);
```

**② 微服务：拦截器取用户并放入 ThreadLocal**

在公共模块（如 `xxx-common`）中定义 Spring MVC 拦截器，从请求头读取 `user-info`，解析后放入 ThreadLocal，业务层从 ThreadLocal 获取当前用户。

**③ OpenFeign 调用时传递用户**

微服务间通过 OpenFeign 调用时，需把当前用户带到下游。使用 **RequestInterceptor** 统一在请求头中携带用户信息：

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String userInfo = UserContext.getUserInfo();  // 从 ThreadLocal 获取
            if (userInfo != null) {
                template.header("user-info", userInfo);
            }
        };
    }
}
```

| 环节           | 做法                                                 |
| -------------- | ---------------------------------------------------- |
| **网关**       | GlobalFilter 校验 JWT，写 `user-info` 到请求头       |
| **微服务内**   | HandlerInterceptor 读请求头，写入 ThreadLocal        |
| **Feign 调用** | RequestInterceptor 从 ThreadLocal 取用户，写入请求头 |

### 6.5 安全实践

网关或微服务内常用 **Spring Security** 的 **OAuth2 Resource Server** 做 JWT 校验：只放行已携带有效 Token 的请求，其余返回 401。`requestMatchers(...).permitAll()` 放行白名单，`anyRequest().authenticated()` 要求其余请求已认证，`oauth2ResourceServer(oauth2 -> oauth2.jwt())` 使用 JWT 解析并校验。

```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt());
        return http.build();
    }
}
```

### 6.6 可观测性

| 维度     | 工具                                     | 说明                                           |
| -------- | ---------------------------------------- | ---------------------------------------------- |
| **日志** | ELK（Elasticsearch + Logstash + Kibana） | 统一收集、存储、检索各服务日志                 |
| **指标** | Prometheus + Grafana                     | 采集 JVM、QPS、延迟等指标并可视化              |
| **追踪** | Zipkin / SkyWalking                      | 跨服务调用链追踪，定位慢接口与故障节点         |

生产环境建议三者搭配使用。Spring Boot Actuator 暴露 `/actuator/prometheus` 端点供 Prometheus 采集指标；日志建议统一为 JSON 格式输出并通过 Filebeat/Logstash 采集到 Elasticsearch。
