## 一、服务熔断与降级

### 1.1 雪崩效应

**雪崩效应** 是指微服务调用链中某个服务故障（如超时、宕机），导致上游服务线程阻塞、资源耗尽，进而逐级扩散至整条调用链不可用。

```
服务A ──▶ 服务B ──▶ 服务C（故障）
   │         │
   ▼         ▼
 超时阻塞   超时阻塞
   │         │
   ▼         ▼
 资源耗尽   资源耗尽
   │
   ▼
 雪崩
```

常见的高可用手段：

| 手段           | 说明                   |
| -------------- | ---------------------- |
| **多实例部署** | 避免单点故障           |
| **负载均衡**   | 分散压力               |
| **熔断降级**   | 快速失败，防止级联故障 |
| **限流保护**   | 控制流量，保护服务     |
| **健康检查**   | 及时发现异常实例       |

### 1.2 Sentinel 概述

**Sentinel** 是阿里巴巴开源的流量治理组件，以**资源**为核心：对要保护的接口或方法标记资源名，再为资源配置限流、熔断等规则。触发规则时抛出 `BlockException`，需用 `blockHandler` 处理；业务自身的异常则由 `fallback` 处理。

| 概念 / 属性      | 说明                                                                                         |
| ---------------- | -------------------------------------------------------------------------------------------- |
| **资源**         | 用 `@SentinelResource(value = "资源名")` 标记要保护的方法，资源名在控制台配置规则时使用      |
| **blockHandler** | 处理限流或熔断触发的 `BlockException`，方法签名为原方法参数 + `BlockException`，返回降级结果 |
| **fallback**     | 处理业务异常，方法签名为原方法参数 + `Throwable`，返回降级结果                               |

### 1.4 资源定义与降级

#### 1. @SentinelResource

`value` 定义资源名；触发限流或熔断时走 `blockHandler`；业务抛异常时走 `fallback`。

```java
@Service
public class DemoService {
    
    @SentinelResource(
        value = "queryById",
        blockHandler = "queryByIdBlockHandler",
        fallback = "queryByIdFallback"
    )
    public String queryById(Long id) {
        return remoteService.findById(id);
    }
    
    // 限流/熔断降级（BlockException）
    public String queryByIdBlockHandler(Long id, BlockException ex) {
        return "服务繁忙，请稍后重试";
    }
    
    // 业务异常降级（Throwable）
    public String queryByIdFallback(Long id, Throwable ex) {
        return "服务暂时不可用";
    }
}
```

> **注意**：`blockHandler` 和 `fallback` 方法默认须与原方法在**同一类**中，返回值类型一致，参数为原方法参数 + 末尾 `BlockException` / `Throwable`。若要放在其他类中，需通过 `blockHandlerClass` / `fallbackClass` 指定，且方法须为 **static**。

#### 2. Feign 集成 Sentinel

Feign 调用远程服务时，若下游超时或异常，可开启 Feign 的 Sentinel 支持并为 `@FeignClient` 指定 **fallbackFactory**，限流或调用失败时走降级实现，且能拿到异常原因。

```yaml
feign:
  sentinel:
    enabled: true
```

```java
@FeignClient(name = "user-service", fallbackFactory = UserClientFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
}

@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUserById(Long id) {
                log.error("调用用户服务失败: {}", cause.getMessage());
                return new User().setName("未知用户");
            }
        };
    }
}
```

### 1.5 核心规则

| 功能             | 说明                       |
| ---------------- | -------------------------- |
| **流量控制**     | QPS / 线程数限流           |
| **熔断降级**     | 慢调用 / 异常比例 / 异常数 |
| **热点参数限流** | 针对特定参数限流           |
| **系统保护**     | CPU / 负载 / 入口 QPS      |
| **授权规则**     | 黑白名单控制               |

> 💡 Web 请求默认以 URL 路径作为资源名自动注册（如 `/api/xxx/{id}`）；`@SentinelResource(value = "xxx")` 手动标记的以 `value` 值注册。应用需有至少一次请求访问到该接口，资源才会出现在 Dashboard 的簇点链路中。

#### 1. 流量控制

Dashboard → **簇点链路** → 对应资源的「流控」按钮，或 **流控规则** → 新增。

**基本字段：**

| 字段         | 说明                                            |
| ------------ | ----------------------------------------------- |
| **资源名**   | 要限流的资源名                                  |
| **针对来源** | 调用方微服务名，`default` 表示不区分来源        |
| **阈值类型** | **QPS**（每秒请求数）或**线程数**（并发线程数） |
| **单机阈值** | 每台实例的阈值                                  |
| **流控模式** | 直接 / 关联 / 链路                              |
| **流控效果** | 快速失败 / Warm Up / 排队等待                   |

**流控模式：**

| 模式     | 说明                                                     | 额外字段   |
| -------- | -------------------------------------------------------- | ---------- |
| **直接** | 统计当前资源自身，达到阈值即限流                         | —          |
| **关联** | 关联资源达到阈值时限流当前资源（如写接口触发限流读接口） | 关联资源名 |
| **链路** | 仅统计从指定入口进入当前资源的请求                       | 入口资源名 |

> 💡 使用**链路模式**需将 `web-context-unify` 设为 `false`，否则所有 Web 入口统一为同一上下文，链路区分不生效。

**流控效果：**

| 效果         | 说明                                                                        | 额外字段         |
| ------------ | --------------------------------------------------------------------------- | ---------------- |
| **快速失败** | 直接抛出 `FlowException`（默认）                                            | —                |
| **Warm Up**  | 预热模式，阈值从 `单机阈值 / coldFactor`（coldFactor 固定 3）逐步升到设定值 | 预热时长（秒）   |
| **排队等待** | 匀速排队，超过等待时间的请求被拒绝，适合削峰填谷                            | 超时时间（毫秒） |

**配置示例：**

| 场景       | 资源名      | 阈值类型 | 单机阈值 | 流控模式 | 流控效果 | 额外字段        | 效果                                                     |
| ---------- | ----------- | -------- | -------- | -------- | -------- | --------------- | -------------------------------------------------------- |
| QPS 限流   | `queryById` | QPS      | 100      | 直接     | 快速失败 | —               | 单台实例每秒超过 100 次请求时直接拒绝                    |
| 冷启动预热 | `queryById` | QPS      | 200      | 直接     | Warm Up  | 预热时长 10s    | 启动后阈值从 200/3 ≈ 66 逐步升至 200，10 秒内完成预热    |
| 关联限流   | `queryById` | QPS      | 50       | 关联     | 快速失败 | 关联 `saveData` | `saveData` 的 QPS 达到 50 时限流 `queryById`，保护写操作 |

#### 2. 熔断策略

Dashboard → **簇点链路** → 对应资源的「熔断」按钮，或 **熔断规则** → 新增。

**熔断状态流转：**

```
CLOSED（正常）──触发熔断──▶ OPEN（熔断，直接降级）
     ▲                          │
     │                     熔断时长结束
     │                          ▼
     └──探测成功──── HALF_OPEN（放行一次探测）
                        │
                   探测失败 → 回到 OPEN
```

**通用字段：**

| 字段             | 说明                                                     |
| ---------------- | -------------------------------------------------------- |
| **资源名**       | 要熔断的资源名                                           |
| **熔断策略**     | 慢调用比例 / 异常比例 / 异常数                           |
| **熔断时长(s)**  | OPEN 状态持续时间，到期后进入 HALF_OPEN                  |
| **最小请求数**   | 统计时长内请求数达到该值才触发熔断判断，避免少量请求误判 |
| **统计时长(ms)** | 统计窗口，默认 1000ms                                    |

**策略专属字段：**

| 策略           | 专属字段                   | 触发条件                             |
| -------------- | -------------------------- | ------------------------------------ |
| **慢调用比例** | 最大 RT(ms)、比例阈值(0~1) | RT 超过最大 RT 的请求占比 ≥ 比例阈值 |
| **异常比例**   | 比例阈值(0~1)              | 异常请求占比 ≥ 比例阈值              |
| **异常数**     | 异常数                     | 异常请求数 ≥ 异常数阈值              |

> 💡 RT = Response Time（响应时间）。

**配置示例：**

| 场景       | 熔断策略   | 最大 RT | 比例/异常数阈值 | 熔断时长 | 最小请求数 | 统计时长 | 效果                                                  |
| ---------- | ---------- | ------- | --------------- | -------- | ---------- | -------- | ----------------------------------------------------- |
| 慢调用比例 | 慢调用比例 | 200ms   | 0.5             | 10s      | 5          | 1000ms   | 1s 内至少 5 次请求，且 >200ms 的占比 ≥ 50% 时熔断 10s |
| 异常比例   | 异常比例   | —       | 0.3             | 5s       | 10         | 1000ms   | 1s 内至少 10 次请求，且异常占比 ≥ 30% 时熔断 5s       |
| 异常数     | 异常数     | —       | 3               | 10s      | 5          | 1000ms   | 1s 内至少 5 次请求，且异常数 ≥ 3 时熔断 10s           |

> 💡 以上规则均可在 Dashboard 界面配置。生产环境建议通过 Nacos 推送规则做持久化，避免 Dashboard 重启后规则丢失。

### 1.6 Resilience4j（了解）

Spring Cloud 官方推荐的熔断/重试/限流组件（可替代 Hystrix），与 Sentinel 二选一。

| 注解                | 作用                                                |
| ------------------- | --------------------------------------------------- |
| **@CircuitBreaker** | 熔断：失败率达到阈值后短路，直接走 `fallbackMethod` |
| **@Retry**          | 失败重试，可配次数与间隔                            |
| **@RateLimiter**    | 限流，控制调用速率                                  |

```java
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
@Retry(name = "userService")
@RateLimiter(name = "userService")
public User getUser(Long userId) {
    return userClient.getUserById(userId);
}

public User getUserFallback(Long userId, Exception ex) {
    return new User().setName("默认用户");
}
```

## 二、分布式事务

### 2.1 分布式事务问题

微服务中每个服务有独立数据库，一次业务操作跨多个服务时，本地事务只能保证单库的 ACID，无法跨库协调提交或回滚。

```
服务 A（本地写入） ──▶ 服务 B（远程调用） ──▶ 服务 C（远程调用）
      │                      │                       │
     DB1                    DB2                     DB3
```

**CAP 理论：** 分布式系统在网络分区（P）发生时，只能在一致性（C）和可用性（A）之间二选一：

| 特性                                | 说明                                     |
| ----------------------------------- | ---------------------------------------- |
| **C**onsistency（一致性）           | 所有节点在同一时刻看到相同数据           |
| **A**vailability（可用性）          | 每个请求都能得到响应（不保证是最新数据） |
| **P**artition tolerance（分区容错） | 网络分区时系统仍能运行                   |

分布式事务方案的选择本质是一致性与可用性的取舍：

| 一致性模型     | 说明                                     | 对应 Seata 模式 |
| -------------- | ---------------------------------------- | --------------- |
| **强一致性**   | 事务结束时所有节点数据一致，延迟较高     | XA              |
| **最终一致性** | 允许短暂不一致，最终达到一致，可用性更高 | AT、TCC、SAGA   |

### 2.2 Seata 分布式事务

**Seata** 是阿里巴巴开源的分布式事务解决方案，架构上分三个角色：

| 角色                              | 说明                                                               |
| --------------------------------- | ------------------------------------------------------------------ |
| **TC（Transaction Coordinator）** | 事务协调者，维护全局和分支事务的状态，协调全局事务提交或回滚       |
| **TM（Transaction Manager）**     | 事务管理器，定义全局事务的范围，向 TC 发起开始、提交、回滚全局事务 |
| **RM（Resource Manager）**        | 资源管理器，管理分支事务，向 TC 注册分支事务并报告状态             |

```
┌──────────────────────────────────────────┐
│                   TM                     │
│        （发起方微服务，加                │
│     @GlobalTransactional 的方法）        │
└────────────┬─────────────────────────────┘
             │ ① 开启全局事务 / ④ 请求提交或回滚
             ▼
┌──────────────────────────────────────────┐
│                   TC                     │
│          （Seata Server）                │
└─────┬────────────────────────┬───────────┘
      │ ② 注册分支事务          │ ② 注册分支事务
      │ ③ 报告分支状态          │ ③ 报告分支状态
      ▼                        ▼
┌──────────────┐       ┌──────────────┐
│     RM       │       │     RM       │
│  微服务 A    │       │  微服务 B    │
│   (DB1)      │       │   (DB2)      │
└──────────────┘       └──────────────┘
```

TC 需**独立部署**（即 Seata Server），TM 和 RM 以 SDK 形式嵌入各微服务。

**`@GlobalTransactional` 与 xid 传播：** `@GlobalTransactional` 只标注在 **TM（事务发起方）** 的入口方法上，RM 不需要加。TM 开启全局事务后 Seata 生成全局唯一的 **xid**，通过 Feign / RestTemplate 等客户端的拦截器**自动放入请求头**传播到下游；下游 RM 通过 Seata 数据源代理感知 xid，自动向 TC 注册分支事务并报告状态，业务代码无需改动。

#### 1. Seata Server 部署

Seata Server（TC）需先部署启动，微服务才能作为 TM/RM 连接。

**方式一：Docker（推荐）**

```bash
docker run -d \
  --name seata \
  -p 8091:8091 \
  -p 7091:7091 \
  -e SEATA_IP=<宿主机IP> \
  -v ./seata:/seata-server/resources \
  seataio/seata-server:1.5.2
```

| 参数                                   | 说明                                                  |
| -------------------------------------- | ----------------------------------------------------- |
| **-p 8091:8091**                       | 业务端口，TM/RM 通过该端口连接 TC                     |
| **-p 7091:7091**                       | 控制台（Web UI）端口，默认账号密码均为 `seata`        |
| **-e SEATA_IP**                        | TC 对外暴露的 IP；本机填 `127.0.0.1`，跨机填宿主机 IP |
| **-v ./seata:/seata-server/resources** | 挂载配置目录，宿主机放 `application.yml` 等配置文件   |

> **注意**：挂载会用宿主机目录**覆盖**容器内默认配置。首次使用时需先不挂载启动一个临时容器，将默认配置拷到宿主机目录后再挂载启动：
>
> ```bash
> docker run -d --name seata-temp seataio/seata-server:1.5.2
> docker cp seata-temp:/seata-server/resources/. ./seata/
> docker rm -f seata-temp
> ```

> 💡 官方仅提供 amd64 镜像，Apple Silicon Mac 通过 Rosetta 模拟运行，可加 `--platform linux/amd64` 消除警告。

| 配置项            | 说明                         | 开发环境常用                              | 生产环境常用                 |

**生产环境配置（Nacos + DB）**

生产环境 TC 需注册到 Nacos（微服务通过注册中心发现 TC），事务状态存到 MySQL（支持多实例、数据不丢）。修改 Seata Server 的 `application.yml`：

```yaml
seata:
  registry:
    type: nacos
    nacos:
      application: seata-server        # TC 在 Nacos 中的服务名
      server-addr: <nacos地址>:8848
      group: DEFAULT_GROUP
      namespace: ""
  config:
    type: nacos                        # 配置托管到 Nacos（可选，也可继续用 file）
    nacos:
      server-addr: <nacos地址>:8848
      group: DEFAULT_GROUP
      namespace: ""
      dataId: seataServer.properties   # Nacos 中存放 Seata 配置的 dataId
  store:
    mode: db
    session:
      mode: db
    lock:
      mode: db
    db:
      datasource: druid
      db-type: mysql
```

**TM 端代码（发起方）：**

```java
@Service
public class BizService {

    @Autowired
    private RemoteServiceB remoteServiceB;

    @Autowired
    private RemoteServiceC remoteServiceC;

    @GlobalTransactional(name = "create-biz", rollbackFor = Exception.class)
    public void createBiz(BizDTO dto) {
        // 1. 本地写入（RM 向 TC 注册分支）
        bizMapper.insert(dto);

        // 2. 远程调用服务 B（RM 向 TC 注册分支事务）
        remoteServiceB.bindData(dto.getId(), dto.getParam1());

        // 3. 远程调用服务 C（RM 向 TC 注册分支事务）
        remoteServiceC.bindData(dto.getId(), dto.getParam2());

        // 任一步失败 → TM 通知 TC → TC 协调各 RM 回滚
    }
}
```

> 💡 `@GlobalTransactional` 只加在 TM 端的入口方法上。其他 RM 端只需正常实现业务方法，Seata 数据源代理自动处理分支事务注册与状态上报。

#### 3. 事务模式

| 模式     | 机制                                                            | 侵入性 | 适用场景                     |
| -------- | --------------------------------------------------------------- | ------ | ---------------------------- |
| **AT**   | 自动记录数据快照（undo_log），失败时自动回滚，业务无感          | 无侵入 | 大多数场景（推荐）           |
| **TCC**  | 业务自行实现 Try（预留）、Confirm（确认）、Cancel（回滚）三阶段 | 高侵入 | 高性能、需精细控制资源的场景 |
| **SAGA** | 每个参与者提供正向操作和补偿操作，失败时逐步执行补偿            | 中侵入 | 业务流程长、参与者多         |
| **XA**   | 基于数据库 XA 协议的两阶段提交，由数据库保证强一致              | 无侵入 | 对一致性要求极高的传统场景   |

#### 4. XA 模式

**XA** 是 X/Open 组织定义的**分布式事务处理（DTP）**标准，MySQL、Oracle、PostgreSQL 等主流数据库均支持。Seata 的 XA 模式基于该标准，由数据库原生保证强一致性。

**两阶段流程：**

| 阶段               | 工作内容                                                                                    |
| ------------------ | ------------------------------------------------------------------------------------------- |
| **一阶段（执行）** | 各 RM 执行业务 SQL **但不提交**，向 TC 注册分支并报告状态                                   |
| **二阶段（提交）** | TC 根据各分支状态：**全部成功** → 通知所有 RM **提交**；**任一失败** → 通知所有 RM **回滚** |

**XA 与 AT 对比：**

| 对比项       | XA 模式                           | AT 模式                           |
| ------------ | --------------------------------- | --------------------------------- |
| **一致性**   | 强一致（事务结束即一致）          | 最终一致（二阶段异步补偿）        |
| **一阶段**   | 执行 SQL **不提交**，持有数据库锁 | 执行 SQL **直接提交**，释放本地锁 |
| **锁持有**   | 一阶段即加锁，直到二阶段结束      | 本地锁立即释放，通过全局锁防脏写  |
| **性能**     | 锁持有时间长，易阻塞              | 锁持有时间短，吞吐更高            |
| **undo_log** | 不需要                            | 需要（每个业务库建 undo_log 表）  |
| **代理模式** | `data-source-proxy-mode: XA`      | 默认，可不配                      |
| **适用场景** | 强一致要求高、可接受一定阻塞      | 大多数业务，优先选 AT             |

> 💡 使用 XA 模式需在微服务配置中显式设置 `data-source-proxy-mode: XA`，**不需要**建 undo_log 表。

#### 5. AT 模式

AT 模式是 Seata **默认且最常用**的模式，对业务完全无侵入。Seata 通过数据源代理自动拦截 SQL，记录修改前后的数据快照到 **undo_log** 表，失败时根据快照反向补偿回滚。

**两阶段流程：**

| 阶段               | 操作                                                                   |
| ------------------ | ---------------------------------------------------------------------- |
| **一阶段（执行）** | 执行业务 SQL 并**直接提交**，同时自动记录修改前后的数据快照到 undo_log |
| **二阶段（提交）** | 全局事务成功 → 异步删除 undo_log；失败 → 根据 undo_log 反向补偿回滚    |

AT 一阶段即提交本地事务（不像 XA 一直持有数据库锁），通过 **全局锁** 防止其他全局事务对同一行的脏写，因此锁持有时间更短、吞吐更高。

> 💡 每个参与分布式事务的数据库须创建 **undo_log** 表，建表 SQL 见 [Seata 官方文档](https://seata.apache.org/docs/user/quickstart)。AT 为默认代理模式，`data-source-proxy-mode` 可不配置。

#### 6. TCC 模式

TCC 模式需业务自行实现两阶段：**Try**（预留资源）→ 全局事务成功走 **Confirm**（确认提交），失败走 **Cancel**（回滚释放）。

| 注解                                | 作用                                                   |
| ----------------------------------- | ------------------------------------------------------ |
| **@LocalTCC**                       | 标记接口为 TCC 模式                                    |
| **@TwoPhaseBusinessAction**         | 标注 Try 方法，指定 `commitMethod` 和 `rollbackMethod` |
| **@BusinessActionContextParameter** | 将参数放入上下文，供 Confirm/Cancel 方法使用           |

以资源预留为例，Try 阶段冻结资源（可用额度减少、冻结额度增加），Confirm 扣减冻结额度，Cancel 释放冻结额度：

```java
@LocalTCC
public interface ResourceTccService {
    
    @TwoPhaseBusinessAction(
        name = "deductResource",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "resourceId") Long resourceId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    boolean confirm(BusinessActionContext context);
    
    boolean cancel(BusinessActionContext context);
}

@Service
public class ResourceTccServiceImpl implements ResourceTccService {
    
    @Override
    @Transactional
    public boolean tryDeduct(Long resourceId, Integer count) {
        // Try：冻结资源（可用额度 -count，冻结额度 +count）
        resourceMapper.freeze(resourceId, count);
        return true;
    }
    
    @Override
    @Transactional
    public boolean confirm(BusinessActionContext context) {
        // Confirm：扣减冻结额度（冻结额度 -count）
        Long resourceId = context.getActionContext("resourceId", Long.class);
        Integer count = context.getActionContext("count", Integer.class);
        resourceMapper.deductFrozen(resourceId, count);
        return true;
    }
    
    @Override
    @Transactional
    public boolean cancel(BusinessActionContext context) {
        // Cancel：释放冻结额度（冻结额度 -count，可用额度 +count）
        Long resourceId = context.getActionContext("resourceId", Long.class);
        Integer count = context.getActionContext("count", Integer.class);
        resourceMapper.releaseFrozen(resourceId, count);
        return true;
    }
}
```

TCC 实现时需处理三个问题：

| 问题       | 场景                                                          | 解决方式                                       |
| ---------- | ------------------------------------------------------------- | ---------------------------------------------- |
| **空回滚** | Try 未执行（如超时），TC 仍下发 Cancel                        | Cancel 中判断 Try 是否已执行，未执行则直接返回 |
| **幂等**   | Confirm/Cancel 可能被重复调用（网络重试）                     | 记录事务状态，已处理的直接返回                 |
| **悬挂**   | Cancel 先于 Try 到达（网络延迟），导致 Try 在 Cancel 之后执行 | Try 中检查是否已 Cancel，已 Cancel 则不执行    |

## 三、消息驱动

### 3.1 消息队列（MQ）基础

微服务间同步调用（OpenFeign）耦合度高、峰值流量直接冲击下游。对于**不需要立即拿到结果**的操作（如异步通知、数据同步等），可通过**消息队列（Message Queue，MQ）** 实现异步解耦。

| 对比项       | 同步调用（OpenFeign）  | 消息驱动（MQ）                      |
| ------------ | ---------------------- | ----------------------------------- |
| **耦合度**   | 调用方直接依赖被调用方 | 发送方只依赖消息中间件              |
| **响应方式** | 阻塞等待结果           | 发送即返回，消费端异步处理          |
| **流量冲击** | 峰值直达下游，易压垮   | 消息堆积在 Broker，消费端按能力消费 |
| **失败处理** | 需调用方重试或熔断     | 中间件保证投递，支持死信队列兜底    |

**MQ 核心模型：**

```
Producer（生产者）──▶ Broker（消息代理）──▶ Consumer（消费者）
                        │
                  Topic / Queue
                （消息按主题/队列分类存储）
```

| 概念                   | 说明                                                             |
| ---------------------- | ---------------------------------------------------------------- |
| **Producer（生产者）** | 发送消息的一方                                                   |
| **Consumer（消费者）** | 接收并处理消息的一方                                             |
| **Broker（代理）**     | 消息中间件服务端，负责接收、存储、投递消息                       |
| **Topic / Queue**      | 消息的分类通道；Producer 往 Topic 发，Consumer 从 Topic 订阅消费 |
| **Consumer Group**     | 消费者组，同组内只有一个实例消费同一条消息，避免重复消费         |

**主流消息中间件：**

| 中间件       | 特点                                           | 适用场景                 |
| ------------ | ---------------------------------------------- | ------------------------ |
| **RabbitMQ** | 轻量、延迟低、支持灵活路由（Exchange + Queue） | 中小规模、路由需求丰富   |
| **Kafka**    | 高吞吐、持久化、分区有序                       | 大数据、日志、流式处理   |
| **RocketMQ** | 高吞吐、事务消息、延迟消息                     | 电商、金融等国内主流场景 |

直接使用这些中间件的原生 API 会导致业务代码与具体中间件强绑定，更换中间件时需大量修改。Spring Cloud Stream 通过统一抽象解决了这个问题。

### 3.2 RabbitMQ

**RabbitMQ** 是基于 **AMQP（Advanced Message Queuing Protocol）** 协议的开源消息中间件，以轻量、低延迟、灵活路由著称，是中小规模微服务中最常用的 MQ 之一。

#### 1. AMQP 模型

RabbitMQ 的消息流转遵循 AMQP 模型：**Producer → Exchange → Queue → Consumer**。Exchange 根据类型和路由规则将消息分发到一个或多个 Queue。

```
Producer ── RoutingKey ──▶ Exchange ──匹配──▶ Queue ──▶ Consumer
                              │                 ▲
                              └── Binding ───────┘
                           （BindingKey 定义匹配规则）
```

| 概念                   | 说明                                                    |
| ---------------------- | ------------------------------------------------------- |
| **Producer（生产者）** | 将消息发送到 Exchange                                   |
| **Exchange（交换机）** | 接收消息并根据类型和路由规则分发到 Queue，不存储消息    |
| **Queue（队列）**      | 存储消息，等待消费者消费                                |
| **Binding（绑定）**    | 定义 Exchange 到 Queue 的路由规则，携带 BindingKey      |
| **Consumer（消费者）** | 从 Queue 获取消息并处理                                 |
| **RoutingKey**         | 生产者发送消息时携带的路由键                            |
| **BindingKey**         | 绑定时指定的匹配键，Exchange 用它与 RoutingKey 比较     |
| **Virtual Host**       | 虚拟主机，用于资源隔离（类似数据库的 schema），默认 `/` |

#### 2. Exchange 类型

| 类型        | RoutingKey 匹配规则                                     | 典型场景                                  |
| ----------- | ------------------------------------------------------- | ----------------------------------------- |
| **Direct**  | 精确匹配：RoutingKey 必须与 BindingKey 完全相同         | 定向投递（如按事件类型路由）              |
| **Fanout**  | 忽略 RoutingKey，广播到所有绑定的 Queue                 | 广播通知（如系统公告）                    |
| **Topic**   | 通配符匹配：`*` 匹配一个单词，`#` 匹配零或多个单词      | 按模式分类（如 `log.#` 匹配所有日志事件） |
| **Headers** | 根据消息 header 属性匹配，不使用 RoutingKey（较少使用） | 多属性条件路由                            |

**Direct Exchange：**

```
Producer ── RoutingKey: "error" ──▶ Direct Exchange
                                        │
                           BindingKey: "error" → Queue A → Consumer A
                           BindingKey: "info"  → Queue B → Consumer B
```

只有 Queue A 收到消息（RoutingKey 与 BindingKey 精确匹配）。

**Fanout Exchange：**

```
Producer ──▶ Fanout Exchange
                  │
                  ├──▶ Queue A → Consumer A
                  ├──▶ Queue B → Consumer B
                  └──▶ Queue C → Consumer C
```

所有绑定的 Queue 都收到消息，RoutingKey 被忽略。

**Topic Exchange 通配符：**

| 通配符 | 说明             | BindingKey 示例 | 匹配的 RoutingKey                    |
| ------ | ---------------- | --------------- | ------------------------------------ |
| `*`    | 匹配恰好一个单词 | `log.*`         | `log.info`、`log.error`              |
| `#`    | 匹配零或多个单词 | `log.#`         | `log`、`log.info`、`log.info.detail` |

> 💡 同一个 Queue 可以用不同的 BindingKey 与同一个 Exchange **多次绑定**，匹配任意一条绑定规则的消息都会投递到该 Queue。例如一个告警队列同时绑定 `"error"` 和 `"warning"` 两个 BindingKey，就能同时接收两种级别的消息。

> 💡 默认 Exchange 是 Direct 类型，名为空字符串 `""`。每个 Queue 自动绑定到默认 Exchange，BindingKey 为队列名本身。

**JSON 序列化：**

默认使用 Java 序列化（`SimpleMessageConverter`），推荐改为 JSON：

```java
@Configuration
public class RabbitConfig {

    @Bean
    public MessageConverter messageConverter() {
```

声明为 `@Bean` 的 **MessageConverter 会作为全局转换器**，被 `RabbitTemplate` 和监听容器共用：发送时对 Java 对象做序列化，消费时对消息体做反序列化。发送时是否经过转换器取决于调用的 API：

| 发送方式 | 入参 | 是否经全局转换器 |
| -------- | ---- | ----------------- |
| `convertAndSend(exchange, routingKey, object)` | 业务对象（POJO） | 是，由转换器序列化为消息体 |
| `send(exchange, routingKey, message)` | 已构建的 `Message` | 否，消息原样发往 Broker |

消费端监听器方法参数由容器配置的转换器对消息体反序列化得到。若消息来源并非本系统同一转换器（如其他服务、原始字节），可把方法参数改为 `Message` 或 `byte[]` 自行解析。

| 方法                         | 说明                                         |
| ---------------------------- | -------------------------------------------- |
| `convertAndSend(e, rk, msg)` | 发送消息到指定 Exchange，携带 RoutingKey     |
| `convertAndSend(queue, msg)` | 发送到默认 Exchange，RoutingKey 为队列名     |
| `@RabbitListener(queues)`    | 声明监听指定队列，自动反序列化消息并调用方法 |

**Work Queue（工作队列）：**

多个消费者绑定同一个队列，共同消费消息以提高处理速度。RabbitMQ 默认使用**轮询（Round-Robin）分发**，不考虑消费者处理能力差异。

```
Producer ──▶ Queue ──▶ Consumer 1
                  ──▶ Consumer 2
```

通过 `prefetch` 控制消费者预取数量，处理完一条再取下一条，实现**能者多劳**：

```yaml
spring:
  rabbitmq:
    listener:
      simple:
        prefetch: 1    # 每次只预取 1 条，处理完再取下一条
```

| 配置项       | 说明                                                                          |
| ------------ | ----------------------------------------------------------------------------- |
| **prefetch** | 消费者预取消息数，设为 1 可按处理能力分配（能者多劳），默认无限制（轮询均分） |

#### 5. 声明 Exchange、Queue、Binding

**方式一：@Bean 声明**

```java
@Configuration
public class RabbitConfig {

    // 声明交换机（Direct）
    @Bean
    public DirectExchange demoExchange() {
        return ExchangeBuilder.directExchange("demo.exchange").durable(true).build();
    }

    // 声明队列（持久化）
    @Bean
    public Queue demoQueue() {
        return QueueBuilder.durable("demo.queue").build();
    }

    // 声明绑定（队列 ← 交换机，routingKey = "demo.created"）
    @Bean
    public Binding demoBinding(Queue demoQueue, DirectExchange demoExchange) {
        return BindingBuilder.bind(demoQueue).to(demoExchange).with("demo.created");
    }
}
```

| Builder               | 常用方法                                                                                           |
| --------------------- | -------------------------------------------------------------------------------------------------- |
| **`ExchangeBuilder`** | `directExchange(name)` / `topicExchange(name)` / `fanoutExchange(name)` → `.durable(true).build()` |
| **`QueueBuilder`**    | `durable(name)`（持久化队列）/ `nonDurable(name)`（非持久化）→ `.build()`                          |
| **`BindingBuilder`**  | `bind(queue).to(exchange).with(routingKey)` / `.noargs()`（Fanout 不需要 RoutingKey）              |

> 💡 也可以直接用构造函数：`new DirectExchange("demo.exchange")`、`new Queue("demo.queue", true)`，效果相同。Builder 写法链式调用更清晰，且便于追加 `lazy()`、`deadLetterExchange()` 等额外参数。

***

**方式二：@RabbitListener 注解声明**

在消费者注解中直接声明，更简洁。声明是**幂等的**：队列、交换机若已存在则直接复用并确保绑定存在；若不存在则先创建再绑定。

```java
@RabbitListener(bindings = @QueueBinding(
    value = @Queue(name = "demo.queue", durable = "true"),           // 声明队列
    exchange = @Exchange(name = "demo.exchange", type = ExchangeTypes.DIRECT),  // 声明交换机
    key = "demo.created"                                              // 绑定 routingKey
))
public void handleMessage(String message) {
    log.info("收到消息：{}", message);
}
```

| 方式                     | 优点                   | 缺点               |
| ------------------------ | ---------------------- | ------------------ |
| **@Bean**                | 集中管理，结构清晰     | 配置类代码较多     |
| **@RabbitListener 注解** | 声明与消费在一起，简洁 | 分散在各个消费者中 |

#### 6. 消息可靠性

消息从 Producer 到 Consumer 的完整链路中任何一环都可能丢失，需分段保障：

```
Producer ──①──▶ Exchange ──②──▶ Queue ──③──▶ Consumer
```

| 环节                      | 风险                           | 解决方案                          |
| ------------------------- | ------------------------------ | --------------------------------- |
| **① Producer → Exchange** | 消息发送失败或 Exchange 不存在 | Publisher Confirm                 |
| **② Exchange → Queue**    | 路由失败（无匹配的 Queue）     | Publisher Return                  |
| **③ Queue → Consumer**    | 消费者处理失败                 | Consumer Ack                      |
| **Queue 存储**            | RabbitMQ 宕机丢消息            | 持久化（Exchange + Queue + 消息） |

**生产者重连：**

网络波动可能导致客户端连接 RabbitMQ 失败，可开启连接失败后的重试机制：

```yaml
spring:
  rabbitmq:
    connection-timeout: 1s           # 连接超时
    template:
      retry:
        enabled: true                # 开启发送重试
        initial-interval: 1000ms     # 首次重试间隔
        multiplier: 1                # 重试间隔倍数
        max-attempts: 3              # 最大重试次数
```

> **注意**：重连是**阻塞式**的，等待重连期间当前线程被阻塞，不适合高频发送场景。

***

**Publisher Confirm（发送确认）：**

开启后，消息到达 Exchange 时 Broker 返回 ack/nack：

```yaml
spring:
  rabbitmq:
    publisher-confirm-type: correlated    # 异步回调确认（推荐）
    publisher-returns: true               # 开启 Return 机制
    template:
      mandatory: true                     # 消息路由失败时触发 Return 回调（而非静默丢弃）
```

| publisher-confirm-type 值 | 说明                   |
| ------------------------- | ---------------------- |
| **none**                  | 关闭（默认）           |
| **simple**                | 同步等待确认，性能较低 |
| **correlated**            | 异步回调确认（推荐）   |

```java
@Configuration
public class RabbitConfig {

    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
        RabbitTemplate template = new RabbitTemplate(connectionFactory);
        template.setMessageConverter(new Jackson2JsonMessageConverter());

        // Confirm 回调：消息是否到达 Exchange
        template.setConfirmCallback((correlationData, ack, cause) -> {
            if (!ack) {
                log.error("消息未到达 Exchange, cause: {}", cause);
            }
        });

        // Return 回调：消息到达 Exchange 但未路由到任何 Queue
        template.setReturnsCallback(returned -> {
            log.warn("消息未路由到 Queue, exchange: {}, routingKey: {}, replyCode: {}",
                returned.getExchange(), returned.getRoutingKey(), returned.getReplyCode());
        });

        return template;
    }
}
```

***

**Consumer Ack（消费确认）：**

消费者处理消息后向 Broker 确认，未确认的消息会重新投递：

```yaml
spring:
  rabbitmq:
    listener:
      simple:
        acknowledge-mode: auto             # 默认，正常返回 ack，异常 nack
        retry:
          enabled: true                    # 开启本地重试
          max-attempts: 3                  # 最大重试次数（含首次）
          initial-interval: 1000           # 首次重试间隔（ms）
          multiplier: 2                    # 重试间隔倍数
        default-requeue-rejected: false    # 重试耗尽后不重回队列（配合 DLQ 使用）
```

| acknowledge-mode 值 | 说明                                                |
| ------------------- | --------------------------------------------------- |
| **none**            | 自动确认，消息投递即视为成功（可能丢消息）          |
| **auto**            | 方法正常返回自动 ack，抛异常自动 nack（默认，推荐） |
| **manual**          | 手动确认，需代码调用 `channel.basicAck()`           |

**ack / nack / reject 关系与何时返回：**

| 回执       | 含义                   | 典型触发时机                                                                             |
| ---------- | ---------------------- | ---------------------------------------------------------------------------------------- |
| **ack**    | 处理成功               | 消费者方法正常返回（auto 模式）或手动 `basicAck`                                         |
| **nack**   | 处理失败，可带 requeue | 消费者抛异常时 Spring 发 nack；可指定是否重新入队；支持批量                              |
| **reject** | 处理失败，单条拒绝     | 同 nack 表示“未成功”，但仅单条；RejectAndDontRequeueRecoverer 会发 reject(requeue=false) |

nack 与 reject 都是向 Broker 表示“这条消息没处理成功”。Broker 根据 **requeue** 决定：requeue=true 则重新入队，requeue=false 则丢弃或进入死信队列（若配置了 DLX）。单条且 requeue 相同时，效果等价；nack 多一个“可批量”的能力。

> 💡 `auto` 模式配合本地重试即可覆盖大多数场景：正常处理自动 ack；异常触发重试，重试耗尽后 reject（或 nack，requeue=false）配合 DLQ 兜底。

***

**失败消息处理策略（MessageRecoverer）：**

重试耗尽后消息默认直接丢弃。Spring AMQP 提供三种 `MessageRecoverer` 实现：

| 实现                                 | 失败时对 Broker 的回执              | 行为                                                       |
| ------------------------------------ | ----------------------------------- | ---------------------------------------------------------- |
| **RejectAndDontRequeueRecoverer**    | reject(requeue=false)               | 直接丢弃消息（默认）                                       |
| **ImmediateRequeueMessageRecoverer** | nack(requeue=true)                  | 消息重新入队（可能再次失败，慎用）                         |
| **RepublishMessageRecoverer**        | ack（转发后再对原消息 ack）        | 将失败消息投递到错误交换机，交由人工处理（**推荐**）       |

```
consumer ──重试耗尽──▶ RepublishMessageRecoverer ──▶ error.exchange ──▶ error.queue
                                                                          │
                                                                    人工排查 / 补偿
```

```java
@Bean
public MessageRecoverer messageRecoverer(RabbitTemplate rabbitTemplate) {
    return new RepublishMessageRecoverer(rabbitTemplate, "error.exchange", "error");
}
```

> 💡 需提前声明 `error.exchange` 和用于接收失败消息的队列，并将队列绑定到该交换机、binding key 为 `"error"`（与 RepublishMessageRecoverer 的 routing key 一致）。失败消息会携带异常堆栈信息投递到该队列，便于排查和人工补偿。

***

**持久化：**

RabbitMQ 的 Exchange、Queue 和消息默认存储在内存中，RabbitMQ 服务重启或宕机会导致元数据和消息全部丢失。**持久化**是将这三者写入磁盘，保证服务重启后能自动恢复。三个层级缺一不可：

- Exchange 不持久化 → 重启后交换机消失，生产者发送失败
- Queue 不持久化 → 重启后队列消失，消息无处投递
- 消息不持久化 → 重启后队列还在，但队列中的消息全部丢失

| 对象         | 持久化方式                                                 |
| ------------ | ---------------------------------------------------------- |
| **Exchange** | 声明时 `durable = true`（默认）                            |
| **Queue**    | 声明时 `durable = true`（`new Queue(name, true)`）         |
| **消息**     | `MessageDeliveryMode.PERSISTENT`（Spring AMQP 默认持久化） |

> 💡 Spring AMQP 中 Exchange、Queue、消息默认都是持久化的，通常不需要额外配置。持久化会带来少量磁盘 IO 开销，但对于需要可靠投递的业务消息是必要的。

***

**Lazy Queue（惰性队列）：**

RabbitMQ 默认将消息存储在内存中以降低延迟，但宕机会丢失数据、消息堆积时内存容易不足。从 3.6.0 版本起引入 **Lazy Queue**：消息接收后直接写入磁盘（内存仅保留最近的消息，默认 2048 条），支持存储数百万条消息。**3.12 版本后所有队列默认为 Lazy Queue 模式。**

| 对比项         | 默认队列           | Lazy Queue                         |
| -------------- | ------------------ | ---------------------------------- |
| **消息存储**   | 优先存内存         | 直接写磁盘，内存仅保留少量最新消息 |
| **消息上限**   | 受内存限制         | 支持数百万条                       |
| **消费延迟**   | 低（直接读内存）   | 略高（需从磁盘加载到内存）         |
| **宕机安全性** | 未持久化的消息丢失 | 消息已在磁盘，更安全               |

3.12 以下版本声明时指定 `x-queue-mode` 为 `lazy`。推荐**全 Bean**（队列、交换机、绑定都在配置类）或**全注解**（都用 `@QueueBinding`），避免只声明队列、交换机和绑定分散写导致混淆。

**Bean 方式：**

```java
@Bean
public Queue lazyQueue() {
    return QueueBuilder.durable("xxx.queue")
        .lazy()    // 等价于 .withArgument("x-queue-mode", "lazy")
        .build();
}
// 同配置类中再声明对应的 Exchange、Binding 等
```

**注解方式（@QueueBinding 同时声明 Lazy 队列、交换机、绑定）：**

```java
@RabbitListener(bindings = @QueueBinding(
    value = @Queue(name = "xxx.queue", durable = "true",
        arguments = @Argument(name = "x-queue-mode", value = "lazy")),
    exchange = @Exchange(name = "xxx.exchange", type = ExchangeTypes.DIRECT),
    key = "xxx.routing.key"
))
public void handleMessage(String message) { /* ... */ }
```

> 💡 开启持久化和生产者确认时，RabbitMQ 在消息**持久化完成后**才给生产者返回 ACK 回执，确保持久化与确认的一致性。

#### 7. 消息过期时间（TTL）

**TTL（Time To Live）** 即消息或队列的过期时间。消息在队列中存活超过 TTL 未被消费时，会被丢弃或转发到死信队列（若配置了 DLX）。

| 设置方式   | 说明                                                                 |
| ---------- | -------------------------------------------------------------------- |
| **队列级** | 队列声明时通过参数 `x-message-ttl` 指定，该队列中所有消息使用同一 TTL |
| **消息级** | 发送单条消息时通过属性 `expiration` 指定，仅对该条消息生效           |

同一条消息若同时存在队列 TTL 和消息 TTL，取**两者中较小的值**。未设置 TTL 的消息不会过期。

**队列级 TTL：**

```java
@Bean
public Queue ttlQueue() {
    return QueueBuilder.durable("ttl.queue")
        .ttl(60_000)   // 单位毫秒，此处 60 秒；等价于 .withArgument("x-message-ttl", 60_000)
        .build();
}
```

**消息级 TTL：**

```java
rabbitTemplate.convertAndSend("demo.exchange", "demo.created", payload, message -> {
    message.getMessageProperties().setExpiration("30000");  // 字符串，单位毫秒，30 秒
    return message;
});
```

| 对比项     | 队列级 TTL              | 消息级 TTL                    |
| ---------- | ----------------------- | ------------------------------ |
| **适用场景** | 队列内消息过期时间一致  | 同队列内不同消息不同过期时间   |
| **实现方式** | `QueueBuilder.ttl()` / `x-message-ttl` | `MessageProperties.setExpiration()` |
| **过期后** | 变为死信（配 DLX）或丢弃 | 同左                           |

> **注意**：消息级 TTL 只有在消息即将被投递到消费者时才会检查是否过期。若队列堆积严重，可能已过期的消息仍会先于未过期消息被投递；需要按过期时间严格顺序处理时，可配合**延迟消息**或为不同 TTL 建不同队列。

#### 8. 死信队列（DLQ）

消息变成**死信（Dead Letter）** 的三种情况：

| 条件         | 说明                                         |
| ------------ | -------------------------------------------- |
| **被拒绝**   | 消费者 reject/nack 且 `requeue = false`      |
| **TTL 过期** | 消息或队列设置了 TTL，超时未消费             |
| **队列满**   | 队列达到 `max-length` 上限，最早的消息被丢弃 |

为业务队列绑定 **Dead Letter Exchange（DLX）** 和 **Dead Letter Queue（DLQ）**，死信消息自动路由到 DLQ：

```java
@Configuration
public class DlqConfig {

    // ========== 死信 Exchange + Queue ==========
    @Bean
    public DirectExchange dlxExchange() {
        return ExchangeBuilder.directExchange("dlx.exchange").durable(true).build();
    }

    @Bean
    public Queue dlqQueue() {
        return QueueBuilder.durable("dlq.queue").build();
    }

    @Bean
    public Binding dlqBinding(Queue dlqQueue, DirectExchange dlxExchange) {
        return BindingBuilder.bind(dlqQueue).to(dlxExchange).with("dlq");
    }

    // ========== 业务队列（绑定 DLX） ==========
    @Bean
    public Queue bizQueue() {
        return QueueBuilder.durable("biz.queue")
            .deadLetterExchange("dlx.exchange")     // 指定死信交换机
            .deadLetterRoutingKey("dlq")            // 指定死信路由键
            .build();
    }
}
```

> 💡 DLQ 中的消息可通过专门的消费者读取，进行告警、日志记录或人工重新投递。

#### 9. 业务幂等性

**幂等性**是指同一个业务操作执行一次或多次，对业务状态的影响是一致的。消息可能因网络重试、消费者重启等原因被重复投递，需在消费端保证幂等。

**方案一：唯一消息 ID**

每条消息携带全局唯一 ID，消费者处理成功后记录该 ID，收到相同 ID 的消息直接跳过：

```java
@Bean
public MessageConverter messageConverter() {
    Jackson2JsonMessageConverter converter = new Jackson2JsonMessageConverter();
    converter.setCreateMessageIds(true);  // 自动为每条消息生成唯一 ID
    return converter;
}
```

消费端流程：收到消息 → 查询消息 ID 是否已处理 → 未处理则执行业务并记录 ID → 已处理则直接丢弃。

***

**方案二：业务状态判断（推荐）**

根据业务本身的状态做判断，无需额外存储消息 ID。例如修改数据状态为"已确认"前，先判断当前状态是否为"待处理"：

```java
@RabbitListener(queues = "simple.queue")
public void handleMessage(Long dataId) {
    // WHERE id = ? AND status = '待处理'  →  只有待处理的数据才会被更新
    int rows = bizMapper.updateStatusIfPending(dataId);
    if (rows == 0) {
        log.info("数据已处理，跳过: {}", dataId);
    }
}
```

| 方案             | 优点               | 缺点                            |
| ---------------- | ------------------ | ------------------------------- |
| **唯一消息 ID**  | 通用，适用任何业务 | 需额外存储和查询 ID，有性能开销 |
| **业务状态判断** | 无额外存储，性能好 | 需根据具体业务设计判断逻辑      |

> 💡 生产环境建议在消费端做业务幂等保障的同时，配合**生产者确认 + 消费者确认 + 失败重试 + 持久化**形成完整的可靠性链路。即使 MQ 通知失败，还可通过定时任务查询状态作为兜底方案，确保最终一致性。

#### 10. 延迟消息

RabbitMQ 原生不支持延迟消息，可通过 **DelayExchange 插件**实现；也可用 **TTL + 死信交换机（DLX）** 间接实现，但需为不同延迟时间建不同队列或只能统一 TTL，灵活性差，本节仅介绍推荐的插件方案。

**安装插件：**

**方式一：下载 .ez 插件后拷入容器（沿用现有镜像时）**

1. 在 [GitHub Releases](https://github.com/rabbitmq/rabbitmq-delayed-message-exchange/releases) 下载与当前 RabbitMQ 版本匹配的 `rabbitmq_delayed_message_exchange-*.ez`（如 3.13.x 对应 3.13.0 插件）。
2. 将 `.ez` 拷入容器的插件目录并启用：

```bash
# 在宿主机执行（路径与文件名按实际修改）
docker cp rabbitmq_delayed_message_exchange-3.13.0.ez rabbitmq:/opt/rabbitmq/plugins/
docker exec rabbitmq rabbitmq-plugins enable rabbitmq_delayed_message_exchange
docker restart rabbitmq
```

**方式二：使用已预装该插件的镜像**

使用 Docker Hub 上已包含 `rabbitmq_delayed_message_exchange` 的第三方镜像，或基于官方镜像在 Dockerfile 中 COPY `.ez` 到 `/opt/rabbitmq/plugins/` 并 `enable` 后构建新镜像。创建容器后无需再执行上述拷贝与 enable。

**若镜像中已包含插件**，只需在容器内执行：

```bash
docker exec -it rabbitmq bash
rabbitmq-plugins enable rabbitmq_delayed_message_exchange
```

**声明延迟交换机并消费：**

**方式一：@Bean 声明**

延迟交换机类型为 `x-delayed-message`，需使用 `CustomExchange`，并通过参数 `x-delayed-type` 指定延迟到期后的路由方式（如 `direct`、`topic`）：

```java
@Configuration
public class DelayMQConfig {

    @Bean
    public CustomExchange delayExchange() {
        Map<String, Object> args = new HashMap<>();
        args.put("x-delayed-type", "direct");
        return new CustomExchange("delay.exchange", "x-delayed-message", true, false, args);
    }

    @Bean
    public Queue delayQueue() {
        return QueueBuilder.durable("delay.queue").build();
    }

    @Bean
    public Binding delayBinding(Queue delayQueue, CustomExchange delayExchange) {
        return BindingBuilder.bind(delayQueue).to(delayExchange).with("delay").noargs();
    }
}
```

消费者使用 `@RabbitListener(queues = "delay.queue")` 监听即可。

**方式二：@RabbitListener 注解声明**

```java
@RabbitListener(bindings = @QueueBinding(
    value = @Queue(name = "delay.queue", durable = "true"),
    exchange = @Exchange(name = "delay.exchange", delayed = "true"),
    key = "delay"
))
public void handleDelayMessage(String message) {
    log.info("收到延迟消息: {}", message);
}
```

**发送延迟消息：**

通过 `x-delay` 头指定延迟时间（毫秒）。应使用 `setDelayLong(long)`：Spring AMQP 中 `setDelay(int)` 已废弃，且延迟毫秒数可能很大（如 30 分钟、数天），用 Long 避免溢出。

```java
public void sendDelayMessage(String exchange, String routingKey, Object message, long delayMs) {
    rabbitTemplate.convertAndSend(exchange, routingKey, message, msg -> {
        msg.getMessageProperties().setDelayLong(delayMs);
        return msg;
    });
}
```

**方案对比：**

- **TTL + DLX（无插件）**：为队列设置 TTL 或为消息设置 TTL，到期后消息变为死信并转发到死信交换机（DLX），由绑定在 DLX 上的业务队列消费，从而“间接”实现延迟。缺点是：每条消息的延迟时间若不同，要么为每种 TTL 建一个队列（队列数量多），要么只能按队列级 TTL 统一延迟，灵活性差。
- **DelayExchange 插件**：在交换机层做延迟，消息先留在交换机，到期后再按 `x-delayed-type` 路由到队列；同一队列可接收任意延迟时长的消息，使用简单、精度高，本节上文即为此方案。

| 方案                   | 说明                                                                              |
| ---------------------- | --------------------------------------------------------------------------------- |
| **DelayExchange 插件** | 推荐，Exchange 级延迟，精度高，使用简单                                           |
| **TTL + DLX**          | 无需插件，利用消息 TTL 到期后转死信实现延迟；但不同延迟时间需不同 Queue，灵活性差 |

***

**延迟消息实践：超时任务处理**

业务操作发起后若未在指定时间内确认则自动取消。直接发送一条 30 分钟延迟消息虽然简单，但存在问题：并发高时大量消息在 MQ 中长时间堆积，且多数任务短时间内就完成确认，白白占用资源。

**优化方案：阶梯式延迟检测。** 发送多条递增延迟的消息逐步探测任务状态，已确认则结束，未确认且未超时则发送下一档延迟消息继续探测，超时仍未确认则取消任务并执行回退。

```
发起任务 → 延迟 10s → 检查 → 未确认 → 延迟 10s → 检查 → 未确认 → 延迟 30s → ...
                            → 已确认 → 结束
                                                               → 超时 → 取消任务、执行回退
```

延迟阶梯示例：`10s, 10s, 10s, 15s, 15s, 30s, 30s, 60s, 60s, 120s, 300s, 600s, 600s`（合计约 30 分钟）。

```java
public class DelayCheckHandler {

    // 延迟阶梯（毫秒）
    private static final long[] DELAY_STEPS = {
        10_000, 10_000, 10_000, 15_000, 15_000,
        30_000, 30_000, 60_000, 60_000, 120_000,
        300_000, 600_000, 600_000
    };

    @RabbitListener(bindings = @QueueBinding(
        value = @Queue(name = "delay.check.queue", durable = "true"),
        exchange = @Exchange(name = "delay.exchange", delayed = "true"),
        key = "delay.check"
    ))
    public void handleDelayCheck(DelayCheckMessage msg) {
        BizRecord record = bizMapper.selectById(msg.getBizId());
        if (record == null || record.getStatus() != BizStatus.PENDING) {
            return;  // 记录不存在或已确认/已取消，结束
        }

        int step = msg.getStep();
        if (step >= DELAY_STEPS.length) {
            // 超时未确认，取消任务并执行回退
            bizMapper.cancel(msg.getBizId());
            remoteService.rollback(msg.getBizId());
            return;
        }

        // 发送下一档延迟消息
        msg.setStep(step + 1);
        rabbitTemplate.convertAndSend("delay.exchange", "delay.check", msg, m -> {
            m.getMessageProperties().setDelayLong(DELAY_STEPS[step]);
            return m;
        });
    }
}
```

| 方案           | 优点                                | 缺点                         |
| -------------- | ----------------------------------- | ---------------------------- |
| **单次长延迟** | 实现简单                            | 大量消息长时间堆积，浪费资源 |
| **阶梯式延迟** | 早确认早结束，MQ 压力小，资源利用高 | 实现稍复杂                   |

### 3.3 Spring Cloud Stream

#### 1. 概述

**Spring Cloud Stream** 提供统一的消息中间件编程模型，底层可切换 RabbitMQ、Kafka、RocketMQ 等而不改业务代码。

```
应用代码（StreamBridge / Consumer / Function）
    │
    ▼
Binding（input / output）
    │
    ▼
Binder（适配层，屏蔽中间件差异）
    │
    ▼
消息中间件（RabbitMQ / Kafka / RocketMQ）
```

| 概念            | 说明                                                                 |
| --------------- | -------------------------------------------------------------------- |
| **Binder**      | 消息中间件的适配层，屏蔽底层差异（如 RabbitMQ Binder、Kafka Binder） |
| **Binding**     | 应用与消息中间件之间的桥梁，分 input（消费）和 output（生产）        |
| **Destination** | 消息目的地，对应 RabbitMQ 的 Exchange 或 Kafka 的 Topic              |

新版（Spring Cloud 2022+）推荐**函数式模型**，支持三种函数类型：

| 函数类型             | 作用                                   | 典型用途       |
| -------------------- | -------------------------------------- | -------------- |
| **`Supplier<T>`**    | 定时产生消息（默认每秒触发），较少使用 | 定时轮询推送   |
| **`Function<T, R>`** | 消费一条消息并产出一条消息             | 消息转换、路由 |
| **`Consumer<T>`**    | 消费消息，无返回值                     | 接收事件并处理 |

实际项目中**发送端最常用 `StreamBridge`**（按需发送，不受定时触发限制），**消费端用 `Consumer<T>`**。`Function<T, R>` 适合消息转换场景。

> 💡 旧版 `@EnableBinding` + `@StreamListener` 已废弃，新项目统一使用函数式写法。

#### 2. 添加依赖

根据使用的消息中间件选择对应 Binder starter：

| 中间件       | 依赖                                   |
| ------------ | -------------------------------------- |
| **RabbitMQ** | `spring-cloud-starter-stream-rabbit`   |
| **Kafka**    | `spring-cloud-starter-stream-kafka`    |
| **RocketMQ** | `spring-cloud-starter-stream-rocketmq` |

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-stream-rabbit</artifactId>
</dependency>
```

#### 3. 发送端（StreamBridge）

`StreamBridge.send(bindingName, payload)` 发送消息到指定 binding：

```java
@Service
public class BizService {
    
    @Autowired
    private StreamBridge streamBridge;
    
    public void createData(Object data) {
        bizMapper.insert(data);
        // binding 名称需在 YAML 的 bindings 中配置
        streamBridge.send("data-created", data);
    }
}
```

#### 4. 消费端（Consumer）

声明 `Consumer<T>` 类型的 Bean，Spring Cloud Stream 自动将其与 input binding 关联：

```java
@Configuration
public class EventHandler {
    
    @Bean
    public Consumer<String> dataCreated() {
        return message -> {
            log.info("收到消息：{}", message);
        };
    }
}
```

#### 5. 消息转换（Function）

`Function<T, R>` 消费一条消息并产出一条消息，适用于格式转换、路由等场景：

```java
@Bean
public Function<InputMessage, OutputMessage> transform() {
    return input -> {
        OutputMessage output = new OutputMessage();
        output.setId(input.getId());
        output.setContent("已处理: " + input.getData());
        return output;
    };
}
```

消息流向：`input-topic` → `transform` → `output-topic`。配置中需同时声明 `-in-0` 和 `-out-0` 两个 binding。

#### 6. Binding 配置

```yaml
spring:
  cloud:
    stream:
      bindings:
        # ========== 发送端 binding ==========
        # StreamBridge.send("data-created", payload) 使用此名称
        data-created:
          destination: demo-topic            # 实际 Topic / Exchange 名称
          content-type: application/json
        # ========== 消费端 binding ==========
        # 命名规则：<函数名>-in-<index>
        dataCreated-in-0:
          destination: demo-topic            # 与发送端的 destination 一致
          group: demo-group                  # 消费者组，同组内只有一个实例消费
      function:
        definition: dataCreated              # 声明要激活的函数 Bean 名称，多个用分号分隔
```

**Binding 命名规则：**

| 方向       | 格式                   | 示例（函数名 `dataCreated`） |
| ---------- | ---------------------- | ---------------------------- |
| **input**  | `<函数名>-in-<index>`  | `dataCreated-in-0`           |
| **output** | `<函数名>-out-<index>` | `dataCreated-out-0`          |

> 💡 `index` 从 0 开始，单输入/输出固定为 0。`StreamBridge` 发送时 binding 名直接自定义，不受此规则约束。

| 配置项                  | 说明                                                     |
| ----------------------- | -------------------------------------------------------- |
| **destination**         | 实际的 Topic 或 Exchange 名称                            |
| **content-type**        | 序列化格式，默认 `application/json`                      |
| **group**               | 消费者组，同组内只有一个实例消费同一条消息，避免重复消费 |
| **function.definition** | 声明要激活的函数 Bean 名称，多个用分号分隔               |

> **注意**：未指定 `group` 时，每个消费者实例都会收到消息（广播模式）。生产环境**必须设置 group**，确保同组内只有一个实例消费同一条消息。

#### 7. 消费重试与死信队列

**重试配置：**

消费失败时 Spring Cloud Stream 默认重试 3 次（含首次执行），可通过 `consumer` 配置调整：

```yaml
spring:
  cloud:
    stream:
      bindings:
        dataCreated-in-0:
          destination: demo-topic
          group: demo-group
          consumer:
            concurrency: 5                     # 并发消费线程数，默认 1
            max-attempts: 3                    # 最大重试次数（含首次），设 1 不重试
            back-off-initial-interval: 1000    # 首次重试间隔（ms）
            back-off-multiplier: 2.0           # 重试间隔倍数
            back-off-max-interval: 10000       # 最大重试间隔（ms）
```

| 配置项                        | 说明                                    | 默认值 |
| ----------------------------- | --------------------------------------- | ------ |
| **concurrency**               | 并发消费线程数                          | 1      |
| **max-attempts**              | 最大重试次数（含首次），设 1 表示不重试 | 3      |
| **back-off-initial-interval** | 首次重试间隔（毫秒）                    | 1000   |
| **back-off-multiplier**       | 重试间隔倍数                            | 2.0    |
| **back-off-max-interval**     | 最大重试间隔（毫秒）                    | 10000  |

***

**死信队列（DLQ）：**

重试耗尽后消息默认丢弃。启用 **DLQ（Dead Letter Queue）** 可将失败消息路由到死信队列，便于排查和人工补偿。

**RabbitMQ DLQ：**

```yaml
spring:
  cloud:
    stream:
      rabbit:
        bindings:
          dataCreated-in-0:
            consumer:
              auto-bind-dlq: true              # 自动创建并绑定 DLQ
              republish-to-dlq: true           # 将异常堆栈附加到 DLQ 消息头，便于排查
```

***

**Kafka DLQ：**

```yaml
spring:
  cloud:
    stream:
      kafka:
        bindings:
          dataCreated-in-0:
            consumer:
              enable-dlq: true                 # 启用 DLQ
              dlq-name: demo-topic.dlq         # DLQ Topic 名称（可选，有默认值）
```

> 💡 DLQ 中的消息可通过专门的消费者读取，进行告警、日志记录或人工重新投递。

## 五、微服务最佳实践

### 5.1 服务拆分原则

**什么时候拆分：**

| 项目类型           | 建议                                           |
| ------------------ | ---------------------------------------------- |
| **创业型项目**     | 先采用单体架构快速开发、试错，规模扩大后再拆分 |
| **确定的大型项目** | 可直接选微服务架构，避免后期拆分的成本         |

**拆分目标：** 高内聚、低耦合。**高内聚**指每个微服务职责尽量单一、业务关联度高；**低耦合**指功能相对独立，减少对其它服务的依赖。

**拆分方式与原则：**

| 维度       | 要点       | 说明                         |
| ---------- | ---------- | ---------------------------- |
| **怎么拆** | 纵向拆分   | 按业务模块拆（如用户、订单） |
|            | 横向拆分   | 抽取公共服务，提高复用       |
| **原则**   | 单一职责   | 每个服务只做一件事           |
|            | 业务边界   | 按业务领域划分（DDD）        |
|            | 数据独立   | 每个服务拥有独立数据库       |
|            | 接口稳定   | 服务间通过稳定接口通信       |
|            | 渐进式拆分 | 从单体逐步拆分               |

### 5.2 工程结构

| 方式             | 说明                                                 |
| ---------------- | ---------------------------------------------------- |
| **独立 Project** | 每个微服务一个独立工程，互不依赖，适合大团队独立部署 |
| **Maven 聚合**   | 父工程下多个 module，统一依赖版本、一键构建          |

Maven 聚合的典型目录结构：

```
project-root/
├── pom.xml                    # 父工程（统一管理依赖版本）
├── xxx-common/                # 公共模块（工具类、DTO、Feign 接口等）
│   └── pom.xml
├── xxx-gateway/               # 网关
│   └── pom.xml
├── xxx-service-a/             # 业务服务 A
│   └── pom.xml
└── xxx-service-b/             # 业务服务 B
    └── pom.xml
```

> 💡 公共模块（`xxx-common`）通常存放各服务共用的实体类、DTO、Feign 接口、工具类等，各业务服务通过 Maven 依赖引入。

### 5.3 服务间通信原则

| 场景         | 推荐方式                     | 选用依据                                   |
| ------------ | ---------------------------- | ------------------------------------------ |
| **同步调用** | OpenFeign + LoadBalancer     | 调用方需要等待结果，如查询数据、校验资源   |
| **异步解耦** | 消息队列（RocketMQ / Kafka） | 不需要立即拿到结果，关注吞吐量和解耦       |
| **事件驱动** | Spring Cloud Stream          | 服务间通过事件松耦合，发布方不关心谁来消费 |

### 5.4 微服务登录与用户传递

**整体思路：** 网关统一做登录校验（如 JWT），校验通过后将用户信息写入转发请求的请求头；下游微服务通过拦截器从请求头解析用户并放入 ThreadLocal；Feign 通过 **RequestInterceptor** 从 ThreadLocal 取用户并写入请求头，保证跨服务调用时用户信息不丢失。

```
客户端 ──JWT──▶ 网关（校验 JWT，写 user-info 请求头）
                  │
                  ▼
              微服务 A（拦截器读 user-info → ThreadLocal）
                  │
              Feign 调用（RequestInterceptor 从 ThreadLocal 取用户 → 写请求头）
                  │
                  ▼
              微服务 B（拦截器读 user-info → ThreadLocal）
```

**① 网关：校验后写入请求头**

在网关的 `GlobalFilter` 中校验 JWT，通过后把用户信息放入转发请求的请求头：

```java
ServerWebExchange newExchange = exchange.mutate()
    .request(builder -> builder.header("user-info", userInfoJson))
    .build();
return chain.filter(newExchange);
```

**② 微服务：拦截器读请求头并放入 ThreadLocal**

在公共模块中定义 Spring MVC 拦截器，所有微服务引入公共模块即可生效：

```java
public class UserInfoInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        String userInfo = request.getHeader("user-info");
        if (userInfo != null) {
            UserContext.setUser(userInfo);
        }
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        UserContext.clear();  // 请求结束清除，防止内存泄漏
    }
}
```

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new UserInfoInterceptor());
    }
}
```

**③ OpenFeign 调用时传递用户**

Feign 调用下游时，通过 `RequestInterceptor` 把当前用户从 ThreadLocal 写入请求头：

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String userInfo = UserContext.getUserInfo();
            if (userInfo != null) {
                template.header("user-info", userInfo);
            }
        };
    }
}
```

| 环节           | 做法                                                 |
| -------------- | ---------------------------------------------------- |
| **网关**       | GlobalFilter 校验 JWT，写 `user-info` 到请求头       |
| **微服务内**   | HandlerInterceptor 读请求头，写入 ThreadLocal        |
| **Feign 调用** | RequestInterceptor 从 ThreadLocal 取用户，写入请求头 |

### 5.5 安全实践

网关或微服务内常用 **Spring Security** 的 **OAuth2 Resource Server** 做 JWT 校验：只放行已携带有效 Token 的请求，其余返回 401。

```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()   // 白名单，无需认证
                .anyRequest().authenticated()                    // 其余请求必须已认证
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt());      // 使用 JWT 解析并校验
        return http.build();
    }
}
```

| 维度     | 工具                                     | 说明                                   |
| -------- | ---------------------------------------- | -------------------------------------- |
| **日志** | ELK（Elasticsearch + Logstash + Kibana） | 统一收集、存储、检索各服务日志         |
| **指标** | Prometheus + Grafana                     | 采集 JVM、QPS、延迟等指标并可视化      |
| **追踪** | Zipkin / SkyWalking                      | 跨服务调用链追踪，定位慢接口与故障节点 |
