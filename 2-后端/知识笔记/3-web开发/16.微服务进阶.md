## 一、服务熔断与降级

### 1.1 雪崩效应

**雪崩效应** 是指微服务调用链中某个服务故障（如超时、宕机），导致上游服务线程阻塞、资源耗尽，进而逐级扩散至整条调用链不可用。

```
服务A ──▶ 服务B ──▶ 服务C（故障）
   │         │
   ▼         ▼
 超时阻塞   超时阻塞
   │         │
   ▼         ▼
 资源耗尽   资源耗尽
   │
   ▼
 雪崩
```

常见的高可用手段：

| 手段           | 说明                   |
| -------------- | ---------------------- |
| **多实例部署** | 避免单点故障           |
| **负载均衡**   | 分散压力               |
| **熔断降级**   | 快速失败，防止级联故障 |
| **限流保护**   | 控制流量，保护服务     |
| **健康检查**   | 及时发现异常实例       |

### 1.2 Sentinel 概述

**Sentinel** 是阿里巴巴开源的流量治理组件，以**资源**为核心：对要保护的接口或方法标记资源名，再为资源配置限流、熔断等规则。触发规则时抛出 `BlockException`，需用 `blockHandler` 处理；业务自身的异常则由 `fallback` 处理。

| 概念 / 属性      | 说明                                                                                        |
| ----------------- | ------------------------------------------------------------------------------------------- |
| **资源**          | 用 `@SentinelResource(value = "资源名")` 标记要保护的方法，资源名在控制台配置规则时使用     |
| **blockHandler**  | 处理限流或熔断触发的 `BlockException`，方法签名为原方法参数 + `BlockException`，返回降级结果 |
| **fallback**      | 处理业务异常，方法签名为原方法参数 + `Throwable`，返回降级结果                              |

### 1.4 资源定义与降级

#### 1. @SentinelResource

`value` 定义资源名；触发限流或熔断时走 `blockHandler`；业务抛异常时走 `fallback`。

```java
@Service
public class OrderService {
    
    @SentinelResource(
        value = "getOrder",
        blockHandler = "getOrderBlockHandler",
        fallback = "getOrderFallback"
    )
    public Order getOrder(Long orderId) {
        return orderRepository.findById(orderId);
    }
    
    // 限流/熔断降级（BlockException）
    public Order getOrderBlockHandler(Long orderId, BlockException ex) {
        return new Order().setMessage("服务繁忙，请稍后重试");
    }
    
    // 业务异常降级（Throwable）
    public Order getOrderFallback(Long orderId, Throwable ex) {
        return new Order().setMessage("服务暂时不可用");
    }
}
```

> **注意**：`blockHandler` 和 `fallback` 方法默认须与原方法在**同一类**中，返回值类型一致，参数为原方法参数 + 末尾 `BlockException` / `Throwable`。若要放在其他类中，需通过 `blockHandlerClass` / `fallbackClass` 指定，且方法须为 **static**。

#### 2. Feign 集成 Sentinel

Feign 调用远程服务时，若下游超时或异常，可开启 Feign 的 Sentinel 支持并为 `@FeignClient` 指定 **fallbackFactory**，限流或调用失败时走降级实现，且能拿到异常原因。

```yaml
feign:
  sentinel:
    enabled: true
```

```java
@FeignClient(name = "user-service", fallbackFactory = UserClientFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUserById(@PathVariable("id") Long id);
}

@Component
public class UserClientFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUserById(Long id) {
                log.error("调用用户服务失败: {}", cause.getMessage());
                return new User().setName("未知用户");
            }
        };
    }
}
```

### 1.5 核心规则

| 功能             | 说明                       |
| ---------------- | -------------------------- |
| **流量控制**     | QPS / 线程数限流           |
| **熔断降级**     | 慢调用 / 异常比例 / 异常数 |
| **热点参数限流** | 针对特定参数限流           |
| **系统保护**     | CPU / 负载 / 入口 QPS      |
| **授权规则**     | 黑白名单控制               |

> 💡 Web 请求默认以 URL 路径作为资源名自动注册（如 `/api/orders/{id}`）；`@SentinelResource(value = "xxx")` 手动标记的以 `value` 值注册。应用需有至少一次请求访问到该接口，资源才会出现在 Dashboard 的簇点链路中。

#### 1. 流量控制

Dashboard → **簇点链路** → 对应资源的「流控」按钮，或 **流控规则** → 新增。

**基本字段：**

| 字段         | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| **资源名**   | 要限流的资源名                                           |
| **针对来源** | 调用方微服务名，`default` 表示不区分来源                 |
| **阈值类型** | **QPS**（每秒请求数）或**线程数**（并发线程数）          |
| **单机阈值** | 每台实例的阈值                                           |
| **流控模式** | 直接 / 关联 / 链路                                       |
| **流控效果** | 快速失败 / Warm Up / 排队等待                            |

**流控模式：**

| 模式     | 说明                                                                         | 额外字段   |
| -------- | ---------------------------------------------------------------------------- | ---------- |
| **直接** | 统计当前资源自身，达到阈值即限流                                             | —          |
| **关联** | 关联资源达到阈值时限流当前资源（如写接口触发限流读接口）                     | 关联资源名 |
| **链路** | 仅统计从指定入口进入当前资源的请求                                           | 入口资源名 |

> 💡 使用**链路模式**需将 `web-context-unify` 设为 `false`，否则所有 Web 入口统一为同一上下文，链路区分不生效。

**流控效果：**

| 效果         | 说明                                                                      | 额外字段         |
| ------------ | ------------------------------------------------------------------------- | ---------------- |
| **快速失败** | 直接抛出 `FlowException`（默认）                                          | —                |
| **Warm Up**  | 预热模式，阈值从 `单机阈值 / coldFactor`（coldFactor 固定 3）逐步升到设定值 | 预热时长（秒）   |
| **排队等待** | 匀速排队，超过等待时间的请求被拒绝，适合削峰填谷                          | 超时时间（毫秒） |

**配置示例：**

| 场景       | 资源名     | 阈值类型 | 单机阈值 | 流控模式 | 流控效果   | 额外字段       | 效果                                                            |
| ---------- | ---------- | -------- | -------- | -------- | ---------- | -------------- | --------------------------------------------------------------- |
| QPS 限流   | `getOrder` | QPS      | 100      | 直接     | 快速失败   | —              | 单台实例每秒超过 100 次请求时直接拒绝                           |
| 冷启动预热 | `getOrder` | QPS      | 200      | 直接     | Warm Up    | 预热时长 10s   | 启动后阈值从 200/3 ≈ 66 逐步升至 200，10 秒内完成预热          |
| 关联限流   | `getOrder` | QPS      | 50       | 关联     | 快速失败   | 关联 `addOrder` | `addOrder` 的 QPS 达到 50 时限流 `getOrder`，保护写操作         |

#### 2. 熔断策略

Dashboard → **簇点链路** → 对应资源的「熔断」按钮，或 **熔断规则** → 新增。

**熔断状态流转：**

```
CLOSED（正常）──触发熔断──▶ OPEN（熔断，直接降级）
     ▲                          │
     │                     熔断时长结束
     │                          ▼
     └──探测成功──── HALF_OPEN（放行一次探测）
                        │
                   探测失败 → 回到 OPEN
```

**通用字段：**

| 字段             | 说明                                                     |
| ---------------- | -------------------------------------------------------- |
| **资源名**       | 要熔断的资源名                                           |
| **熔断策略**     | 慢调用比例 / 异常比例 / 异常数                           |
| **熔断时长(s)**  | OPEN 状态持续时间，到期后进入 HALF_OPEN                  |
| **最小请求数**   | 统计时长内请求数达到该值才触发熔断判断，避免少量请求误判 |
| **统计时长(ms)** | 统计窗口，默认 1000ms                                    |

**策略专属字段：**

| 策略           | 专属字段                    | 触发条件                             |
| -------------- | --------------------------- | ------------------------------------ |
| **慢调用比例** | 最大 RT(ms)、比例阈值(0~1) | RT 超过最大 RT 的请求占比 ≥ 比例阈值 |
| **异常比例**   | 比例阈值(0~1)              | 异常请求占比 ≥ 比例阈值              |
| **异常数**     | 异常数                      | 异常请求数 ≥ 异常数阈值              |

> 💡 RT = Response Time（响应时间）。

**配置示例：**

| 场景       | 熔断策略   | 最大 RT | 比例/异常数阈值 | 熔断时长 | 最小请求数 | 统计时长 | 效果                                                   |
| ---------- | ---------- | ------- | --------------- | -------- | ---------- | -------- | ------------------------------------------------------ |
| 慢调用比例 | 慢调用比例 | 200ms   | 0.5             | 10s      | 5          | 1000ms   | 1s 内至少 5 次请求，且 >200ms 的占比 ≥ 50% 时熔断 10s |
| 异常比例   | 异常比例   | —       | 0.3             | 5s       | 10         | 1000ms   | 1s 内至少 10 次请求，且异常占比 ≥ 30% 时熔断 5s       |
| 异常数     | 异常数     | —       | 3               | 10s      | 5          | 1000ms   | 1s 内至少 5 次请求，且异常数 ≥ 3 时熔断 10s            |

> 💡 以上规则均可在 Dashboard 界面配置。生产环境建议通过 Nacos 推送规则做持久化，避免 Dashboard 重启后规则丢失。

### 1.6 Resilience4j（了解）

Spring Cloud 官方推荐的熔断/重试/限流组件（可替代 Hystrix），与 Sentinel 二选一。

| 注解                | 作用                                                    |
| ------------------- | ------------------------------------------------------- |
| **@CircuitBreaker** | 熔断：失败率达到阈值后短路，直接走 `fallbackMethod`     |
| **@Retry**          | 失败重试，可配次数与间隔                                |
| **@RateLimiter**    | 限流，控制调用速率                                      |

```java
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
@Retry(name = "userService")
@RateLimiter(name = "userService")
public User getUser(Long userId) {
    return userClient.getUserById(userId);
}

public User getUserFallback(Long userId, Exception ex) {
    return new User().setName("默认用户");
}
```

## 二、分布式事务

### 2.1 分布式事务问题

微服务中每个服务有独立数据库，一次业务操作跨多个服务时，本地事务只能保证单库的 ACID，无法跨库协调提交或回滚。

```
订单服务（创建订单） ──▶ 库存服务（扣减库存） ──▶ 账户服务（扣款）
    │                        │                       │
   DB1                      DB2                     DB3
```

**CAP 理论：** 分布式系统在网络分区（P）发生时，只能在一致性（C）和可用性（A）之间二选一：

| 特性                                | 说明                                         |
| ----------------------------------- | -------------------------------------------- |
| **C**onsistency（一致性）           | 所有节点在同一时刻看到相同数据               |
| **A**vailability（可用性）          | 每个请求都能得到响应（不保证是最新数据）     |
| **P**artition tolerance（分区容错） | 网络分区时系统仍能运行                       |

分布式事务方案的选择本质是一致性与可用性的取舍：

| 一致性模型     | 说明                                       | 对应 Seata 模式  |
| -------------- | ------------------------------------------ | ---------------- |
| **强一致性**   | 事务结束时所有节点数据一致，延迟较高       | XA               |
| **最终一致性** | 允许短暂不一致，最终达到一致，可用性更高   | AT、TCC、SAGA    |

### 2.2 Seata 分布式事务

**Seata** 是阿里巴巴开源的分布式事务解决方案，架构上分三个角色：

| 角色                                    | 说明                                                               |
| --------------------------------------- | ------------------------------------------------------------------ |
| **TC（Transaction Coordinator）**       | 事务协调者，维护全局和分支事务的状态，协调全局事务提交或回滚       |
| **TM（Transaction Manager）**           | 事务管理器，定义全局事务的范围，向 TC 发起开始、提交、回滚全局事务 |
| **RM（Resource Manager）**              | 资源管理器，管理分支事务，向 TC 注册分支事务并报告状态             |

```
┌──────────────────────────────────────────┐
│                   TM                     │
│        （发起方微服务，加                │
│     @GlobalTransactional 的方法）        │
└────────────┬─────────────────────────────┘
             │ ① 开启全局事务 / ④ 请求提交或回滚
             ▼
┌──────────────────────────────────────────┐
│                   TC                     │
│          （Seata Server）                │
└─────┬────────────────────────┬───────────┘
      │ ② 注册分支事务          │ ② 注册分支事务
      │ ③ 报告分支状态          │ ③ 报告分支状态
      ▼                        ▼
┌──────────────┐       ┌──────────────┐
│     RM       │       │     RM       │
│  微服务 A    │       │  微服务 B    │
│   (DB1)      │       │   (DB2)      │
└──────────────┘       └──────────────┘
```

TC 需**独立部署**（即 Seata Server），TM 和 RM 以 SDK 形式嵌入各微服务。

**`@GlobalTransactional` 与 xid 传播：** `@GlobalTransactional` 只标注在 **TM（事务发起方）** 的入口方法上，RM 不需要加。TM 开启全局事务后 Seata 生成全局唯一的 **xid**，通过 Feign / RestTemplate 等客户端的拦截器**自动放入请求头**传播到下游；下游 RM 通过 Seata 数据源代理感知 xid，自动向 TC 注册分支事务并报告状态，业务代码无需改动。

#### 1. Seata Server 部署

Seata Server（TC）需先部署启动，微服务才能作为 TM/RM 连接。

**方式一：Docker（推荐）**

```bash
docker run -d \
  --name seata \
  -p 8091:8091 \
  -p 7091:7091 \
  -e SEATA_IP=<宿主机IP> \
  -v ./seata:/seata-server/resources \
  seataio/seata-server:1.5.2
```

| 参数 | 说明 |
| ---- | ---- |
| **-p 8091:8091** | 业务端口，TM/RM 通过该端口连接 TC |
| **-p 7091:7091** | 控制台（Web UI）端口，默认账号密码均为 `seata` |
| **-e SEATA_IP** | TC 对外暴露的 IP；本机填 `127.0.0.1`，跨机填宿主机 IP |
| **-v ./seata:/seata-server/resources** | 挂载配置目录，宿主机放 `application.yml` 等配置文件 |

> **注意**：挂载会用宿主机目录**覆盖**容器内默认配置。首次使用时需先不挂载启动一个临时容器，将默认配置拷到宿主机目录后再挂载启动：
>
> ```bash
> docker run -d --name seata-temp seataio/seata-server:1.5.2
> docker cp seata-temp:/seata-server/resources/. ./seata/
> docker rm -f seata-temp
> ```

> 💡 官方仅提供 amd64 镜像，Apple Silicon Mac 通过 Rosetta 模拟运行，可加 `--platform linux/amd64` 消除警告。

**方式二：jar 包启动**

从 [Seata releases](https://github.com/apache/seata/releases) 下载 `seata-server-xxx.zip`，解压后执行：

```bash
sh bin/seata-server.sh -p 8091 -h 192.168.150.101 -m file
```

| 参数 | 说明 |
| ---- | ---- |
| **-p** | 服务端口，默认 8091 |
| **-h** | 本机 IP，即 SEATA_IP |
| **-m** | 存储模式：`file`（单机文件）、`db`（MySQL 等） |

**配置要点**

| 配置项 | 说明 | 开发环境常用 | 生产环境常用 |
| ------ | ---- | ------------ | ------------- |
| **注册中心** | TC 是否注册到注册中心 | `file`（不注册，微服务用 grouplist 直连） | Nacos 等，与项目一致 |
| **存储** | 全局事务与分支事务状态持久化 | `file`（单机文件） | `db`（MySQL），支持多实例 |
| **SEATA_IP / -h** | TC 对外地址 | `127.0.0.1`；Docker 时填宿主机 IP | 由 Nacos 发现或填 TC 节点 IP |

> 💡 开发环境 Seata Server **无需改配置**（默认即 file 模式），只需在微服务的 `application.yml` 里配好事务组和 grouplist 即可。

**生产环境配置（Nacos + DB）**

生产环境 TC 需注册到 Nacos（微服务通过注册中心发现 TC），事务状态存到 MySQL（支持多实例、数据不丢）。修改 Seata Server 的 `application.yml`：

```yaml
seata:
  registry:
    type: nacos
    nacos:
      application: seata-server        # TC 在 Nacos 中的服务名
      server-addr: <nacos地址>:8848
      group: DEFAULT_GROUP
      namespace: ""
  config:
    type: nacos                        # 配置托管到 Nacos（可选，也可继续用 file）
    nacos:
      server-addr: <nacos地址>:8848
      group: DEFAULT_GROUP
      namespace: ""
      dataId: seataServer.properties   # Nacos 中存放 Seata 配置的 dataId
  store:
    mode: db
    session:
      mode: db
    lock:
      mode: db
    db:
      datasource: druid
      db-type: mysql
      driver-class-name: com.mysql.cj.jdbc.Driver
      url: jdbc:mysql://<mysql地址>:3306/seata?rewriteBatchedStatements=true
      user: root
      password: <密码>
      global-table: global_table
      branch-table: branch_table
      lock-table: lock_table
      distributed-lock-table: distributed_lock
```

需在 MySQL 中提前创建 `seata` 库及四张表：

| 表名 | 作用 |
| ---- | ---- |
| **global_table** | 全局事务记录 |
| **branch_table** | 分支事务记录 |
| **lock_table** | 全局锁 |
| **distributed_lock** | 分布式锁（TC 选主等） |

> 💡 建表 SQL 见 [Seata 官方 GitHub - server 端建表脚本](https://github.com/apache/seata/tree/develop/script/server/db)。

#### 2. 微服务接入

所有参与分布式事务的微服务（TM 和 RM）均需添加 Seata 依赖并配置事务组。

**添加依赖：**

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

**开发环境配置（file 直连 TC）：**

```yaml
seata:
  tx-service-group: default_tx_group     # 事务组名称，同一分布式事务内各服务须一致
  service:
    vgroup-mapping:
      default_tx_group: default          # 事务组映射到 TC 集群名
    grouplist:
      default: 127.0.0.1:8091            # TC 地址（端口以实际为准）
  registry:
    type: file
  # data-source-proxy-mode: XA           # 默认 AT；使用 XA 模式须显式设为 XA
```

**生产环境配置（通过 Nacos 发现 TC）：**

```yaml
seata:
  tx-service-group: default_tx_group
  service:
    vgroup-mapping:
      default_tx_group: default
  registry:
    type: nacos
    nacos:
      server-addr: <nacos地址>:8848
      namespace: ""
      group: DEFAULT_GROUP
      application: seata-server          # 与 Seata Server 注册到 Nacos 的服务名一致
      username: nacos
      password: nacos
  # data-source-proxy-mode: XA           # 默认 AT；使用 XA 模式须显式设为 XA
```

| 配置项 | 说明 |
| ------ | ---- |
| **tx-service-group** | 当前微服务所属事务组，同一分布式事务内各服务须一致 |
| **vgroup-mapping** | 事务组 → TC 集群名，集群名与 Seata Server 端一致 |
| **grouplist** | 仅 `registry.type: file` 时生效，写死 TC 地址 |
| **registry** | `file` 直连；`nacos` 时从注册中心查 seata-server 地址 |
| **data-source-proxy-mode** | 数据源代理模式：默认 `AT`（可不配）；使用 XA 模式须显式设为 `XA` |

**TM 端代码（发起方）：**

```java
@Service
public class OrderService {

    @Autowired
    private StockClient stockClient;

    @Autowired
    private AccountClient accountClient;

    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单（本地事务，RM 向 TC 注册分支）
        orderMapper.insert(orderDTO);

        // 2. 扣减库存（远程调用，RM 向 TC 注册分支事务）
        stockClient.deduct(orderDTO.getProductId(), orderDTO.getCount());

        // 3. 扣减余额（远程调用，RM 向 TC 注册分支事务）
        accountClient.debit(orderDTO.getUserId(), orderDTO.getMoney());

        // 任一步失败 → TM 通知 TC → TC 协调各 RM 回滚
    }
}
```

> 💡 `@GlobalTransactional` 只加在 TM 端的入口方法上。库存服务、账户服务等 RM 端只需正常实现业务方法，Seata 数据源代理自动处理分支事务注册与状态上报。

#### 3. 事务模式

| 模式     | 机制                                                             | 侵入性 | 适用场景                       |
| -------- | ---------------------------------------------------------------- | ------ | ------------------------------ |
| **AT**   | 自动记录数据快照（undo_log），失败时自动回滚，业务无感           | 无侵入 | 大多数场景（推荐）             |
| **TCC**  | 业务自行实现 Try（预留）、Confirm（确认）、Cancel（回滚）三阶段  | 高侵入 | 高性能、需精细控制资源的场景   |
| **SAGA** | 每个参与者提供正向操作和补偿操作，失败时逐步执行补偿             | 中侵入 | 业务流程长、参与者多           |
| **XA**   | 基于数据库 XA 协议的两阶段提交，由数据库保证强一致               | 无侵入 | 对一致性要求极高的传统场景     |

#### 4. XA 模式

**XA** 是 X/Open 组织定义的**分布式事务处理（DTP）**标准，MySQL、Oracle、PostgreSQL 等主流数据库均支持。Seata 的 XA 模式基于该标准，由数据库原生保证强一致性。

**两阶段流程：**

| 阶段               | 工作内容                                                                 |
| ------------------ | ------------------------------------------------------------------------ |
| **一阶段（执行）** | 各 RM 执行业务 SQL **但不提交**，向 TC 注册分支并报告状态 |
| **二阶段（提交）** | TC 根据各分支状态：**全部成功** → 通知所有 RM **提交**；**任一失败** → 通知所有 RM **回滚** |

**XA 与 AT 对比：**

| 对比项       | XA 模式                          | AT 模式                              |
| ------------ | -------------------------------- | ------------------------------------ |
| **一致性**   | 强一致（事务结束即一致）         | 最终一致（二阶段异步补偿）           |
| **一阶段**   | 执行 SQL **不提交**，持有数据库锁 | 执行 SQL **直接提交**，释放本地锁    |
| **锁持有**   | 一阶段即加锁，直到二阶段结束     | 本地锁立即释放，通过全局锁防脏写     |
| **性能**     | 锁持有时间长，易阻塞             | 锁持有时间短，吞吐更高              |
| **undo_log** | 不需要                           | 需要（每个业务库建 undo_log 表）     |
| **代理模式** | `data-source-proxy-mode: XA`     | 默认，可不配                         |
| **适用场景** | 强一致要求高、可接受一定阻塞     | 大多数业务，优先选 AT                |

> 💡 使用 XA 模式需在微服务配置中显式设置 `data-source-proxy-mode: XA`，**不需要**建 undo_log 表。

#### 5. AT 模式

AT 模式是 Seata **默认且最常用**的模式，对业务完全无侵入。Seata 通过数据源代理自动拦截 SQL，记录修改前后的数据快照到 **undo_log** 表，失败时根据快照反向补偿回滚。

**两阶段流程：**

| 阶段               | 操作                                                                      |
| ------------------ | ------------------------------------------------------------------------- |
| **一阶段（执行）** | 执行业务 SQL 并**直接提交**，同时自动记录修改前后的数据快照到 undo_log    |
| **二阶段（提交）** | 全局事务成功 → 异步删除 undo_log；失败 → 根据 undo_log 反向补偿回滚      |

AT 一阶段即提交本地事务（不像 XA 一直持有数据库锁），通过 **全局锁** 防止其他全局事务对同一行的脏写，因此锁持有时间更短、吞吐更高。

> 💡 每个参与分布式事务的数据库须创建 **undo_log** 表，建表 SQL 见 [Seata 官方文档](https://seata.apache.org/docs/user/quickstart)。AT 为默认代理模式，`data-source-proxy-mode` 可不配置。

#### 6. TCC 模式

TCC 模式需业务自行实现两阶段：**Try**（预留资源）→ 全局事务成功走 **Confirm**（确认提交），失败走 **Cancel**（回滚释放）。

| 注解                                | 作用                                                         |
| ----------------------------------- | ------------------------------------------------------------ |
| **@LocalTCC**                       | 标记接口为 TCC 模式                                          |
| **@TwoPhaseBusinessAction**         | 标注 Try 方法，指定 `commitMethod` 和 `rollbackMethod`       |
| **@BusinessActionContextParameter** | 将参数放入上下文，供 Confirm/Cancel 方法使用                 |

以库存扣减为例，Try 阶段冻结库存（可用库存减少、冻结库存增加），Confirm 扣减冻结库存，Cancel 释放冻结库存：

```java
@LocalTCC
public interface StockTccService {
    
    @TwoPhaseBusinessAction(
        name = "deductStock",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryDeduct(
        @BusinessActionContextParameter(paramName = "productId") Long productId,
        @BusinessActionContextParameter(paramName = "count") Integer count
    );
    
    boolean confirm(BusinessActionContext context);
    
    boolean cancel(BusinessActionContext context);
}

@Service
public class StockTccServiceImpl implements StockTccService {
    
    @Override
    @Transactional
    public boolean tryDeduct(Long productId, Integer count) {
        // Try：冻结库存（可用库存 -count，冻结库存 +count）
        stockMapper.freezeStock(productId, count);
        return true;
    }
    
    @Override
    @Transactional
    public boolean confirm(BusinessActionContext context) {
        // Confirm：扣减冻结库存（冻结库存 -count）
        Long productId = context.getActionContext("productId", Long.class);
        Integer count = context.getActionContext("count", Integer.class);
        stockMapper.deductFrozen(productId, count);
        return true;
    }
    
    @Override
    @Transactional
    public boolean cancel(BusinessActionContext context) {
        // Cancel：释放冻结库存（冻结库存 -count，可用库存 +count）
        Long productId = context.getActionContext("productId", Long.class);
        Integer count = context.getActionContext("count", Integer.class);
        stockMapper.releaseFrozen(productId, count);
        return true;
    }
}
```

TCC 实现时需处理三个问题：

| 问题       | 场景                                                           | 解决方式                                       |
| ---------- | -------------------------------------------------------------- | ---------------------------------------------- |
| **空回滚** | Try 未执行（如超时），TC 仍下发 Cancel                         | Cancel 中判断 Try 是否已执行，未执行则直接返回 |
| **幂等**   | Confirm/Cancel 可能被重复调用（网络重试）                      | 记录事务状态，已处理的直接返回                 |
| **悬挂**   | Cancel 先于 Try 到达（网络延迟），导致 Try 在 Cancel 之后执行  | Try 中检查是否已 Cancel，已 Cancel 则不执行    |

## 三、链路追踪

### 3.1 链路追踪概述

**链路追踪（Distributed Tracing）** 用于追踪一次请求在多个微服务间的完整调用路径，记录每个服务的处理耗时和调用关系。

| 用途         | 说明                                       |
| ------------ | ------------------------------------------ |
| **问题定位** | 通过 traceId 快速串联所有服务日志，定位故障 |
| **性能分析** | 查看每个服务/方法的耗时，发现慢接口        |
| **依赖分析** | 可视化服务间调用拓扑                       |

**核心概念：**

| 概念        | 说明                                                         |
| ----------- | ------------------------------------------------------------ |
| **Trace**   | 一次完整请求的调用链，由一个全局唯一的 **traceId** 标识      |
| **Span**    | 调用链中的一个工作单元（如一次服务调用），由 **spanId** 标识  |

```
用户请求 ──▶ 网关（spanId=A）──▶ 订单服务（spanId=B）──▶ 库存服务（spanId=C）
              │                     │                       │
              └─── traceId=xxx ─────┴───────────────────────┘
                        （同一 traceId 贯穿整条链路）
```

日志中自动注入 traceId 和 spanId（通过 MDC），排查时用同一个 traceId 即可串联所有服务的日志：

```
INFO [gateway,5c7f3b4a2d1e6f8a,1a2b3c4d]
INFO [order-service,5c7f3b4a2d1e6f8a,5e6f7890]
INFO [stock-service,5c7f3b4a2d1e6f8a,9a0b1c2d]
```

> 💡 OpenFeign、RestTemplate、WebClient 等客户端会自动将 traceId 传播到下游服务，无需额外配置。

### 3.2 Micrometer Tracing + Zipkin（Spring Boot 3.x 推荐）

Spring Cloud 2022+ / Spring Boot 3.x 起，链路追踪迁移到 **Micrometer Tracing**，替代了已停用的 Sleuth。各组件层级关系：

```
应用代码
  │
  ▼
Micrometer Tracing（统一追踪 API）
  │
  ▼
Bridge（适配层，二选一）
  ├── micrometer-tracing-bridge-brave（常用）
  └── micrometer-tracing-bridge-otel（OpenTelemetry）
  │
  ▼
Reporter（上报 Span 数据到后端）
  └── zipkin-reporter-brave → Zipkin Server
```

#### 1. 部署 Zipkin

```bash

  --name zipkin \
  -p 9411:9411 \
  --restart unless-stopped \
  openzipkin/zipkin
```

访问 http://localhost:9411，在界面中可通过 **traceId** 或**服务名**搜索调用链，查看各 Span 的耗时瀑布图和调用顺序。

#### 2. 添加依赖

#### 3. 配置

```yaml

  tracing:
    sampling:
      probability: 1.0        # 采样率，1.0 = 全部采集；生产环境建议 0.1
  zipkin:
    tracing:
      endpoint: http://localhost:9411/api/v2/spans

logging:
  pattern:
    level: "%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]"
```

| 配置项          | 说明                                                                               |
| --------------- | ---------------------------------------------------------------------------------- |
| **probability** | 采样率 0.0~1.0。开发环境用 `1.0` 全采集，生产环境建议 `0.1`（10% 采样）降低性能开销 |
| **endpoint**    | Zipkin Server 接收 Span 数据的地址                                                 |
| **logging.pattern.level** | 自定义日志格式，注入应用名、traceId、spanId                              |

## 四、消息驱动

### 4.1 Spring Cloud Stream

**Spring Cloud Stream** 提供统一的消息中间件编程模型，底层可切换 RabbitMQ、Kafka、RocketMQ 等而不改业务代码。

| 概念            | 说明                                                                     |
| --------------- | ------------------------------------------------------------------------ |
| **Binder**      | 消息中间件的适配层，屏蔽底层差异（如 RabbitMQ Binder、Kafka Binder）     |
| **Binding**     | 应用与消息中间件之间的桥梁，分 input（消费）和 output（生产）            |
| **Destination** | 消息目的地，对应 RabbitMQ 的 Exchange 或 Kafka 的 Topic                  |

新版（Spring Cloud 2022+）推荐**函数式模型**：发送端用 `StreamBridge`，消费端用 `Consumer<T>` Bean。

> 💡 旧版 `@EnableBinding` + `@StreamListener` 已废弃，新项目统一使用函数式写法。

### 4.2 函数式写法

#### 1. 添加依赖

根据使用的消息中间件选择对应 starter：

| 中间件       | 依赖                                          |
| ------------ | --------------------------------------------- |
| **RabbitMQ** | `spring-cloud-starter-stream-rabbit`          |
| **Kafka**    | `spring-cloud-starter-stream-kafka`           |
| **RocketMQ** | `spring-cloud-starter-stream-rocketmq`        |

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-stream-rabbit</artifactId>
</dependency>
```

#### 2. 发送端

`StreamBridge.send(bindingName, payload)` 发送消息到指定 binding：

```java
@Service
public class OrderService {
    
    @Autowired
    private StreamBridge streamBridge;
    
    public void createOrder(Order order) {
        orderMapper.insert(order);
        streamBridge.send("order-created", order);
    }
}
```

#### 3. 消费端

声明 `Consumer<T>` 类型的 Bean，Spring Cloud Stream 自动将其与配置的 input binding 关联：

```java
@Configuration
public class StockEventHandler {
    
    @Bean
    public Consumer<Order> orderCreated() {
        return order -> {
            stockService.deduct(order.getProductId(), order.getCount());
        };
    }
}
```

#### 4. Binding 配置

代码中的 binding 名称需在 YAML 中与实际 Topic/Queue 绑定。以 RabbitMQ 为例：

```yaml
spring:
  cloud:
    stream:
      bindings:
        # 发送端 binding：StreamBridge.send("order-created", payload) 使用此名称
        order-created:
          destination: order-topic     # 实际 Topic/Exchange 名称
          content-type: application/json
        # 消费端 binding：函数名 + "-in-0" 为默认命名规则
        orderCreated-in-0:
          destination: order-topic     # 与发送端的 destination 一致
          group: stock-group           # 消费者组，同组内只有一个实例消费，避免重复
      function:
        definition: orderCreated       # 声明要激活的函数 Bean 名称
```

| 配置项                  | 说明                                                           |
| ----------------------- | -------------------------------------------------------------- |
| **destination**         | 实际的 Topic 或 Exchange 名称                                  |
| **group**               | 消费者组，同组内只有一个实例消费同一条消息，避免重复消费       |
| **function.definition** | 声明要激活的函数 Bean 名称，多个用分号分隔                     |

> 💡 **消费端 binding 命名规则**：`<函数名>-in-<index>`，如函数 Bean 名为 `orderCreated`，则 input binding 名为 `orderCreated-in-0`。

#### 2. Docker Compose

### 5.2 Kubernetes 部署

| 资源               | 作用                                                       |
| ------------------ | ---------------------------------------------------------- |
| **Deployment**     | 声明 Pod 模板、副本数、更新策略，支持滚动更新与回滚       |
| **Service**        | 为一组 Pod 提供稳定的访问入口（ClusterIP / NodePort 等）   |
| **readinessProbe** | 就绪探针，通过后 Pod 才接收流量                            |
| **livenessProbe**  | 存活探针，持续失败则重启容器                               |

Spring Boot Actuator 提供 `/actuator/health/readiness` 和 `/actuator/health/liveness` 端点可直接用于探针。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
```

## 六、微服务最佳实践

### 6.1 服务拆分原则

**什么时候拆分：**

| 项目类型           | 建议                                           |
| ------------------ | ---------------------------------------------- |
| **创业型项目**     | 先采用单体架构快速开发、试错，规模扩大后再拆分 |
| **确定的大型项目** | 可直接选微服务架构，避免后期拆分的成本         |

**拆分目标：** 高内聚、低耦合。**高内聚**指每个微服务职责尽量单一、业务关联度高；**低耦合**指功能相对独立，减少对其它服务的依赖。

**拆分方式与原则：**

| 维度       | 要点           | 说明                         |
| ---------- | -------------- | ---------------------------- |
| **怎么拆** | 纵向拆分       | 按业务模块拆（如用户、订单） |
|            | 横向拆分       | 抽取公共服务，提高复用       |
| **原则**   | 单一职责       | 每个服务只做一件事           |
|            | 业务边界       | 按业务领域划分（DDD）        |
|            | 数据独立       | 每个服务拥有独立数据库       |
|            | 接口稳定       | 服务间通过稳定接口通信       |
|            | 渐进式拆分     | 从单体逐步拆分               |

### 6.2 工程结构

| 方式             | 说明                                               |
| ---------------- | -------------------------------------------------- |
| **独立 Project** | 每个微服务一个独立工程，互不依赖，适合大团队独立部署 |
| **Maven 聚合**   | 父工程下多个 module，统一依赖版本、一键构建        |

Maven 聚合的典型目录结构：

```
project-root/
├── pom.xml                    # 父工程（统一管理依赖版本）
├── xxx-common/                # 公共模块（工具类、DTO、Feign 接口等）
│   └── pom.xml
├── xxx-gateway/               # 网关
│   └── pom.xml
├── xxx-service-a/             # 业务服务 A
│   └── pom.xml
└── xxx-service-b/             # 业务服务 B
    └── pom.xml
```

> 💡 公共模块（`xxx-common`）通常存放各服务共用的实体类、DTO、Feign 接口、工具类等，各业务服务通过 Maven 依赖引入。

### 6.3 服务间通信原则

| 场景         | 推荐方式                     | 选用依据                                       |
| ------------ | ---------------------------- | ---------------------------------------------- |
| **同步调用** | OpenFeign + LoadBalancer     | 调用方需要等待结果，如查询用户信息、校验库存   |
| **异步解耦** | 消息队列（RocketMQ / Kafka） | 不需要立即拿到结果，关注吞吐量和解耦           |
| **事件驱动** | Spring Cloud Stream          | 服务间通过事件松耦合，发布方不关心谁来消费     |

### 6.4 微服务登录与用户传递

**整体思路：** 网关统一做登录校验（如 JWT），校验通过后将用户信息写入转发请求的请求头；下游微服务通过拦截器从请求头解析用户并放入 ThreadLocal；Feign 通过 **RequestInterceptor** 从 ThreadLocal 取用户并写入请求头，保证跨服务调用时用户信息不丢失。

```
客户端 ──JWT──▶ 网关（校验 JWT，写 user-info 请求头）
                  │
                  ▼
              微服务 A（拦截器读 user-info → ThreadLocal）
                  │
              Feign 调用（RequestInterceptor 从 ThreadLocal 取用户 → 写请求头）
                  │
                  ▼
              微服务 B（拦截器读 user-info → ThreadLocal）
```

**① 网关：校验后写入请求头**

在网关的 `GlobalFilter` 中校验 JWT，通过后把用户信息放入转发请求的请求头：

```java
ServerWebExchange newExchange = exchange.mutate()
    .request(builder -> builder.header("user-info", userInfoJson))
    .build();
return chain.filter(newExchange);
```

**② 微服务：拦截器读请求头并放入 ThreadLocal**

在公共模块中定义 Spring MVC 拦截器，所有微服务引入公共模块即可生效：

```java
public class UserInfoInterceptor implements HandlerInterceptor {
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
        String userInfo = request.getHeader("user-info");
        if (userInfo != null) {
            UserContext.setUser(userInfo);
        }
        return true;
    }
    
    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        UserContext.clear();  // 请求结束清除，防止内存泄漏
    }
}
```

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new UserInfoInterceptor());
    }
}
```

**③ OpenFeign 调用时传递用户**

Feign 调用下游时，通过 `RequestInterceptor` 把当前用户从 ThreadLocal 写入请求头：

```java
@Configuration
public class FeignConfig {
    @Bean
    public RequestInterceptor requestInterceptor() {
        return template -> {
            String userInfo = UserContext.getUserInfo();
            if (userInfo != null) {
                template.header("user-info", userInfo);
            }
        };
    }
}
```

| 环节           | 做法                                                 |
| -------------- | ---------------------------------------------------- |
| **网关**       | GlobalFilter 校验 JWT，写 `user-info` 到请求头       |
| **微服务内**   | HandlerInterceptor 读请求头，写入 ThreadLocal        |
| **Feign 调用** | RequestInterceptor 从 ThreadLocal 取用户，写入请求头 |

### 6.5 安全实践

网关或微服务内常用 **Spring Security** 的 **OAuth2 Resource Server** 做 JWT 校验：只放行已携带有效 Token 的请求，其余返回 401。

```java
@Configuration
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()   // 白名单，无需认证
                .anyRequest().authenticated()                    // 其余请求必须已认证
            )
            .oauth2ResourceServer(oauth2 -> oauth2.jwt());      // 使用 JWT 解析并校验
        return http.build();
    }
}
```

### 6.6 可观测性

| 维度     | 工具                                     | 说明                                           |
| -------- | ---------------------------------------- | ---------------------------------------------- |
| **日志** | ELK（Elasticsearch + Logstash + Kibana） | 统一收集、存储、检索各服务日志                 |
| **指标** | Prometheus + Grafana                     | 采集 JVM、QPS、延迟等指标并可视化              |
| **追踪** | Zipkin / SkyWalking                      | 跨服务调用链追踪，定位慢接口与故障节点         |

生产环境建议三者搭配使用。Spring Boot Actuator 暴露 `/actuator/prometheus` 端点供 Prometheus 采集指标；日志建议统一为 JSON 格式输出并通过 Filebeat/Logstash 采集到 Elasticsearch。
