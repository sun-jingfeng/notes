# 17. Elasticsearch

## 一、Elasticsearch 概述

### 1.1 什么是 Elasticsearch

**Elasticsearch（ES）** 是一个基于 Apache Lucene 的分布式、RESTful 的搜索与分析引擎，使用 JSON 文档存储数据，通过倒排索引实现全文检索与聚合分析。常用于站内搜索、日志存储与检索、指标分析、APM 等场景。

**核心特点：**

| 特点           | 说明                                               |
| -------------- | -------------------------------------------------- |
| **全文检索**   | 分词、倒排索引，支持模糊匹配、高亮、建议词         |
| **近实时**     | 默认约 1 秒内可搜索到新写入的文档（refresh 间隔）   |
| **分布式**    | 索引可分片、副本，水平扩展                         |
| **REST API**  | 增删改查、聚合均通过 HTTP + JSON 完成              |
| **聚合分析**   | 支持指标聚合（sum、avg）与桶聚合（terms、date_histogram） |

### 1.2 Elasticsearch vs 关系型数据库（概念对照）

| ES 概念     | 关系型数据库 | 说明                         |
| ----------- | ------------ | ---------------------------- |
| **Index**   | 数据库       | 一类文档的集合               |
| **Document**| 一行记录     | 以 JSON 存储的一条数据       |
| **Field**   | 列           | 文档中的键值对               |
| **Mapping** | 表结构       | 字段类型、分词器等定义      |
| **Type**    | （已废弃）   | 7.x 起一个索引仅建议一种结构 |

### 1.3 常见应用场景

| 场景           | 说明                                       |
| -------------- | ------------------------------------------ |
| **站内搜索**   | 商品、文章、用户等全文检索、筛选、排序     |
| **日志存储**   | ELK 中作为日志存储与检索（配合 Kibana）    |
| **指标分析**   | 时序/指标数据写入后做聚合、仪表盘          |
| **APM**        | 应用性能监控数据存储与查询                |

### 1.4 ELK 与 Lucene 简介

**ELK** 指 Elasticsearch + **Kibana**（数据可视化、Dev Tools）+ **Logstash** / **Beats**（数据采集），一整套技术栈，广泛应用于日志分析、实时监控等。

| 组件              | 作用                     |
| ----------------- | ------------------------ |
| **Elasticsearch** | 存储、检索、聚合数据     |
| **Kibana**        | 可视化、Dev Tools 控制台 |
| **Logstash / Beats** | 数据抓取、导入 ES    |

**Lucene** 是 Apache 的搜索引擎类库（Java），ES 基于 Lucene 构建。Lucene 提供高性能倒排索引与扩展能力，但仅限 Java、不支持分布式；ES 在此基础上提供 REST API、分布式与水平扩展。

## 二、倒排索引与分词

### 2.1 正向索引与倒排索引

**正向索引**：按文档 ID 建索引，查某条记录快，但按词条查找需扫描文档判断是否包含该词（如 `LIKE '%关键词%'`），效率低。

**倒排索引**：对文档内容分词得到**词条（term）**，对词条建索引并记录出现该词条的文档 ID；查询时先按词条查到文档 ID，再根据 ID 取文档。

| 概念       | 说明                                       |
| ---------- | ------------------------------------------ |
| **文档**   | 每条数据即一条文档                         |
| **词条**   | 对文档内容分词得到的词语                   |
| **正向索引** | 基于文档 ID 建索引，按 ID 查快、按词条查慢 |
| **倒排索引** | 对词条建索引并记录文档 ID，按词条先得 ID 再取文档 |

倒排索引是全文检索的基础；创建倒排索引与用户搜索时都会用到**分词器**（对内容分词）。

### 2.2 IK 分词器

中文需要按语义分词，常用 **IK 分词器**。两种模式：**ik_smart**（粗粒度）、**ik_max_word**（细粒度，最细切分）。

在 Kibana Dev Tools 中测试分词：

```http
POST /_analyze
{
  "analyzer": "ik_smart",
  "text": "程序员学习 Java 太棒了"
}
```

| 作用           | 说明                             |
| -------------- | -------------------------------- |
| **建索引时**   | 对文档分词得到词条，用于建倒排索引 |
| **搜索时**    | 对用户输入分词，再去倒排索引中匹配 |

扩展词库：在 IK 插件 `config` 目录下通过 `IKAnalyzer.cfg.xml` 配置 `ext_dict`，指定扩展词典文件（如 `ext.dic`），在词典中每行一个词条即可。

## 三、安装与快速入门

### 3.1 Docker 部署

```bash
# 单节点（开发/测试）
docker run -d \
  --name elasticsearch \
  -p 9200:9200 -p 9300:9300 \
  -e "discovery.type=single-node" \
  -e "xpack.security.enabled=false" \
  -e "ES_JAVA_OPTS=-Xms512m -Xmx512m" \
  --restart unless-stopped \
  docker.elastic.co/elasticsearch/elasticsearch:8.11.0

# 验证：访问 http://localhost:9200 应返回集群信息 JSON
curl http://localhost:9200
```

| 端口  | 说明                     |
| ----- | ------------------------ |
| **9200** | HTTP REST API，应用与 Kibana 连接 |
| **9300** | 节点间通信（Transport），一般可不暴露 |

> 💡 生产环境建议开启 `xpack.security.enabled=true` 并配置用户密码；内存可根据节点配置调整 `-Xms` / `-Xmx`。

### 3.2 使用 Kibana Dev Tools（推荐）

Kibana 提供 **Dev Tools → Console**，可直接写 REST 请求，便于学习与调试。

```bash
# 启动 Kibana（需能访问 ES）
docker run -d --name kibana -p 5601:5601 \
  -e ELASTICSEARCH_HOSTS=http://host.docker.internal:9200 \
  --restart unless-stopped \
  docker.elastic.co/kibana/kibana:8.11.0
```

浏览器打开 http://localhost:5601，左侧 **Dev Tools → Console** 中执行下述示例。

### 3.3 第一个索引与文档

```http

    "number_of_shards": 1,
    "number_of_replicas": 0
  },
  "mappings": {
    "properties": {
      "title":   { "type": "text" },
      "content": { "type": "text" },
      "count":   { "type": "integer" },
      "createAt": { "type": "date" }
    }
  }

# 写入一条文档（指定 _id）
PUT /demo_index/_doc/1
{
  "title": "Elasticsearch 入门",

# 不指定 _id 时由 ES 自动生成

## 四、核心概念

### 4.1 索引（Index）

**索引** 是一类文档的集合，对应一个或多个分片（Shard）。创建索引时可指定 `settings`（分片数、副本数、refresh_interval 等）和 `mappings`（字段类型与分词器）。

- **number_of_shards**：主分片数，创建后不可改。
- **number_of_replicas**：每个主分片的副本数，可动态调整。
- **refresh_interval**：默认 1s，控制写入后多久可被搜索到。

### 4.2 文档与文档元数据

每条数据是一个 **文档**，以 JSON 存储。系统字段以 `_` 开头：

| 元数据     | 说明                         |
| ---------- | ---------------------------- |
| **_index** | 所属索引                     |
| **_id**    | 文档 ID，PUT 时可指定        |
| **_source**| 原始 JSON  body              |
| **_score** | 查询时的相关度分数（仅搜索） |

### 4.3 映射（Mapping）

**Mapping** 定义索引中字段的类型与行为。常见类型：

| 类型       | 说明                                   |
| ---------- | -------------------------------------- |
| **text**   | 全文检索，会分词；不适合排序、聚合    |
| **keyword**| 不分词，精确匹配、排序、聚合          |
| **long / integer / short** | 数值，可聚合、排序           |
| **date**   | 日期，支持多种格式，可聚合、范围查询  |
| **boolean**| 布尔值                                 |
| **nested** | 嵌套对象，子字段可独立查询            |

常见映射属性：**type**（字段数据类型）；**index**（是否建索引，默认 true，设为 false 则不可被检索）；**analyzer**（分词器，如 ik_smart）；**properties**（子字段，用于 object 类型）。

> 💡 未在 mapping 中声明的字段，ES 会做动态推断；生产环境建议显式定义 mapping，避免类型不符合预期。

### 4.4 分片与副本

- **分片（Shard）**：索引被拆成多个分片分布在节点上，便于水平扩展与并行搜索。
- **副本（Replica）**：每个主分片可有 0 个或多个副本，提高可用性与读吞吐。
- 主分片数创建后不可改；副本数可通过 `_settings` 动态修改。

## 五、索引与文档操作

### 5.1 索引管理

```http
# 查看索引
GET /_cat/indices?v

# 查看索引 mapping
GET /demo_index/_mapping

# 查看索引 settings
GET /demo_index/_settings

# 删除索引
DELETE /demo_index
```

索引库与 mapping 一旦创建**无法修改**（不能改已有字段类型等），只能**添加新字段**：

```http
PUT /demo_index/_mapping
{
  "properties": {
    "newField": { "type": "integer" }
  }
}
```

### 5.2 文档 CRUD

```http
# 创建/全量更新（PUT 指定 _id，存在则覆盖）
PUT /demo_index/_doc/1
{ "title": "标题", "content": "内容", "count": 10 }

# 部分更新（只更新指定字段，未写字段保留）
POST /demo_index/_update/1
{
  "doc": { "count": 20 }
}
```

# 按 _id 删除
DELETE /demo_index/_doc/1

# 按查询删除（慎用）
POST /demo_index/_delete_by_query
{
  "query": { "match": { "title": "待删除" } }
}
```

### 5.3 批量写入（Bulk）

大批量写入使用 **Bulk API**，一次请求多行，每行一个操作（index/update/delete），减少网络往返。

```http
POST /_bulk
{"index":{"_index":"demo_index","_id":"2"}}
{"title":"Bulk 文档一","content":"内容一","count":1}
{"index":{"_index":"demo_index","_id":"3"}}
{"title":"Bulk 文档二","content":"内容二","count":2}
```

> 💡 每行必须是合法 JSON；操作行与数据行交替，最后一行需换行。

## 六、DSL 查询

### 6.1 查询结构

查询写在 `query` 下；可选 `sort`、`from/size`（分页）、`_source`（返回字段）、`highlight`（高亮）等。

```http
GET /demo_index/_search
{
  "query": { ... },
  "sort": [ { "createAt": "desc" } ],
  "from": 0,
  "size": 10,
  "_source": ["title", "content"],
  "highlight": {
    "fields": { "title": {}, "content": {} }
  }
}
```

### 6.2 常用查询子句

ES 查询分为**叶子查询**（在特定字段上查特定值，如 match、term、range）和**复合查询**（组合多个叶子查询或改算分，如 bool、function_score）。常用子句：

| 子句        | 说明                                     |
| ----------- | ---------------------------------------- |
| **match**   | 对字段分词后匹配，用于全文检索           |
| **multi_match** | 与 match 类似，可同时在多个字段上查同一关键词；参与字段越多性能越差 |
| **match_phrase** | 短语匹配，词序一致                     |
| **term**    | 不分词，精确匹配（keyword、数值、日期、布尔） |
| **terms**   | 多值精确匹配（相当于 IN）                |
| **range**   | 范围（gt、gte、lt、lte）                 |
| **exists**  | 字段存在                                 |
| **bool**    | 组合 must、should、must_not、filter      |

**multi_match** 示例：

```http
GET /demo_index/_search
{
  "query": {
    "multi_match": {
      "query": "搜索关键词",
      "fields": ["title", "content"]
    }
  }
}
```

### 6.3 bool 组合查询

```http
GET /demo_index/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "content": "搜索 引擎" } }
      ],
      "filter": [
        { "range": { "count": { "gte": 10, "lte": 1000 } } },
        { "term": { "status": "published" } }
      ],
      "must_not": [
        { "term": { "deleted": true } }
      ]
    }
  }
}
```

- **must**：必须满足，参与算分。
- **should**：满足则提高分数；在 bool 内无 must 时至少满足一条。
- **filter**：必须满足，不参与算分，易被缓存，适合条件筛选。
- **must_not**：必须不满足。

### 6.4 排序与分页

- **sort**：按字段排序；`text` 字段排序需使用其 `.keyword` 子字段（若存在）。
- **from + size**：默认只返回 top10；`from` 为起始位置，`size` 为条数。

**深度分页**：from 很大时（如第 99 页每页 10 条，from=990），各分片都需取出并排序大量数据再汇总，性能与内存压力大。可选方案：
- **search_after**：从上一条的排序值开始查下一页，无上限，但只能向后翻页，适合滚动加载、数据迁移。
- **scroll**：将结果形成快照放内存，官方已不推荐，适合一次性导数据。

```http
GET /demo_index/_search
{
  "query": { "match_all": {} },
  "sort": [ { "createAt": "desc" }, { "_id": "asc" } ],
  "from": 0,
  "size": 20
}
```

### 6.5 高亮

```http
GET /demo_index/_search
{
  "query": { "match": { "content": "Elasticsearch" } },
  "highlight": {
    "pre_tags": ["<em>"],
    "post_tags": ["</em>"],
    "fields": {
      "title": {},
      "content": { "fragment_size": 150, "number_of_fragments": 3 }
    }
  }
}
```

返回中会多出 `highlight` 对象，内含用标签包裹的命中片段。

## 七、聚合

聚合写在 `aggs` 下，与 `query` 同级；可嵌套（桶内再聚指标或子桶）。**三要素**：聚合名称、聚合类型、聚合字段。`query` 与 `aggs` 同级时，用于限定参与聚合的文档范围（不写则对全部文档聚合）。

### 7.1 指标聚合

对某字段做 sum、avg、min、max、value_count、cardinality 等。

```http
GET /demo_index/_search
{
  "size": 0,
  "aggs": {
    "total_count": { "sum": { "field": "count" } },
    "avg_count":   { "avg": { "field": "count" } },
    "unique_ids":  { "cardinality": { "field": "_id" } }
  }
}
```

### 7.2 桶聚合

按某维度分组，常见：**terms**（按字段值）、**date_histogram**（按时间间隔）、**range**（按数值区间）。桶内可再嵌套 **aggs** 做指标聚合（如对每组求 sum、stats）。

```http
GET /demo_index/_search
{
  "size": 0,
  "query": { "term": { "category": "手机" } },
  "aggs": {
    "by_status": {
      "terms": { "field": "status", "size": 10 },
      "aggs": {
        "sum_count": { "sum": { "field": "count" } }
      }
    }
  }
}
```

> 💡 对 **text** 字段做 terms 聚合会报错，需使用其 **keyword** 子字段（或 mapping 里单独建 keyword 字段）。

## 八、Spring Boot 集成

### 8.1 依赖与配置

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-elasticsearch</artifactId>
</dependency>
```

```yaml
spring:
  elasticsearch:
    uris: http://localhost:9200
    # 若开启安全：username: elastic  password: xxx
```

### 8.2 文档实体与 Repository

```java
@Document(indexName = "demo_index")
@Data
public class DemoDoc {
    @Id
    private String id;
    @Field(type = FieldType.Text)
    private String title;
    @Field(type = FieldType.Text)
    private String content;
    @Field(type = FieldType.Integer)
    private Integer count;
    @Field(type = FieldType.Date, format = DateFormat.date_time)
    private LocalDateTime createAt;
}

public interface DemoRepository extends ElasticsearchRepository<DemoDoc, String> {
    List<DemoDoc> findByTitle(String title);
    Page<DemoDoc> findByContentContaining(String content, Pageable pageable);
}
```

- **ElasticsearchRepository** 提供 save、findById、delete、分页查询等。
- 方法名解析支持 `findByXxx`、`findByXxxContaining` 等，底层会转成 DSL。

### 8.3 使用 RestClient 写原生 DSL

复杂查询或聚合可用 **ElasticsearchRestTemplate** 或 **RestClient** 直接拼 JSON。

```java
@Autowired
private ElasticsearchRestTemplate restTemplate;

public List<DemoDoc> search(String keyword, int from, int size) {
    NativeSearchQuery query = new NativeSearchQueryBuilder()
        .withQuery(QueryBuilders.matchQuery("content", keyword))
        .withSort(SortBuilders.fieldSort("createAt").order(SortOrder.DESC))
        .withPageable(PageRequest.of(from / size, size))
        .build();
    SearchHits<DemoDoc> hits = restTemplate.search(query, DemoDoc.class);
    return hits.stream().map(SearchHit::getContent).collect(Collectors.toList());
}
```

| 组件                       | 说明                                   |
| -------------------------- | -------------------------------------- |
| **ElasticsearchRepository** | 基于实体 CRUD + 方法名查询             |
| **ElasticsearchRestTemplate** | 基于 Query 对象构建 DSL，适合复杂查询 |
| **RestClient**              | 最底层，完全手写 JSON 请求             |

### 8.4 写入与更新

```java
@Autowired
private DemoRepository demoRepository;

public void save(DemoDoc doc) {
    demoRepository.save(doc);  // 有 id 则更新，无则新增
}

public void saveAll(List<DemoDoc> list) {
    demoRepository.saveAll(list);  // 批量写入
}
```

> 💡 大批量写入可考虑使用 **BulkProcessor** 或 **ElasticsearchRestTemplate** 的 bulk 接口，避免单条 save 造成性能瓶颈。

### 8.5 RestHighLevelClient（7.x，了解即可）

ES 8.x 起官方推荐使用新的 **Java API Client**，**RestHighLevelClient** 已废弃。若维护 7.x 项目仍在使用 RestHighLevelClient，步骤一致：创建对应 Request（如 CreateIndexRequest、IndexRequest、SearchRequest）→ 准备参数（如 mapping、DSL）→ 调用 `client.indices().create()` / `client.index()` / `client.search()` 等发送请求 → 解析响应（如 GetResponse.getSourceAsString()、SearchResponse.getHits()）。查询条件可用 **QueryBuilders**（如 matchQuery、termQuery、boolQuery）构建后传入 `request.source().query(...)`。

## 九、运维与注意事项

### 9.1 集群健康与常用 cat API

```http
GET /_cluster/health?pretty
GET /_cat/indices?v
GET /_cat/nodes?v
GET /_cat/shards?v
```

- **green**：主分片与副本均分配正常；**yellow**：副本未全分配（单节点时常见）；**red**：有主分片未分配，需排查。

### 9.2 注意事项

| 项           | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| **分片数**   | 创建索引时确定，后续不可改；合理规划，避免过多或过少     |
| **text 与 keyword** | 全文检索用 text；排序、聚合、精确匹配用 keyword 或 .keyword |
| **深度分页** | from+size 过深影响性能，优先 limit 控制或改用 search_after |
| **生产配置** | 开启安全、配置 JVM 堆内存、磁盘水位与分片分配策略       |

### 9.3 与 ELK 的关系

Elasticsearch 常与 **Kibana**（可视化、Dev Tools）、**Logstash / Beats**（数据采集）组合为 ELK/Elastic Stack，用于日志与可观测性。ES 负责存储与检索；日志类索引可配合 **Index Lifecycle Management（ILM）** 做冷热分层与删除策略。
