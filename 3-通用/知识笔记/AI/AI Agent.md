# AI Agent

本文介绍 **AI Agent** 的概念、与普通大模型的区别、核心组成、常见架构以及典型应用场景，便于理解当前「智能体」类产品与开发范式。

---

## 一、什么是 AI Agent

**AI Agent（智能体）** 指在给定目标下能自主感知环境、做决策、调用工具并执行多步任务直至达成的系统；其核心是「**目标驱动 + 自主行动**」，而不只是单轮问答。

**与普通 LLM 的对比：**

| 维度       | 普通大模型（LLM）           | AI Agent                          |
| ---------- | --------------------------- | --------------------------------- |
| **交互方式** | 用户提问 → 模型一次回答     | 用户给目标 → Agent 多轮规划与执行 |
| **工具使用** | 无（或需显式在 prompt 里写） | 可主动选择并调用外部工具/API      |
| **任务跨度** | 单轮、单步为主              | 多步、可拆解子任务并迭代执行      |
| **环境感知** | 仅依赖当前输入文本          | 可读状态、观察执行结果并据此调整  |

通俗理解：LLM 像「一问一答的顾问」；Agent 像「领了任务会自己查资料、点按钮、检查结果、直到做完的助手」。

## 二、核心组成

一个典型 Agent 通常包含以下几类能力（不同实现会有所取舍）：

### 2.1 感知与输入

Agent 需要获取「当前状态」与「用户目标」作为输入。

| 能力         | 说明 |
| ------------ | ---- |
| **目标/任务描述** | 用户给出的自然语言目标（如「帮我查仓库里未关闭的 Issue 并汇总」） |
| **环境状态**     | 当前代码库、打开的文件、终端输出、工具返回结果等可被模型读取的信息 |
| **历史与上下文** | 当前会话中的多轮对话、既往决策与执行结果，用于连贯推理 |

### 2.2 规划与推理

根据目标与当前信息，决定「下一步做什么」。

| 方式         | 说明 |
| ------------ | ---- |
| **任务分解** | 把大目标拆成可执行的子步骤（如：先搜代码 → 再读文件 → 再改代码） |
| **链式/迭代** | 执行一步后根据结果决定下一步（成功则继续，失败则重试或换策略） |
| **反思与修正** | 根据工具返回或校验结果，判断是否达成目标，未达成则调整计划 |

常见范式之一为 **ReAct（Reasoning + Acting）**：交替进行「推理（下一步该做什么）」与「行动（调用某个工具）」，用自然语言推理链 + 工具调用来推进任务。

### 2.3 工具调用（Tool Use）

通过调用外部工具获取信息或改变环境，是 Agent 区别于「纯聊天」的关键。

| 概念         | 说明 |
| ------------ | ---- |
| **工具定义** | 向模型暴露的 API：名称、描述、参数 schema，便于模型选择与填参 |
| **执行**     | 由运行时根据模型输出调用真实工具（如搜索、读文件、执行命令、调 API） |
| **结果回填** | 将工具返回结果再次交给模型，作为下一轮推理的输入 |

在 Cursor 等产品中，MCP 提供的 **Tools** 即 Agent 可调用的工具集；模型在对话中自动决定何时调用哪个工具、传何参数。

### 2.4 记忆

用于在多轮交互中保持连贯与可追溯。

| 类型         | 说明 |
| ------------ | ---- |
| **短期/会话记忆** | 当前对话内的历史消息、工具调用与结果，用于上下文连贯 |
| **长期记忆**（可选） | 跨会话的偏好、事实或知识库，供后续任务复用 |

## 三、常见架构与模式

### 3.1 单 Agent + 工具（Tool-augmented LLM）

一个模型 + 一套工具列表；模型每轮决定是否调用工具、调用哪个、参数是什么，执行后再根据结果继续生成。

```
用户目标
    ↓
模型推理 → 生成「调用工具 A(参数)」或「直接回答」
    ↓
执行工具（若有）→ 结果回填模型
    ↓
模型根据结果继续推理/回答 / 再调工具
    ↓
循环直至任务结束或达到步数上限
```

| 特点         | 说明 |
| ------------ | ---- |
| **实现简单** | 只需定义工具 schema 与执行器，与现有 LLM API 易结合 |
| **可控性高** | 工具边界清晰，便于做权限与安全控制 |
| **适用场景** | 查资料、改代码、执行命令、调 API 等「一个模型 + 多种工具」即可完成的场景 |

### 3.2 ReAct 与思维链

**ReAct** 把「推理」与「行动」写进同一输出流：模型先输出一段思考（Reasoning），再输出具体动作（Acting），如调用某工具；执行结果再作为输入，继续下一轮 Thought → Action → Observation。

| 步骤         | 说明 |
| ------------ | ---- |
| **Thought**  | 模型用自然语言说明当前状况与下一步计划 |
| **Action**   | 选择工具并给出参数（结构化或自然语言，由运行时解析） |
| **Observation** | 工具执行结果，作为下一轮的输入 |

这样既便于人类理解 Agent 的决策过程，也便于在出错时从「哪一步推理或哪次调用错了」入手排查。

### 3.3 多 Agent 协作（可选了解）

多个 Agent 各司其职（如：一个负责规划、一个负责写代码、一个负责审查），通过消息或共享状态协作完成复杂任务。适合分工明确、流程固定的场景，但架构与调试成本更高。

## 四、典型应用场景

| 场景             | 说明 |
| ---------------- | ---- |
| **代码开发**     | 根据需求自动检索代码库、编辑文件、运行命令、修复 Lint；如 Cursor Composer/Agent、GitHub Copilot Agent |
| **信息检索与汇总** | 连接搜索、数据库、API，多步查询并整理成报告或答案 |
| **自动化运维**   | 查日志、执行脚本、重启服务、根据结果决定是否告警或回滚 |
| **客服与工单**   | 查知识库、调业务接口（订单、库存等）、生成回复或转人工 |
| **个人助理**     | 日历、邮件、待办、提醒等跨应用编排与执行 |

***

## 五、在 Cursor 中的体现

Cursor 的 **Composer / Agent 模式** 即典型的「单 Agent + 工具」：以当前项目为环境，以用户自然语言目标为输入，模型可自动执行多步操作。

| 能力         | 对应 Agent 概念 |
| ------------ | ----------------- |
| **多文件编辑、创建、删除** | 通过「编辑类工具」改变环境（代码库） |
| **运行终端命令** | 调用执行类工具，并根据输出决定下一步 |
| **@Codebase、@File、MCP Tools** | 感知与工具：读取代码、调用 MCP 暴露的 API |
| **多轮对话与步骤列表** | 规划与记忆：拆解任务、保留上下文、迭代执行 |

配合 **Rules** 与 **Skills** 可约束 Agent 行为与流程；配合 **MCP** 可扩展其可调用的工具集（如 GitHub、Figma、数据库等），从而在编码场景中实现「说清目标 → Agent 自主执行多步直至完成」的体验。
