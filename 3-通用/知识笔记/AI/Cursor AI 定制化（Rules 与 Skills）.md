## 一、三种定制手段对比

Cursor 提供三种方式影响 AI 的行为，定位各不相同：

| 方式 | 本质 | 触发时机 | 适用场景 |
| ----------- | ----------------------------------- | -------------------------- | -------------------------------- |
| **Rules**   | 随 session 自动注入的约束/上下文    | 每次对话（或打开特定文件） | 编码规范、项目约定、通用风格     |
| **Skills**  | AI 按需读取的操作手册               | AI 判断当前任务需要时       | 有固定流程的专项任务             |
| **MCP**     | AI 可调用的外部工具/数据源          | AI 判断需要调用外部能力时   | 访问 GitHub、Figma、数据库等外部服务 |

> 💡 Rules 解决"AI 应该怎么做"，Skills 解决"AI 如何做某件特定的事"，MCP 解决"AI 能做什么（能力边界）"。

***

## 二、Rules

### 2.1 什么是 Rules

**Rules** 是写在 `.cursor/rules/` 目录下的 `.mdc` 文件，每次对话时 Cursor 会自动将匹配的 Rule 注入 AI 的上下文，相当于持久化的"系统提示词"。

适合放在 Rules 里的内容：编码风格、命名规范、技术栈约定、禁止某些做法等——凡是希望 AI 在整个项目里始终遵守的内容。

### 2.2 文件格式

Rules 文件是带 YAML frontmatter 的 Markdown：

```
.cursor/rules/
    typescript-standards.mdc
    react-patterns.mdc
    api-conventions.mdc
```

```markdown
---
description: 简要描述这条 Rule 的作用（显示在 Rule 选择器中）
globs: **/*.ts        # 文件匹配模式，仅打开匹配文件时生效
alwaysApply: false    # true 表示每次对话都生效
---

# Rule 标题

Rule 内容...
```

| 字段 | 说明 |
| --------------- | ----------------------------------------------- |
| `description`   | Rule 的简述，显示在设置面板和 Rule 选择器中     |
| `globs`         | 文件 glob 模式，匹配时自动应用                  |
| `alwaysApply`   | `true` 则无论打开什么文件都注入                 |

### 2.3 触发类型

| 类型 | 配置方式 | 适用场景 |
| ---------------- | ------------------------------------------ | ------------------------------- |
| **始终生效**     | `alwaysApply: true`                        | 全局编码规范、通用 Git 约定     |
| **文件触发**     | `globs: **/*.ts`（不设 `alwaysApply`）     | 语言/框架专属约定               |
| **手动选择**     | `alwaysApply: false` 且不设 `globs`        | 仅在需要时手动勾选启用          |

### 2.4 示例

**全局代码规范（始终生效）：**

```markdown
---
description: 全局代码规范
alwaysApply: true
---

# 全局规范

- 函数名用 camelCase，类名用 PascalCase
- 不用 `var`，用 `const` / `let`
- 异步函数必须有错误处理，不允许空 catch
```

**TypeScript 专属（打开 .ts 文件时生效）：**

```markdown
---
description: TypeScript 编码规范
globs: **/*.ts
alwaysApply: false
---

# TypeScript 规范

- 始终声明函数返回类型
- 用 `interface` 而非 `type` 定义对象结构（除联合类型外）
- 禁止使用 `any`，用 `unknown` + 类型收窄代替
```

### 2.5 最佳实践

| 实践 | 说明 |
| ---------------------- | ---------------------------------------------------- |
| **单条 Rule 单一职责** | 一个文件只管一类约定，不要把所有规范塞进一个文件    |
| **控制长度**           | 建议 50 行以内，过长会占用 context window           |
| **给出反例**           | 用 `// ❌` / `// ✅` 对比，比纯文字描述更清晰       |
| **只写 AI 不知道的**   | 通用最佳实践不必重复写，重点放项目特有约定           |

## 三、Skills

### 3.1 什么是 Skills

**Skills** 是存放在 `.cursor/skills/` 目录下的操作手册，核心文件是 `SKILL.md`。与 Rules 不同，Skills **不会自动注入**，而是由 AI 根据当前任务判断是否需要读取——只有任务与 Skill 描述匹配时才会被加载，更节省 context。

适合放在 Skills 里的内容：有明确步骤的专项工作流，如"按某种格式生成笔记"、"按团队规范写 commit message"、"生成符合设计稿的组件"等。

### 3.2 目录结构

```
.cursor/skills/
    skill-name/
        SKILL.md          # 必须，AI 读取的主指令文件
        reference.md      # 可选，详细参考资料（AI 按需读取）
        examples.md       # 可选，示例
        scripts/          # 可选，辅助脚本
```

### 3.3 存储位置

| 类型 | 路径 | 作用范围 |
| ---------- | ---------------------------------- | -------------------------------- |
| **项目级** | `.cursor/skills/<skill-name>/`     | 仅当前项目，可随仓库版本管理     |
| **个人级** | `~/.cursor/skills/<skill-name>/`   | 所有项目共用，个人私有           |

> **注意**：`~/.cursor/skills-cursor/` 是 Cursor 内置 Skills 目录，由系统管理，不要在此创建自定义 Skills。

### 3.4 SKILL.md 格式

```markdown
---
name: skill-name              # 小写字母、数字、连字符，最长 64 字符
description: 描述这个 Skill 做什么、何时使用（AI 用此决策是否调用）
---

# Skill 标题

## 操作步骤
具体的步骤说明...

## 示例
...
```

### 3.5 description 是关键

`description` 决定 AI 能否在合适时机发现并调用该 Skill。写法要点：

- **第三人称**：描述 Skill 本身，不要用"你可以"、"我可以"
- **同时写 WHAT 和 WHEN**：说明能做什么，以及什么情况下应该用它
- **包含触发词**：用户可能说的关键词要出现在描述里

```yaml
# ✅ 好的 description
description: 生成符合项目规范的知识笔记，包含格式、结构和深度要求。
  对 /path/to/notes 中所有笔记相关的操作生效（生成、扩写、修改、润色等）。

# ❌ 模糊的 description
description: 帮助写笔记
```

### 3.6 示例

**知识笔记生成 Skill（项目级）：**

```
.cursor/skills/knowledge-notes/
    SKILL.md       ← 格式规范、内容结构要求（主指令）
    reference.md   ← 完整示例笔记（AI 按需参考）
```

```markdown
---
name: knowledge-notes
description: 生成或修改技术知识笔记时遵循的格式与内容规范。
  适用于笔记的生成、扩写、修改、润色等操作。
---

# 知识笔记规范

## 格式要求
- 标题从二级开始（##），最深四级（####）
- 大章节间用 `***` 分隔
- 频繁使用表格对比、归纳

## 内容深度
- 偏实战，重点在"怎么用"
- 每个概念：一句话定义 → 表格归纳 → 代码示例
```

### 3.7 最佳实践

| 实践 | 说明 |
| ---------------------- | -------------------------------------------------- |
| **SKILL.md 控制在 500 行内** | 过长影响性能，细节放 reference.md      |
| **渐进式披露**         | 主文件放核心步骤，详细资料放附属文件，AI 按需读取 |
| **description 要具体** | 模糊描述导致 AI 无法发现或误用 Skill              |
| **提供具体示例**       | 示例比文字说明更有效，尤其是有固定输出格式的任务  |
| **不写 AI 已知的内容** | 只补充项目特有知识，通用最佳实践不必重复           |

## 四、如何选择

```
这个需求是…
    │
    ├── 需要 AI 访问外部服务（GitHub、Figma、数据库）？
    │       └── → MCP
    │
    ├── 是始终要遵守的约定（编码规范、命名规则、禁止某种写法）？
    │       └── → Rules
    │
    └── 是有固定步骤/模板的专项任务（写特定格式的笔记、按团队规范写 commit）？
            └── → Skills
```

三者可以组合使用：Rule 约束通用规范，Skill 提供专项操作指南，MCP 补充外部工具能力，互不冲突。
