# Cursor

本文介绍 **Cursor** 是什么、核心功能、常用操作与模型选择，便于快速上手并在编码中用好 AI 能力。

---

## 一、什么是 Cursor

**Cursor** 是基于 VS Code 的 AI 代码编辑器，在保留 VS Code 的编辑、调试、扩展生态基础上，内置多模型对话、代码补全、多文件编辑与终端执行等 AI 能力，实现「写代码 + 问 AI + 自动改代码」在同一环境内完成。

**核心价值**：减少在编辑器与浏览器/API 之间切换，让 AI 直接基于当前项目、选中代码和对话上下文进行补全、重构与答疑。

| 维度       | 说明 |
| ---------- | ---- |
| **底层**   | 基于 VS Code，兼容大部分 VS Code 扩展与快捷键 |
| **AI 能力**| 对话（Chat）、多文件编辑（Composer/Agent）、行内补全（Tab）、代码库检索等 |
| **模型**   | 支持 OpenAI、Anthropic、Google 等多家模型，可按对话或全局切换 |

***

## 二、核心功能

### 2.1 对话（Chat）

在侧边栏与 AI 对话，可基于当前文件、选中代码或整个代码库回答问题、生成代码、解释逻辑。

| 能力         | 说明 |
| ------------ | ---- |
| **@ 引用**   | 用 `@Codebase` 检索代码库、`@File` 引用文件、`@Web` 查网络、`@Docs` 查文档，为 AI 提供精确上下文 |
| **多标签**   | 多开聊天标签（如 ⌘T / Ctrl+T），不同任务独立上下文，便于并行处理 |
| **引用与跳转** | AI 回复中的代码引用可点击跳转到对应文件与行号 |

对话适合：单点答疑、小范围修改、理解某段逻辑、按自然语言查代码。

### 2.2 Composer / Agent

**Composer**（多文件编辑）支持在一个会话里跨文件编辑、创建文件、运行终端命令（默认需用户确认），适合实现一个完整功能或重构一片代码。

| 特性         | 说明 |
| ------------ | ---- |
| **多文件**   | 一次任务可修改、新建多个文件，AI 自动维护一致性 |
| **终端**     | 可生成并执行命令（如安装依赖、跑测试），执行前可确认或修改 |
| **Lint 修复**| 自动检测并尝试修复当前改动触发的 Lint 报错 |
| **子任务拆分** | 复杂需求会被拆成多步，逐步执行并汇总结果 |

> 💡 Agent 模式强调「端到端完成任务」：从需求到改代码、跑命令、修 Lint，尽量一气呵成，仍由用户做最终确认与提交。

### 2.3 行内补全（Tab）

在编辑器中输入时，AI 会给出单行或多行补全建议，按 **Tab** 接受。

| 能力         | 说明 |
| ------------ | ---- |
| **多行补全** | 可一次补全多行代码或整段逻辑 |
| **光标预测** | 补全后预测下一个光标位置，便于连续 Tab |
| **智能纠错** | 对明显拼写或简单逻辑错误给出修正建议 |

适合：写重复结构、补全函数体、按注释或上下文续写。

### 2.4 代码库检索（Codebase Indexing）

Cursor 会索引当前项目代码，供 Chat 与 Composer 使用，无需手动贴大量文件即可「问代码库」「按语义找代码」。

| 能力         | 说明 |
| ------------ | ---- |
| **语义搜索** | 用自然语言描述意图，检索相关代码位置 |
| **@Codebase** | 在对话中通过 @Codebase 让 AI 只基于索引结果回答，减少幻觉 |
| **索引范围** | 可在设置中排除 `node_modules`、构建产物等，加快索引并减少噪音 |

> **注意**：大仓库首次索引可能耗时数分钟，索引完成后检索与补全效果更好。

## 三、常用操作

### 3.1 打开方式

| 入口         | 说明 |
| ------------ | ---- |
| **Chat**     | 侧边栏聊天图标，或快捷键（如 ⌘L / Ctrl+L，以实际设置为准） |
| **Composer** | 顶部 Composer 入口，或从 Chat 中进入「在 Composer 中打开」 |
| **行内补全** | 编辑时直接出现灰色建议，Tab 接受 |

### 3.2 为 AI 提供上下文

- **选中代码**：在对话前选中一段代码，AI 会默认带上这段内容。
- **@ 引用**：`@文件名`、`@Codebase`、`@Folder`、`@Web`、`@Docs` 等，按需组合。
- **当前文件**：未选中时，当前打开或最近编辑的文件往往会被纳入上下文（视产品行为而定）。

### 3.3 模型与用量

在 **设置 → Models** 中可配置：

| 配置项       | 说明 |
| ------------ | ---- |
| **默认模型** | 对话、Composer 使用的默认模型（如 Claude、GPT 等） |
| **按会话切换** | 单次对话可临时切换为其他模型 |
| **用量与计费** | 使用 Cursor 订阅或自带 API Key 时，可在设置或官网查看用量与成本 |

## 四、与纯编辑器的对比

| 对比项       | 传统 IDE / 编辑器     | Cursor |
| ------------ | --------------------- | ------ |
| **补全**     | 多为语法/片段补全     | 语义补全、多行、可带项目上下文 |
| **改代码**   | 手动查找、批量替换    | 自然语言描述需求，由 AI 跨文件修改 |
| **答疑**     | 需切到浏览器/文档     | 在编辑器内对话，可 @Codebase / @Docs |
| **执行命令** | 自己敲终端            | 可让 AI 生成命令并在终端执行（需确认） |
| **扩展**     | 依赖插件生态          | 继承 VS Code 扩展，并增加 AI 能力层 |

***

## 五、使用建议

| 场景             | 建议 |
| ---------------- | ---- |
| **查用法、解释代码** | 用 Chat，配合 @File / 选中代码 / @Docs |
| **实现一个小功能** | Chat 或 Composer，需求说清输入输出与约束 |
| **跨文件重构/新模块** | 用 Composer（Agent），一次交代清楚范围和规则 |
| **日常敲代码**   | 多用 Tab 补全，减少重复劳动 |
| **规范与项目约定** | 使用 Rules 约束行为；固定流程用 Skills；需 GitHub、Figma 等外部服务时配置 MCP |
