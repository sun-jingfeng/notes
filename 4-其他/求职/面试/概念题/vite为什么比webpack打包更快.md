# Vite 为什么比 Webpack 打包更快？

Vite 比 Webpack 打包更快，主要原因有以下几点：

## 1. 开发模式：原生 ESM vs Bundle

| Webpack | Vite |
| :--- | :--- |
| 启动时需要打包整个应用 | 利用浏览器原生 ES Modules，按需编译 |
| 冷启动慢，项目越大越慢 | 冷启动极快，与项目大小关系不大 |

- **Webpack 启动流程**：
  入口 → 分析依赖 → 打包所有模块 → 启动服务器 → 可访问

- **Vite 启动流程**：
  启动服务器 → 可访问 → 按需编译请求的模块

## 2. 依赖预构建 (Dependency Pre-Bundling)

Vite 使用 `esbuild` 对 `node_modules` 依赖进行预构建：
- `esbuild` 用 Go 编写，比 JavaScript 编写的打包器快 10-100 倍。
- 只在依赖变化时重新构建，结果会被缓存。

## 3. HMR (热模块替换) 更快

| Webpack | Vite |
| :--- | :--- |
| 修改文件后需要重新构建整个 bundle | 只需让浏览器重新请求该模块 |
| 项目越大，HMR 越慢 | HMR 速度与项目大小无关，始终很快 |

## 4. 生产构建：Rollup

生产环境 Vite 使用 Rollup 打包：
- Tree-shaking 更高效。
- 输出更优化的代码。
- 原生支持 ES Modules。

```text
│ └──────┘     │ 按需编译     │                           │

│              (即时启动,按需加载)                        │
└─────────────────────────────────────────────────────────┘
```

## 总结

| 特性 | Webpack | Vite |
| :--- | :--- | :--- |
| **开发启动** | 慢（全量打包） | 快（原生 ESM） |
| **HMR** | 慢（重新打包） | 快（模块级别） |
| **依赖处理** | JS 打包器 | esbuild（Go） |
| **适合场景** | 复杂配置、老项目 | 新项目、快速开发 |
