# 虚拟 DOM 与 Diff 算法

## 一、虚拟 DOM 是什么

**虚拟 DOM（Virtual DOM）** 是用 **JavaScript 对象** 对真实 DOM 的一层抽象描述。React 将组件树先转成虚拟 DOM 树（一般包含 type、props、key、children 等），在内存中对比前后两棵虚拟 DOM 的差异（**Diff**），再**只把变更应用到真实 DOM**，从而减少直接操作 DOM 的次数，并便于跨平台（如 React Native）。

***

## 二、为什么用虚拟 DOM

- **集中批量更新**：先算好所有变更，再一次性写回 DOM，减少重排重绘。
- **抽象层**：不依赖具体 DOM API，同一套组件逻辑可渲染到不同环境。
- **声明式**：开发者写“状态 → UI”，由框架负责“怎么更新 DOM”。

***

## 三、Diff 策略（React）

React 的 Diff 基于以下假设和策略，将 O(n³) 的树比较优化到接近 O(n)：

### 1. 同层比较

只比较**同一层级**的节点，不跨层移动。若同一层节点类型不同，直接**删除旧节点、创建新节点**；类型相同则复用并更新属性，再递归比较子节点。

### 2. 类型判断

- **节点类型（如 tag）不同**：视为不同组件/元素，直接**销毁旧节点、创建新节点**，不继续比较子树。
- **节点类型相同**：复用该节点，更新 props 等，再对子节点列表做 Diff。

### 3. key 优化列表

对**列表**进行 Diff 时，通过 **key** 识别“同一项”，从而尽量复用 DOM、仅做顺序或增删，避免“按索引一一对比”导致的错误复用或大量重绘。

```jsx
// ✅ 用稳定、唯一的 key（如 id）
{items.map(item => <Item key={item.id} data={item} />)}

// ❌ 用 index 作 key：列表增删、重排时会导致错误复用与状态错乱
{items.map((item, index) => <Item key={index} data={item} />)}
```

**key 不要用 index** 的原因：在列表中间插入或删除时，index 会整体变化，React 会误以为“同一位置的元素还是同一个”，导致组件实例复用错、状态错乱或性能浪费。

***

## 四、面试答题要点

- **虚拟 DOM**：用 JS 对象描述 DOM 结构，在内存中做 Diff，再最小化更新真实 DOM。
- **Diff 策略**：同层比较、类型不同则替换、类型相同则复用并更新属性、列表用 key 标识元素。
- **key**：用稳定唯一 id，不用 index，避免列表增删重排时的错误复用和状态问题。
