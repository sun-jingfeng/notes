# 高并发下如何防止超卖

## 一、问题

高并发下对同一库存**并发扣减**，若只读再写、无并发控制，会出现**超卖**（库存被扣成负或下单数大于库存）。

***

## 二、常见方案

| 方案 | 说明 |
| --- | --- |
| **数据库乐观锁** | 表加 version 或直接用库存条件：`UPDATE stock SET num=num-1 WHERE id=? AND num>=1`，根据影响行数判断是否扣成功；或 `UPDATE ... SET num=num-1, version=version+1 WHERE version=?` |
| **Redis 预扣减** | 库存同步到 Redis，用 **DECR** 等原子操作扣减；扣成功再异步或同步落库、创建订单；需处理 Redis 与 DB 一致性与恢复 |
| **分布式锁** | 按商品 ID 加锁（如 Redisson），保证同一商品同一时刻只有一个线程扣减；锁内查库存、扣减、写库 |
| **MQ 串行化** | 请求先入 MQ，单消费者或按商品 ID 分区消费，串行扣减，避免并发写 |

***

## 三、组合使用

可组合：**Redis 扣减**做前置快速校验与限流，**DB 乐观锁或分布式锁**做最终一致与强一致；或 **MQ 削峰 + 数据库乐观锁**。

***

## 四、面试答题要点

- **乐观锁**：UPDATE 带条件（库存≥1 或 version），根据影响行数判断。
- **Redis**：原子 DECR、预扣减；需与 DB 一致性与恢复方案。
- **分布式锁**：按商品加锁，保证单线程扣减。
- **MQ**：削峰、串行消费，再配合 DB 乐观锁。
