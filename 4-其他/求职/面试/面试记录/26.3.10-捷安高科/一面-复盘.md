### 3. 跨 iframe（FM）边界截图实现思路

**问题**：简历里「跨 FM 边界截图」具体实现思路？

**我的答案要点**
- 已有截图组件基于 html2canvas，可拖动选区、调整大小后生成图。
- 问题：父子 iframe 下父无法直接截子内容，html2canvas 是重绘 DOM 不是真截图。
- 方案：截图插件在父项目，子项目点击截图用 postMessage 通知父（同源），父启动插件、拖拽选区；用 requestAnimationFrame 优化、处理鼠标出界和状态初始化；通过父子通信跨越 iframe 限制（后半段被打断）。

**评价**
- 优点：能说清 html2canvas 限制和 postMessage 设计，细节到位。
- 不足：最终「父子各自截再合成」的策略未说完整；可补 origin 校验、多子项目统一协议。

**参考答案要点**
- 问题本质：html2canvas 在当前 document 重绘 DOM，无法穿透 iframe。
- 方案：父项目放统一截图控制器，子项目只发起指令；子 postMessage → 父显示遮罩+选区，用 requestAnimationFrame 优化拖拽；若需截子内容，子在 iframe 内截自身区域，把图或区域信息 postMessage 回父，父按布局合成。
- 安全与扩展：postMessage 校验 origin，统一消息协议，支持多子系统共用。

**话术示例**
这个问题本质在于：html2canvas 是在当前 document 把 DOM 重绘成图片，它是看不到 iframe 里面的 DOM 的，所以父页面没法直接截子页面。
我的做法是把截图能力集中在父项目做成一个统一截图控制器，子项目只是发起「我要截图」这个指令。
具体流程是：子页面通过 postMessage 通知父页面，父页面收到后加一层全屏遮罩，用户在父页面上拖拽选区，拖拽过程中用 requestAnimationFrame 做动画和性能优化。
如果只需要截父页面，直接在父页面用 html2canvas 重绘选区即可；如果同时要截子页面，子页面在自己的 iframe 内用 html2canvas 截出对应区域，把图片数据或区域坐标通过 postMessage 传给父页面，父页面根据布局把多张图片合成一张。
同时会对 postMessage 做 origin 校验，并且约定统一的消息协议，这样多个子系统都可以共用这一套截图能力。

### 4. 项目数据量、表结构、如何抗数据量

**问题**：数据量多大？核心业务表怎么存？怎么解决数据量大的问题？

**我的答案要点**
- 本地部署，总量几十 G。
- 表主要是用户、权限、气象名词、预警等；大量气象数据由算法产出 JSON/文件，不落库；色斑图按规则拼 URL，前端用插件解析。
- 对「数据量大」的应对主要是离线处理 + 文件 + 规则拼 URL，而不是塞进数据库。

**评价**
- 优点：解释了为什么表不复杂、数据在文件系统的领域特性；算法前置 + 规则拼 URL 思路正确。
- 不足：面试官还想听「若核心表条数很大，从索引、分表、归档等角度怎么考虑」，这部分几乎没涉及。

**参考答案要点**
- 先说明现状：气象原始数据在文件/对象存储，库主要存元数据与业务数据（用户、权限、预警、任务、文件索引等）。
- 若核心表数据条数很大：合理索引（时间、区域、要素等）；热点与冷数据分离、历史归档；必要时时间/区域分表；读多时用 Redis 缓存配置与规则；列表分页限制深度。

**话术示例**
我们现在的存储结构是这样的：气象的原始格点数据、雷达图这些体量比较大的数据，更多是放在文件系统或者对象存储里，数据库里主要存元数据和业务数据，比如用户、权限、预警记录、任务调度、文件索引这些。
如果把场景放宽，假设某些核心业务表数据量非常大，我一般会从几个方向来考虑：
第一是索引，把常用的查询维度，比如时间、区域、要素类型等建成组合索引，保证常见查询都能走索引。
第二是冷热分离和历史归档，把历史很久的数据按时间归档到历史表或历史库，线上只保留近一段时间的热数据。
第三是必要的时候按时间或区域做分表，避免单表行数过大。
第四是对读多写少的配置类、规则类数据，用 Redis 做缓存，配合分页限制深度查询，这样整体在数据量上来之后也能比较平稳。

### 5. 通用组件的基本标准

**问题**：封装组件给团队共用，你认为通用组件的基本标准是什么？怎样做出大家都能用的好组件？

**我的答案要点**
- 对业务熟，知道哪些会反复出现（如时间轴）；共性封装、差异用参数/插槽；新项目通过配置即可复用又能满足差异；组件内部做好拆分、组合式函数/hooks。

**评价**
- 优点：抓住了「共性抽象 + 差异参数化」。
- 不足：未提易用性和协作——API 设计、文档、默认值、可插拔、兼容性。

**参考答案要点**
- 标准：复用性（覆盖多数场景）、可配置性（Props/插槽/回调暴露差异）、易用性（默认值、命名、文档与 Demo）、稳定性（内部处理边界与异常）。
- 实践：先在一两个场景验证再推广；收集团队反馈，平衡通用与易用。

**话术示例**
我做通用组件，一般会先想清楚几个维度。
第一是复用性，它要能覆盖团队里大多数类似场景，而不是只为一个页面定制。
第二是可配置性，把差异点通过 Props、插槽和回调暴露出来，让使用方可以按需组合，而不用改组件内部。
第三是易用性，像属性命名要直观，有合理的默认值，配一份简单的文档和 Demo，团队同学看一眼就知道怎么用。
第四是稳定性，在组件内部把边界情况和异常处理好，避免在业务代码里到处判断。
落地的时候我会先在一两个业务场景里试用，收集大家的反馈，再根据大家实际使用的痛点去微调 API 设计，在通用和易用之间做一个平衡。

### 6. Vue3 浅层响应式（shallowRef / shallowReactive）

**问题**：简历里「浅层响应式 API 减少深层响应开销」用的是 Vue3 的哪个功能？怎么用的？

**我的答案要点**
- 用 shallowRef、shallowReactive。
- 场景：地图等大对象是第三方实例，只需监听「是否创建好」，不需要深度侦听内部属性；接手时发现大量用 ref/reactive 导致性能问题，把这类大对象改为浅层响应式。

**评价**
- 优点：场景和 API 正确，能说清「只监听引用」的原因。
- 不足：可补充何时仍需 watch 关键字段、类型/使用上的约束。

**参考答案要点**
- API：shallowRef、shallowReactive。
- 场景：大对象（地图实例、chart 实例等），不需对内部属性做细粒度响应式。
- 用法：如 mapRef = shallowRef(null)，只在实例 ready 时触发 UI；内部属性由第三方管理。
- 效果：避免深层 Proxy 与依赖收集，减少无效更新，缓解卡顿。

**话术示例**
这里用的是 Vue3 的浅层响应式 API，比如 shallowRef 和 shallowReactive。
像地图实例、图表实例这类大对象，其实我们只关心「有没有创建好」「引用有没有变化」，不需要对内部每个属性做细粒度响应式，所以用普通的 ref/reactive 会有很多不必要的依赖收集和更新。
我会把这类对象改成 shallowRef，比如 `const mapRef = shallowRef(null)`，只在实例 ready 或重新创建时更新它，触发一次 UI 刷新；实例内部的状态由第三方库自己管理，不让 Vue 去深度代理。
这样可以明显减少深层 Proxy 带来的开销，避免因为一些无关紧要的内部变化导致整棵组件树频繁更新，整体性能会好很多。

### 7. 高频排序场景的前端性能优化（不考虑后端）

**问题**：数据量大、筛选排序高频，仅从前端角度，除了浅层响应式还有哪些优化手段？

**我的答案要点**
- 提到用 key 保证 DOM 复用；被追问后只补充了「能想到的就是 key」。

**评价**
- 优点：key 是一个点。
- 不足：未从「计算→渲染→交互」系统回答；结果缓存、虚拟列表、Web Worker、防抖节流等都没提到。

**参考答案要点**

按「排序算在哪」分两种场景：

- **排序在后端**（常规做法）  
  - 渲染：给列表项绑稳定 **key**，保证 diff 时 DOM 复用，减少不必要的销毁和重建。  
  - 交互：对排序操作做 **防抖**（多次点击只发最后一次请求）；用 **AbortController** 取消前一次未完成的请求，只保留最后一次调用。

- **排序在前端**（如题目限定仅从前端考虑）  
  - 优化长 JS 执行、避免卡顿：把重计算放到 **Web Worker**，单独线程算，不阻塞主线程。  
  - 对同一条件的结果做 **缓存**：优先 **内存缓存**（如 Map，按排序/筛选条件做 key），同一会话内来回切换不用重算；若需跨刷新或数据较静态，可再用 **Storage** 做持久缓存。

**话术示例**
我会先区分排序是后端做还是前端做。  
排序在后端的话，前端主要做两件事：一是列表项用稳定的 key，保证 DOM 复用；二是排序操作防抖，用户连续点击时只发最后一次请求，并且用 AbortController 把前面未完成的请求取消掉，避免无效请求和竞态。  
如果排序在前端算，就要避免长任务卡住主线程，可以把排序放到 Web Worker 里算；同时按条件把结果缓存起来，一般用内存缓存就够了，同一套条件切回来直接用缓存，需要的话再考虑用 Storage 做持久化。

### 9. 大文件 / 高刷新度数据的后端处理与用户体验

**问题**：大电子文件需解析、实时渲染、前端还要实时滚动更新和保存，同时保证体验，怎么考虑？

**我的答案要点**
- 色斑图等大数据提前处理成 PNG，前端直接加载；少量实时预警点：后端实时读文件+插值，控制在 500ms 内，用线程池（8 线程）、结果缓存 Redis；更大耗时的都提前跑脚本。

**评价**
- 优点：「能预处理的尽量离线 + 在线路径分块/并行/缓存」思路对；和项目实际一致。
- 不足：可再点一下「分块加载、增量更新」等与「实时滚动、实时保存」的配合。

**参考答案要点**
- 能预处理的离线处理，缩短在线路径。
- 在线：大文件分块加载、先首屏再异步解析推送；重计算用线程池+限流；结果缓存（Redis/本地）；实时保存用异步/批量写入，避免每次滚动都同步写盘。

**话术示例**
遇到大文件或者高刷新度的数据，我会先把能离线做的事情尽量前移，用脚本或离线任务预处理，缩短在线请求要走的链路。
在线部分，我会把大文件做分块加载，优先保证首屏可用，然后后台异步解析剩余部分，再逐步推送到前端；重计算的地方放在线程池里并做限流，避免瞬时把资源打满。
一些中间结果会做缓存，比如 Redis 或本地缓存，避免重复计算。
至于实时滚动和实时保存，我会把保存做成异步或批量写入，而不是每滚动一下就同步写盘，这样既能保证数据安全，又不会把用户交互卡住。

### 10. 为什么线程池是 8？如何评估线程池大小？

**问题**：线程池设为 8 是怎么评估的？数据量大、用户多时如何评估某业务线程池的线程数？

**我的答案要点**
- 当前并发不高，「8 个够用」、一般跑不满。
- 高并发时：提升服务器/CPU、用动态线程池、实在不行提前脚本预处理。
- 评估：用压力测试工具模拟高并发，看响应时间；响应时间很长说明抗压不足（工具名未说出）。

**评价**
- 优点：知道用压测；有动态线程池、提升硬件的方向。
- 不足：未从「CPU 核数 + 任务类型（CPU/IO）+ 期望利用率」系统讲；压测指标可补 CPU、QPS、错误率、队列长度。

**参考答案要点**
- 原则：CPU 密集型 ≈ 核数 N 或 N+1；IO 密集型 ≈ 核数 × (1 + IO/CPU)，可先 2N～4N。先讲原理，不直接报死一个数。
- 做法：用 **`Runtime.getRuntime().availableProcessors()`** 获取核数；本业务是 CPU 密集型，核数小于 8 用核数（或 +1），核数大于等于 8 取 8 作为上限（环境一般不超过 8 核，再往上收益不大）。
- 留余地：结合 JMeter/wrk 压测与监控（响应时间、CPU、队列、错误率）再微调。
- 加分点：`availableProcessors()` 返回逻辑核数（含超线程）；容器内做了 CPU limit 时，部分 JDK 会按限制后的核数返回。

**话术示例**
我们这块是 CPU 密集型，所以线程数不会设得比核数多太多，一般是核数或者核数加一，避免上下文切换开销。
具体实现上用 `Runtime.getRuntime().availableProcessors()` 拿核数，核数小于 8 就用核数或核数+1，核数大于等于 8 就取 8，因为我们环境一般不超过 8 核，对 CPU 密集型来说 8 个线程基本够用，再往上加收益不大。
实际还会结合压测和监控再微调，不会只靠公式定死。如果面试官追问，可以补一句：availableProcessors 返回的是逻辑核数；在容器里做了 CPU 限制的话，有的 JDK 会按限制后的核数返回。

### 11. 分布式事务：何时用 Seata？如何权衡？

**问题**：项目里有用分布式事务吗？是否引入 Seata 这类组件？如何权衡？

**我的答案要点**
- 项目规模小，多用本地事务；知道 Seata，有「Java 本地事务」和「分布式事务工具 Seata」的区分。
- 主张先单体，规模大、多省复用时再微服务和分布式事务；未给出「是否用 Seata」的清晰标准。

**评价**
- 优点：有「先单体后拆分」的工程观；知道 Seata 存在和运维成本。
- 不足：未区分「跨库/跨服务强一致」与「可接受最终一致」；未提本地消息表、事务消息、SAGA 等替代。

**参考答案要点**
- 判断：跨服务必须强一致（如资金）可考虑 Seata/TCC；可接受短时不一致则优先本地事务 + 可靠消息（本地消息表、事务消息）做最终一致。
- Seata 代价：需部署维护集群、业务侵入、排障复杂。
- 建议：小规模、跨库不多时优先本地事务+可靠消息；只有强一致关键链路再考虑 Seata，且仅用在必要处。

**话术示例**
我判断要不要上 Seata 这类分布式事务组件，主要看两个点：是不是跨服务/跨库，以及业务对强一致性的要求。
如果是类似资金这类链路，多服务之间必须强一致，那可以考虑用 Seata 做 AT/TCC 这一类模式，但同时要接受它在部署、运维、排障上的成本，以及对业务代码的侵入。
如果业务可以接受短时间的不一致，我会优先用本地事务加可靠消息的方式来做最终一致，比如本地消息表或者事务消息，既减轻了系统复杂度，又能满足大部分业务需求。
总体建议是：小规模或者跨库不多的系统，尽量用本地事务 + 可靠消息解决，只有在少数关键强一致链路上再考虑引入 Seata，而且只用在必要的地方。

### 12. AI 解决复杂技术问题（雷达可视化性能）— 角色 / 上下文 / 思维链

**问题**：如何用 AI 解决复杂技术难题？Prompt 怎么设计？在「雷达海量数据实时可视化、性能差」的场景下，如何把问题描述成 AI 可处理的思维链？

**我的答案要点**
- 在 Cursor skills 里写固定背景：技术栈、气象软件特点、数据处理方式等。
- 实际提问：@ 相关文件、描述问题、让 AI 改代码并测试，不对再补充或自己改。
- 未显式拆出「角色设定 / 上下文注入 / 思维链」三步，也没给出完整 Prompt 示例。

**评价**
- 优点：确实用 skills + @ 文件，有实战；人工 Review 的模式健康。
- 不足：面试官要的是「系统化 Prompt 设计」；雷达场景未构造出一条可迁移的思维链（现状→拆问题→方案→优劣→边界）。

**参考答案要点**
- **角色**：用一句话限定 AI 的专长视角，避免泛泛而谈。人设提前配置在全局规则（Rules）或 Skills 里——若希望「只在某类任务（如复杂技术问题、雷达性能优化）时才启用这套人设+流程」，用 Skills 更合适；若希望「在本项目/某类文件下默认就是这个人设」，用 Rules。示例人设：「熟悉 React + WebGL + 高性能可视化的资深前端架构师」。面试时说清「全局规则」或「Skills」，避免口误。
- **上下文**：把问题相关信息给足，让 AI 在同一份事实上推理。代码类：@ 相关文件或复制目录；涉及其他工具：用 MCP 连接后再描述；再用自然语言补充数据量级、刷新频率、当前实现（如全量重绘还是增量）、FPS/CPU 等关键指标。
- **思维链**：不直接让 AI 改代码，而是规定推理顺序。**思维链本身适合保存成 Skill**，按任务触发（如用户问复杂技术问题时才生效），内容大致是：当我问你问题时，不要一上来直接改代码；先分析问题可能产生的原因，并给出每个原因对应的具体解决方案，等待用户下一步指示；当用户选定原因及方案并确认执行代码更改时，再进行更改。若在对话里手把手用，建议步骤：① 先让 AI 列出可能瓶颈；② 按计算、渲染、交互分层给出方案并标优先级；③ 选一两个高优先级方案，让 AI 出代码草稿与改造步骤，并说明风险与回滚思路；④ 人工判断问题是否找准、方案是否合理；⑤ 确认后再让 AI 执行，执行完用 Git 对比改动、做最后审核再提交。这样得到的是「推理过程+结论」，便于判断和落地。

**话术示例**
我会按「角色、上下文、思维链」三步来设计 Prompt。
角色这块，我会提前在全局规则或 Skills 里配置好人设，比如「熟悉 React + WebGL + 高性能可视化的资深前端架构师」，这样 AI 会从高性能可视化的视角给方案。如果是某类复杂问题才用这套流程，我就放在 Skills 里按任务触发；如果整个项目都希望保持这个人设，就用 Rules。
上下文就是把和问题相关的东西给足：代码用 @ 文件或复制目录，用到其他工具就通过 MCP 连上，再简单说明我遇到的现象和现状，比如数据量多大、刷新频率、现在是全量重绘还是增量、当前 FPS 和 CPU 占用，让 AI 在同一个事实基础上分析。
思维链上我不会一上来就让 AI 改代码。我会把这段流程做成 Skill，按任务触发：当我向 AI 提问时，先让它分析问题可能产生的原因，并给出每个原因对应的具体解决方案，等我来选；我选定原因和方案、确认要执行改代码之后，它再动手改。这样不会在所有对话里都生效，只在解决复杂问题时才走这套流程。具体步骤就是：先列瓶颈，再按计算、渲染、交互分层给方案并排优先级，选一两个让它出代码草稿和改造步骤、风险和回滚思路；我判断问题认没认准、方案可不可行，确认后再让它执行，执行完用 Git 看改了哪些，做最后一轮审核再提交。这样既有推理过程又有结论，我敢用、也方便后续维护。

### 13. 用 AI 从 0→1 搭建微服务：AI 做什么、人做什么？

**问题**：全新项目从 0 到 1 搭微服务原型（前端/后端/部署），如何规划 AI 工作流？每一步 AI 做什么、你做什么？

**我的答案要点**
- 前端：Cursor + skills + Figma，AI 生成基础架构；自己设计架构和组件模板，AI 按示例生成其余。
- 后端：IDEA AI；自己写一个接口和一个 SQL，其余类似代码让 AI 生成。
- 部署：用 OpenClaw 等，描述前端包+后端 jar+脚本，让 AI 生成 Docker/Compose 一键部署。
- 面试官指出漏了「核心业务逻辑填充」和「测试」。

**评价**
- 优点：清楚划分「模板人写、AI 扩展」；部署阶段也纳入 AI。
- 不足：测试讲得少；「核心逻辑」由人主导的位置不够突出。

**参考答案要点**
- 分三阶段：前端开发、后端开发、部署；每阶段用的 AI 工具不同，人定主架构/规范，AI 做扩展与重复。
- **前端**：Cursor（内置 Claude），人写主架构和示例代码，类似代码交给 AI 生成。
- **后端**：IDEA + JetBrains AI；人给基础结构并拆分一个微服务、写一个接口，其余微服务与接口按该接口规范让 AI 生成；核心领域与关键流程人建模，AI 扩展 CRUD、DTO、重复接口。
- **测试**：AI 根据接口生成单测/接口测试，人重点检查边界与异常。
- **部署**：前端打成静态资源（如 html），后端打成 jar，准备好建表脚本；用 OpenClaw 配 Deepseek，让 AI 生成 Dockerfile（把后端 jar 打成镜像）和 docker-compose 的 YAML 配置文件，再用 docker compose 一键部署；人负责环境与安全、最终验证。

**话术示例**
从 0 到 1 搭微服务我会按前端、后端、部署三个阶段来分工，用的 AI 工具也不一样。
前端主要用 Cursor，里面用的是 Claude。主架构和示例代码我自己写，类似的代码交给 AI 生成。后端用 IDEA 和 JetBrains AI，我给出基础结构、先拆出一个微服务，其它微服务让 AI 按我的拆分思路生成；接口我写一个，按这个接口的规范让 AI 生成其它接口；核心业务和关键流程我自己建模，AI 只做扩展和重复的 CRUD、DTO 这些。测试会让 AI 根据接口生成单测和接口测试，我重点看边界和异常路径。最后部署阶段，前端打成静态包、后端打成 jar、准备好数据库建表脚本，用 OpenClaw 配 Deepseek，让 AI 生成 Dockerfile（把后端 jar 打成镜像）和 docker-compose 的 YAML 配置文件，再用 docker compose 一键部署；环境和安全、最终能不能跑通由我把关。

### 14. 用 AI 辅助定位并解决前端内存泄漏

**问题**：项目大量代码是 AI 写的，长期运行后出现内存泄漏，如何用 AI 辅助定位和解决，而不是纯靠自己看代码？

**我的答案要点**
- 用 Chrome Memory 面板确认是内存问题；把测试结果（快照）给 AI；能定位到功能就把对应代码 @ 给 AI，不知道就 @ 整个项目；说是「最省人力的方式」，未展开「如何分步提问」。

**评价**
- 优点：知道用 Memory 定位。
- 不足：未讲「多次快照对比、定位增长对象、查引用链」的标准流程；提问偏粗（整个项目给 AI 容易得到泛泛结论）。

**参考答案要点**
- 流程：DevTools Memory 多次 Heap Snapshot（初始 / 操作一段时间后 / 重复操作后），对比找出持续增长的对象类型；根据 retained path 定位到组件/模块。
- 把该部分代码 + 快照分析结果 @ 给 AI，问题描述示例：「该组件多次打开关闭后实例数持续增加，快照中 XXX 类型对象增长明显，请检查是否有未清理的监听/定时器/全局引用，并给出修复建议。」
- 让 AI 标出可能泄漏点、给出修改；自己再打快照验证。

**话术示例**
前端内存泄漏我一般不会直接把整项目扔给 AI，而是先用 DevTools 的 Memory 面板做几轮 Heap Snapshot。
比如在页面刚打开时拍一张快照，经过一段时间操作后再拍一张，然后重复几轮典型操作再拍第三张，通过对比找出哪些对象类型在持续增长。
根据 retained path 可以大致锁定到哪个组件或模块，然后我会把这部分代码和快照分析的关键信息整理出来，@ 给 AI，描述成类似这样的问题：「某组件多次打开关闭后实例数持续增加，快照中 XXX 类型对象增长明显，请帮我检查是否有未清理的事件监听、定时器或全局引用，并给出修改建议」。
AI 给出可能的泄漏点和修复方案后，我会按建议修改，再重新打快照验证，确保对象数量不再异常增长。

### 15. 上下文过长、幻觉、代码膨胀时如何管理 AI 质量

**问题**：① 新项目让 AI 做了很多理解和生成，后续上下文过长导致幻觉，怎么办？② AI 代码量 often 是人写的 3～5 倍，如何保证质量？

**我的答案要点**
- 上下文：新问题新对话；同一问题链在同一对话；拆解问题、避免一次问太大、限定到少数文件或一个组件。
- 幻觉：问题拆解、缩小范围、多会话并行。
- 代码膨胀与质量：不允许黑盒，AI 代码必须人工审核；至少每个方法/关键步骤要懂；确认没问题才合入。

**评价**
- 优点：拆解、缩小范围、多会话、人工审核都到位。
- 不足：可补充「让 AI 自检、生成自测用例、配合静态分析」等。

**参考答案要点**
- 上下文：按模块/场景分会话；让 AI 总结当前上下文为短摘要再续问；限定只读/只改某目录或文件。
- 幻觉与质量：让 AI 给出变更对比和自测步骤；对生成代码跑 lint、单测、类型检查；关键模块先让 AI 出设计说明，认可后再写代码。
- 代码膨胀：要求最少必要代码、refactor 建议；对已有文件「最小增量修改」而非重写。

**话术示例**
在管理 AI 上下文和代码质量这块，我会尽量把问题按模块或场景拆开，每个会话只围绕一块内容，这样上下文不会无限膨胀。
当对话变长时，会先让 AI 用几句话总结当前上下文，再在这个摘要基础上继续问，同时限定它只读/只改某个目录或某几个文件，减少无关扩写。
对于幻觉和质量问题，我会要求 AI 在给出修改后，同时给出本次变更的对比说明和自测步骤，然后配合 lint、类型检查和单元测试一起跑，关键模块会先让它写一份设计说明，我确认思路没问题再让它动代码。
代码量方面，会明确要求「在满足需求的前提下写最少必要的代码」，对已有文件优先做最小增量修改而不是整文件重写，同时也会让它提出 refactor 建议，必要时再由我主导重构。
