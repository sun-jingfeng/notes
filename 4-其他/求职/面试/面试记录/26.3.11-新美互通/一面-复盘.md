### 1. Java / Spring Boot 了解程度

**问题**：想了解一下你后端技术了解的程度，主要后端用 Java，对 Java 里 Spring Boot 这套框架的一些理解。

**我的答案要点**
- Spring Boot 是后端主流框架，做后端项目都用它；类比前端相当于 Vue 或 React，提供现成环境和很多工具。
- 和 Maven 配合，Maven 类似前端的 npm，是依赖和打包工具，一起构成后端项目基础框架。

**评价**
- 优点：能类比前端，说明定位准确。
- 不足：偏「是什么」，未点到自动配置、起步依赖、内嵌容器等 Spring Boot 特色。

**参考答案要点**
- Spring Boot 是基于 Spring 的快速开发框架，提供**自动配置**、**起步依赖**（starter）、**内嵌 Tomcat**，开箱即用。
- 与 Maven/Gradle 配合做依赖管理和打包；约定优于配置，减少 XML，适合微服务和单体快速交付。

**话术示例**
Spring Boot 是现在 Java 后端最常用的框架，可以理解成在 Spring 基础上做了很多约定和封装，比如自动配置、各种 starter 起步依赖、内嵌 Tomcat，和 Maven 一起用，做依赖管理和打包，基本上新建项目就能跑起来，不用像以前那样配很多 XML，所以无论是单体还是微服务，大家都会优先选 Spring Boot。

### 3. 控制反转、依赖注入的好处

**问题**：控制反转和依赖注入有什么好处？

**我的答案要点**
- 以前用别的类要自己 new，耦合高；Spring 把实例交给容器管理，控制权反转，解耦。
- 自己注册并注入叫依赖注入；注册分自己写的 Bean（加 @Component 或 @Controller/@Service/@Repository）和第三方（配置类里 @Bean 返回实例）。
- 好处包括复用，作用域有全局单例、一次请求一个实例、每次 new；多数用全局单例，整个项目共用一个实例。

**评价**
- 优点：IoC、DI、Bean 注册方式、作用域都提到了。
- 不足：可更突出「可测试、易替换实现、统一管理生命周期」等好处。

**参考答案要点**
- **解耦**：依赖由容器创建和注入，调用方不 new，降低耦合，便于单元测试（可注入 Mock）和替换实现。
- **统一管理**：Bean 生命周期、作用域（单例、多例、请求作用域）由容器管理，复用单例减少开销。
- **与 AOP 等配合**：只有交给 Spring 管理的 Bean 才能享受 AOP、事务等能力。

**话术示例**
最大的好处是解耦和可维护。以前要自己 new 依赖，类之间绑得紧，测试也不好做。交给 Spring 之后，控制权在容器手里，我们只关心接口和注入，实现可以随时换，单元测试也可以注入 Mock。另外单例这种作用域由容器统一管理，整个应用共用一个实例，既省资源又方便配合 AOP、事务这些能力。

### 4. 项目里哪些 Bean 是全局单例、哪些是每请求一个

**问题**：项目里哪些 Bean 是全局单例，哪些是每个请求一个实例的？

**我的答案要点**
- 用的全都是全局单例，没用别的；Controller、Service、Repository 以及领域里定义的类都是单例。

**评价**
- 优点：如实说明项目现状。
- 不足：未说明「为什么可以全是单例」「什么场景才需要 request 作用域」，容易被追问线程安全。

**参考答案要点**
- 项目里 Controller、Service、Repository、普通工具类都是**单例**，因为一般不存请求级状态，线程安全靠方法局部变量和参数。
- **每请求一个**常用于：要存当前请求/用户上下文（如当前用户、租户 ID），或临时状态不适合共享；可用 `@Scope("request")` 或 `@RequestScope`。

**话术示例**
我们项目里 Controller、Service、Repository 都是单例的，因为业务逻辑一般不往成员变量里存请求相关的状态，数据都在方法参数和局部变量里，所以单例没问题。如果真有需求要存当前请求或用户信息在 Bean 里，才会用 request 作用域，比如 @RequestScope，那种就是每个请求一个实例。

### 5. Controller 单例与多用户是否会数据冲突

**问题**：Controller 也是单例的话，多个用户请求同一个 Controller，数据会不会冲突或相互污染？如果 Bean 里有类级别的共享属性（如成员变量 Map、List），会不会出现这种情况？这块之前有考虑过吗？

**我的答案要点**
- 目前没出现过问题；每个用户每次请求的数据是单独处理的，数据都从参数传进来。
- Controller 单例里没有状态，没有存不同用户、每次请求不同的属性，属性都从参数里来，Controller 只是请求接收器。
- 如果真在 Controller 里存请求相关的状态，就需要改成每次请求新建一个 Bean（request 作用域）。
- 之前没用过这种写法，因为没在 Controller 里存这类数据；如果要存就需要改作用域。

**评价**
- 优点：最后能说清「无状态则单例安全」「有状态要改作用域」，思路正确。
- 不足：前面「目前没出现问题」略显被动；应先点明「无状态所以安全」，再补「若有成员变量存请求数据就会有问题」。

**参考答案要点**
- **不会冲突的前提**：Controller 不存请求相关状态，数据全靠方法参数、局部变量和调用 Service 传递，单例是线程安全的。
- **会冲突的情况**：若在类里定义成员变量（如 Map、List）存用户/请求数据，多线程共享同一实例就会互相污染。
- **做法**：要么不在 Controller 存请求级状态（最佳实践），要么对该 Bean 使用 `@RequestScope` 每请求一个实例。
- 平时设计时就要避免在 Controller/Service 单例里放可变的请求级状态。

**话术示例**
不会冲突的前提是 Controller 里不存和请求相关的状态，数据都通过方法参数进来，再交给 Service 处理，这样单例是安全的。如果反过来，在 Controller 里搞一个成员变量 Map 或 List 存每个请求的数据，那多用户共用同一个 Controller 实例就会互相污染，这时候就要把这个 Bean 改成 request 作用域，每个请求一个实例。我们项目里一直没在 Controller 存这种状态，所以之前没特别考虑，但心里知道这条线：无状态就单例，有状态就改作用域。

### 6. AOP 了解与使用场景

**问题**：AOP 这块了解吗？项目中用到 AOP 吗？什么场景？切面/注入点在哪？日志记什么东西？

**我的答案要点**
- AOP 是切面编程，是代码复用里最细粒度的方式；把方法里重复的代码抽成切面再注入进去，类似依赖注入但细化到代码行级别。
- 用到的注解是 @Aspect，写切面时用。
- 项目中用到了，场景是日志；切面单独提成一个 Aspect，注入点是需要写日志的包，把项目里主要的 Service 层、Controller 层包路径都配进去了，这些层方法被调用时都会走切面写日志。
- 有日志等级配置，达到输出等级才会输出。
- 记的主要是运行时间；个性化日志自己用 logger 写，发布时会删掉。

**评价**
- 优点：能说清 AOP 是切面、复用、以及用 @Aspect + 包路径做统一日志。
- 不足：面试官追问「一般是 Controller/Service 方法级别的统一日志，你记的是啥」——答「运行时间」后又被追问「每步耗时的话日志量会很大」，暴露出对日志量和磁盘压力的考虑不足。

**参考答案要点**
- **AOP**：面向切面编程，把横切关注点（日志、权限、事务）从业务代码中抽离，通过切点 + 增强在指定位置织入。
- **实现**：@Aspect 定义切面，@Pointcut 指定切点（如 execution 某包下所有方法），@Around/@Before/@After 写增强逻辑。
- **日志切面**：通常记**方法级别**的入口参数、返回结果、**方法总耗时**、异常，而不是方法内每一步的耗时，否则日志量巨大。
- 若需细粒度耗时，应对关键步骤采样或单独打点，并控制日志级别和输出量，考虑磁盘 IO 和日志轮转。

**话术示例**
AOP 就是面向切面编程，把像日志、权限、事务这种横切逻辑从业务里抽出来，用切点加增强在指定位置织入，我们项目里用 @Aspect 配了切面，切点是 Service、Controller 的包路径，所以这些层的方法被调用时都会统一走切面。日志这块我们记的是方法级别的信息，比如入参、返回、还有**整个方法的耗时**，用来排查慢请求；不会在方法内部每一步都打耗时，那样日志量会非常大，对磁盘和 IO 压力也大，所以只做方法粒度的耗时统计。

### 7. 日志量、磁盘压力与查看磁盘 IO 命令

**问题**：运行时间是每个请求的耗时还是每一步的耗时？每一步都打日志的话日志量会很大，日志往哪打？有评估过磁盘压力吗？如果让你去查当前磁盘 IO 压力，用什么命令？

**我的答案要点**
- 说是每个程序每一步执行的耗时。
- 日志往磁盘打，发布时存磁盘，开发时打控制台。
- 没考虑过磁盘压力和日志量，没出现过问题。
- 查看磁盘 IO 的命令没关注过，只做过部署项目。

**评价**
- 优点：如实承认未评估和不知道命令。
- 不足：每步耗时打日志在生产里不可行；Linux 查磁盘 IO 的常用命令（如 iostat、iotop）应掌握基础。

**参考答案要点**
- 生产环境应避免「每一步都打耗时」，只做**方法级或请求级**耗时；否则日志量和磁盘 IO 都会很大，需做日志采样、级别控制、轮转。
- 日志可打文件（Logback/Log4j 配置 file appender）、控制台；生产一般按天或按大小轮转，可对接 ELK 等。
- **查看磁盘 IO**：`iostat -x 1` 看 IO 读写、利用率；`iotop` 看进程级 IO；`df -h` 看磁盘空间。

**话术示例**
我们后来是只记方法级别的耗时，不会每一步都打，否则日志量会很大，磁盘 IO 也会上去。查磁盘 IO 的话，我一般用 iostat，比如 iostat -x 1 看读写和利用率，或者 iotop 看哪个进程在打 IO；磁盘空间用 df -h。生产上我们会做日志轮转和级别控制，避免无限制写盘。

### 8. MySQL 索引与建表最佳实践

**问题**：MySQL 索引了解多少？项目里建表是你负责的吗？建表有哪些最佳实践？

**我的答案要点**
- 索引指的是表的索引；主键、外键算索引吗？主键算索引。
- 建表是自己负责的；学理论时知道三范式，实际建表时直接让 AI 生成了。

**评价**
- 优点：诚实说建表用 AI，并会自己审。
- 不足：未主动说索引类型（B+树）、唯一索引、联合索引、覆盖索引、建表时的字段类型、主键设计、是否反范式等，容易被认为基础弱。

**参考答案要点**
- **索引**：主键、唯一索引、普通索引、联合索引；InnoDB 用 B+ 树；合理索引加速查询，过多索引影响写和空间。
- **建表**：主键尽量稳定、短（如自增或雪花）；常用查询条件建索引，注意最左前缀；字段类型合适（如用 int 存数字、varchar 长度适中）；大表考虑分区、分表；可适当反范式减少 join。
- 三范式是基础，实际会按查询和性能做权衡。

**话术示例**
索引这块我们用的主要是主键和常用查询字段上的索引，InnoDB 是 B+ 树索引，建表时我会注意主键设计和常用查询条件，避免全表扫描。建表时字段类型、长度会选合适，也了解三范式，但实际会按查询和性能做一点反范式取舍。具体建表语句我们项目里会用 AI 生成，但我自己会审一遍，保证主键、索引和字段设计合理。

### 10. 前端框架与为什么转 React

**问题**：前端主要用什么框架？为什么换成 React？当初为什么选 React？Vue 和 React 你自己有什么理解？你觉得 Vue 开发效率更高？函数式编程 / hooks？

**我的答案要点**
- 之前用 Vue，现在用 React；部门合并后军端民端统一成一个技术部，领导统一拍板用 React。
- 自己认为 Vue 在技术、性能、开发体验上优于 React，但 React 发布早、生态大，优势不足以触发技术迁移，所以行业更多人用 React；作为员工领导要求用啥就用啥。
- 函数式 / hooks：React 里叫 hooks，因为 React 以组件为核心不是显式响应式，hooks 是把程序片段挂到主流程上；React 更灵活、限制更少。

**评价**
- 优点：对 Vue/React 有对比、对部门技术选型原因说得清楚；hooks 的理解基本正确。
- 不足：可说清「React 函数式 + 不可变、Vue 响应式 + 可变」的差异，以及 hooks 的本质是复用状态逻辑。

**参考答案要点**
- **选型**：部门/公司统一技术栈是常见原因；React 生态大、招聘多，Vue 上手快、模板更贴近传统前端。
- **对比**：Vue 响应式 + 模板，上手快；React 函数式 + 不可变数据 + JSX，更灵活、组件模型统一。
- **Hooks**：在函数组件里复用状态和副作用逻辑，如 useState、useEffect，不依赖 class 和 this，逻辑复用比 HOC/render props 更直观。

**话术示例**
我们这边是部门合并后统一用 React，所以就从 Vue 切过去了。我自己理解 Vue 上手快、模板和响应式对前端同学更友好，React 更偏函数式和不可变，生态大、招人多。Hooks 就是在函数组件里复用状态和副作用，不用 class，逻辑复用比以前 HOC 那种方式更直接，所以现在 React 项目都会用 hooks。

### 13. 后端性能优化经验

**问题**：后端有哪些性能优化或调优的经验？

**我的答案要点**
- 当场答的是架构：单省单体 → 多省微服务，公共模块 + 各省独立服务；前端分支策略、后端微服务拆分。
- 复盘后整理的性能层面回答：用 JMeter 压测找瓶颈 → 配置熔断和备选方案；非核心任务走 MQ 异步；用 AOP 统计每个接口耗时，定位慢接口后深入优化；先看 SQL，通过合理建索引提升查询效率，有 N+1 或慢查询一并排查；大数据或热点用 Redis 缓存，配合线程池、多线程提高响应；必要时根据压测调线程池、连接池参数。

**评价**
- 优点：能把「单省单体 → 多省微服务」的演进说清楚；复盘后能归纳出完整性能优化链路（压测 + AOP 定位 → SQL/缓存/线程池/MQ → 熔断兜底），手段具体、可落地。
- 不足：当场偏架构、少性能；口述时注意顺序：先定位再单接口优化再兜底，熔断作为稳定性兜底带过即可。

**参考答案要点**
- **架构**：单体拆微服务、多省共用公共服务，属于架构演进，可简要带过。
- **性能·定位**：JMeter 压测找整体瓶颈；AOP 统计接口耗时，找出慢接口。
- **性能·单接口优化**：先看 SQL——是否有慢查询、N+1，通过合理索引、改查询方式优化；热点或大数据用 Redis 缓存；可并发的用线程池、多线程，非核心任务走 MQ 异步；根据压测调线程池、DB 连接池等参数；必要时批量处理。
- **性能·兜底**：关键路径配置熔断和备选方案，防止雪崩。

**话术示例**
后端这块我们做过从单体到微服务的拆分，多省共用公共服务，各省单独服务，这是架构层面的。性能上的话，我们会先用 JMeter 做压测找瓶颈，再用 AOP 对每个接口做耗时统计，把慢接口筛出来逐个优化。单接口会先看 SQL，有没有慢查询、N+1，通过加索引、改查询方式提效；能缓存的用 Redis，适合并发的用线程池、多线程，非核心任务丢到 MQ 异步；线程池、连接池也会根据压测结果调。关键链路还会配熔断和备选方案，防止雪崩。有批量读写场景的话会做批量处理，减少往返。

### 14. 微服务部署方式与打包

**问题**：微服务是怎么部署的？是多个节点、服务发现，还是原始方式？你说打了一个包是什么意思？微服务拆完应该是多个 jar 或多个节点吧？

**我的答案要点**
- 开发时每个服务一个端口，有一个 gateway 做路由；部署时记得是打了一个包就部署了。
- 后来补充：好像是多个包，用容器名字访问的。

**评价**
- 优点：如实说部署是自己负责，开发时用 gateway。
- 不足：「打一个包」和「微服务多节点」矛盾，暴露出对部署形态不清晰；若实际是单体部署或 fat jar 一起跑，应说清「目前是多模块/多 jar 但同一台机或同一编排」而不是「一个包」。

**参考答案要点**
- 典型微服务部署：每个服务独立打包成 jar/镜像，多实例、通过服务发现（如 Nacos）或网关（如 Spring Cloud Gateway）访问；容器化用 Docker/K8s 部署多节点。
- 若当前是「多模块打在一个工程里、一个进程跑多个模块」或「多 jar 同一台机」，应如实说清，并说明和标准微服务（多节点、独立扩缩容）的差异；若用 Docker 则说是多容器、通过容器名或服务名访问。

**话术示例**
我们开发时是每个服务一个端口，前面有 Gateway 做路由。部署时是每个服务打成独立的 jar 或镜像，用 Docker 跑的话就是多个容器，通过容器名或服务名互相访问；如果是多台机器就多节点部署，通过服务发现或网关统一入口。我们当时具体是多包、多容器部署的，用容器名做服务间调用。

### 15. 分布式事务 Seata：场景、实现与回滚通知

**问题**：简历里用到分布式事务？当时遇到什么问题要引入？分布式事务应该在哪些场景才有必要？Seata 是怎么实现跨服务统一事务的？如果某一个服务事务失败了要回滚，怎么通知前面已经成功的服务去回滚？

**我的答案要点**
- 当时是学习目的试了一下，项目没有适用场景就没用。
- 使用场景：微服务下 Spring 事务没法跨两个服务，需要 Seata；当一个服务调另一个服务且两个操作要「要么都成功要么都失败」时用 Seata。
- 实现：每个微服务的数据库里配一个表存回滚信息，操作数据库后把反向操作记到这个表；事务失败时查这个表执行回滚。
- 怎么感知和通知：有一个第三方协调者，和这几个服务平级，做统一管理，具体实现忘了。

**评价**
- 优点：知道 Seata、AT 模式存回滚信息、有 TC 协调者概念。
- 不足：说成「学习试了一下、项目没用」容易让简历上的「分布式事务」显得虚；TC 是 Seata Server，负责发全局锁和回滚指令，应能说清「TM 开启全局事务、RM 注册分支，失败时 TC 通知各 RM 回滚」。

**参考答案要点**
- **何时引入**：跨服务、跨库且需要**强一致**的场景（如扣款+积分）；能接受最终一致则优先用本地事务+可靠消息（本地消息表、事务消息）。
- **Seata**：AT 模式每个库有 undo_log 表，存前后镜像；TC（Seata Server）协调全局事务和分支，失败时根据 undo_log 生成反向 SQL 在各库回滚。
- **回滚通知**：TM 向 TC 发起全局回滚，TC 通知各分支 RM 回滚，RM 根据 undo_log 执行补偿；不是「第三方通知」而是 Seata 的 TC 统一协调。

**话术示例**
我们项目里其实没有真正的跨服务强一致场景，所以分布式事务是学习时试的，没有在生产用。一般引入是跨服务、跨库并且要强一致的时候，比如扣款和积分必须一起成功或一起失败；如果能接受最终一致，我会优先用本地事务加消息表或事务消息。Seata 的 AT 模式是每个库有一张 undo_log 表，记录变更的前后镜像，有一个 Seata Server 叫 TC 负责协调全局事务，各个服务的 RM 向 TC 注册分支；一旦某个分支失败，TM 向 TC 发起回滚，TC 会通知各个 RM，RM 根据 undo_log 里的信息执行反向 SQL 做补偿，这样就能实现跨服务的回滚。

### 17. AI 相关概念：MCP、Skills、Rules、模型

**评价**

## 六、小结

- **优势**：Spring IoC/DI、Bean 作用域在被追问下能说清；前端雷达单站优化（shallowRef、预缓存、AbortController）讲得具体；AI 使用（Cursor、Rules、MCP）和离职原因表述清晰。
- **劣势**：日志量/磁盘 IO、MySQL 建表与索引、后端性能调优、微服务实际部署形态、Seata 回滚机制等偏弱或表述不清；部分问题先答「没考虑/没出现」再补解释，容易显得被动。
- **岗位匹配**：岗位为全栈 AI 应用开发，技术栈 React/Java 或 Python 均可；加强运维基础（磁盘/日志）、数据库与后端性能表述，并理清微服务与分布式事务的简历表述，更利于二面或同类岗位。
