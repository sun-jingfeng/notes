## 二、面试流程与问题回顾

### 2.1 自我介绍与项目

- 自我介绍：学历、年限、技术栈（React + Java 全栈、AI 辅助）、角色（项目负责人、研发组长）、最近项目概要。
- 项目深挖：气象多省系统、前后端分工（前端 3 人 + 后端 2 人，本人前后端都做）、部署由本人负责、分支与微服务演进。
- **跨 iframe 截图**：实现思路（html2canvas 限制、父子 postMessage、选区窗口、requestAnimationFrame、鼠标出界处理）——面试官表示已理解，未再追问。

### 2.2 数据与存储

- 数据量：项目本地部署，数据几十 G 级别。
- 表结构：核心表为用户、权限、气象名词、报警等；气象数据多为算法产出 JSON/文件，不落库；色斑图通过规则拼 URL，前端用插件解析。
- 大文件/实时处理：提前处理成 PNG 传递；少量实时读取+插值用线程池，结果缓存到 Redis；明显耗时的都提前跑脚本处理。

### 2.3 前端

- **组件封装标准**：业务熟悉 → 识别可复用场景（如时间轴）→ 共性封装、差异参数化；组件内拆分、组合式函数/自定义 Hooks。
- **Vue3 浅层响应式**：`shallowRef`、`shallowReactive` 用于地图等大对象，只监听引用变化，减少深层响应式开销。
- **高频排序场景优化**：先答了 `key` 保证 DOM 复用；面试官提示「计算后做缓存」——未主动想到，需补。

### 2.4 后端与性能

- **线程池**：业务中用 8 线程，理由是「够用、一般跑不满」；被追问如何评估线程数。
- **评估方式**：提到压力测试工具（名未说出）、看响应时间；高并发时提升服务器/CPU、动态线程池、实在不行提前跑脚本。
- **线程池数量评估**：被追问「给一台服务器如何评估」——答得偏笼统，未系统讲 CPU 核数、IO/CPU 型、压测曲线等。

### 2.5 数据库与事务

- 数据库：MySQL + Redis；未用 MongoDB。
- **分布式事务**：提到 Seata（说成「c 塔」）；被问是否要维护 Seata 集群、如何权衡是否上分布式事务。
- 回答：小项目单体即可；微服务化因多省复用+个性化；项目规模不确定时先单体，大了再拆；未清晰区分「跨库事务」与「跨服务事务」的选型标准。

### 2.6 AI 使用（重点）

- **使用比例**：约 50% 代码为 AI 生成（模板+AI 扩写），最终都会人工审核。
- **Prompt 结构**：角色与业务背景写在 Cursor skills；具体问题用 @ 文件/路径注入；未明确拆出「角色设定 / 上下文注入 / 思维链」三块来答。
- **场景题（雷达数据实时可视化性能差）**：面试官期望「角色设定 + 上下文注入 + 思维链」——答得偏泛，缺少一条清晰的思维链示例。
- **从 0 到 1 的 AI 工作流**：前端 Cursor+设计图、后端 IDEA AI；脚手架+模板+AI 生成其余；部署用 OpenCompose 写 Docker/Compose。面试官指出少了「测试」和「核心逻辑填充」。
- **骨架/框架生成后的审查**：答了「和已有项目对比、不合理处让 AI 记到 rules」——未突出：依赖/安全、入口与路由、关键路径是否完整。
- **AI 生成代码导致内存泄漏，用 AI 辅助排查**：答了用 Chrome memory 面板、把结果和代码 @ 给 AI；面试官追问「排查思路、怎么提问」——未给出清晰的提问链（如：堆快照对比、保留对象路径、定位增长点再让 AI 看对应代码）。
- **上下文过长**：新问题新对话；同一问题链在同一对话。未主动说「摘要、分会话、锁文件范围」等。
- **幻觉/后期不准**：答了「问题拆解、缩小范围、多会话并行」——可以。
- **AI 代码膨胀与质量**：必须人工审核；按方法/步骤理解在做什么，不允许黑盒代码进库。
- **智能体/AI 产品开发**：无实战经验；了解嵌入式 AI（API、前后端对接）。
- **Cursor 与模型的关系 / 上下文如何关联**：答了读写权限、项目目录、用户输入传给模型；被问「模型无状态，上下文谁存」——答「存在本机或消费者侧」，面试官建议去了解（实际多为：对话历史在客户端或中间服务拼成 prompt，模型本身无状态）。

### 2.7 薪资与意向

- 期望：28k×13。
- 地点：当前古城地铁站附近租房，可随公司搬迁。
- 强度：现公司周中偶有加班到 9 点，周末双休；对方为工作量导致的大小周，可接受。

### 2.8 反问

- 技术栈：前端 React、后端 Java；全栈岗有偏前/偏后，按能力分配（如 7:3）。
- 业务：AI 研发部（约 25 年 10 月成立）；对内做类似 OpenCompose 的部署/排障助手（读现场环境、日志、通过客户端答疑）；对外有虚拟仿真培训、AI 眼镜生产辅助等。
- 分工：算法团队约 8～10 人；业务研发与产品/一线对接，需求会到本人。

## 四、待改进与盲区

### 4.1 技术回答

| 问题               | 不足                         | 可补内容                                       |
| ------------------ | ---------------------------- | ---------------------------------------------- |
| 高频排序前端优化   | 未主动提「计算/排序结果缓存」 | 虚拟列表、Web Worker、缓存排序结果、防抖节流。 |
| 线程池数量评估     | 偏经验（8 个够用）           | CPU 核数、IO/CPU 型、公式（如 IO 密集型 2N）、压测与监控曲线。 |

### 4.2 AI 相关（面试重点）

| 问题               | 不足                         | 可补内容                                       |
| ------------------ | ---------------------------- | ---------------------------------------------- |
| Prompt 三件套      | 未显式拆成角色+上下文+思维链 | 准备一个雷达可视化场景的完整示例（含思维链）。 |
| 0→1 工作流         | 漏了测试、核心逻辑填充       | 在流程图里加上「单测/集成测」「关键逻辑人工填与复核」。 |

- 压力测试工具、Seata 等专有名词提前记牢，避免现场卡名字。

## 五、后续行动

### 5.1 短期（针对二面/同类岗位）

1. **AI 能力展示**：写一份「雷达数据实时可视化性能优化」的 Prompt 示例（角色 + 上下文注入 + 思维链）。

## 六、小结

- **优势**：项目经历实在、前端与工程化表述清晰、跨 iframe 截图等实现讲得清楚；AI 日常使用多，有拆解问题和审核习惯。
- **劣势**：AI 的「角色+上下文+思维链」和排查类提问未形成固定话术；线程池评估、分布式事务选型、压力测试工具等偏薄弱。
- **岗位匹配**：AI 研发部偏 AI 工程与落地，对方已提示会做「类 OpenCompose」等产品；补足 AI 使用方法论与排查思路的表述，更利于二面。
